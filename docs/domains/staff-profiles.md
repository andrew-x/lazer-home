# Domain: Staff profiles

**Status: partially realized.** The People of the consultancy, and everything that describes their capacity and capability. The core `staff` + `staff_employment` + `staff_pto` tables now exist in code (`src/lib/db/staff-schema.ts`, barrelled by `src/lib/db/schema.ts`), and two local-only **CSV import tools** populate them from Rippling exports — staff and PTO (see *Import tools* below). A self-service **"My profile"** page (`/profile`) surfaces a logged-in person's own record, and a **browse-staff** feature adds a **directory** (`/staff`) and **per-person profile pages** (`/staff/[id]`) that show *other people's* records via the same presentation; both let you **edit profile links, client intro, a free-text résumé, skills, and a "Manual of Me"** (the résumé editor can extract text from an uploaded PDF; skills are picked from a hardcoded catalogue on a dedicated edit page; Manual of Me is a guided, autosaving survey on its own page — see *Manual of Me* below; owner always, others need the `staff.edit` permission — see *Authorization*) and show a **category-agnostic history feed** (see *Key flows*). **Skills are now built** (inline `jsonb`, catalogue-driven — see below) and so is **compensation** (effective-dated on `staff_employment`, import-only, view-gated — see [ADR 0020](../decisions/0020-compensation-effective-dated-import-only.md)); seniority is still proposed.

## Purpose

Be the source of truth about who works here — their identity, role, line of business, billability, and utilization target — feeding staffing decisions and performance.

## Key entities

- **`staff`** — the *durable* record of a person: `ripplingId` (**notNull, unique** — the external [Rippling](https://rippling.com) HR id; the import's match key), `name`, `email` (**not unique** — duplicates are tolerated), an optional `userId` link to the auth `user` (see below), profile links (`linkedinUrl`/`githubUrl`/`portfolioUrl`), a `clientIntro` blurb (with its own `clientIntroUpdatedAt`), a free-text `resume` (with `resumeUpdatedAt`), and lifecycle dates (`joinDate`, `terminationDate`, `isActive`). Holds nothing that changes role-to-role.
  - **`resume` / `resumeUpdatedAt`** — nullable `text` + nullable `timestamp`, added in `drizzle/0011_boring_big_bertha.sql`, modelled exactly on the `clientIntro` pair. The résumé is **plain text only — the uploaded PDF is never persisted** (it's parsed to text client-then-server and discarded). `resumeUpdatedAt` is stamped **explicitly by `updateStaffResume`** when the text changes, deliberately **not** `$onUpdate` — an import re-sync writing the `staff` row would otherwise bump it falsely.
  - **`userId`** — nullable, unique FK → auth `user.id` (`onDelete: set null`). The optional 1:1 link from a staff record to its login account: **null until the person signs in** (staff can be synced from Rippling before they ever log in), **unique** so at most one staff per user, and `set null` on delete so removing the auth account keeps the durable staff record intact.
  - **`managerId`** — nullable, self-referential FK → `staff.id` (`onDelete: set null`, added in `drizzle/0025_giant_bullseye.sql`). Who this person reports to — at most one manager, optional. A **durable identity fact on `staff`**, deliberately **not** effective-dated on `staff_employment` (no as-of "who was your manager" query demand for a display-only field). Mirrors `contacts.managerId` ([ADR 0022](../decisions/0022-contact-manager-self-reference.md)); needs the `AnyPgColumn` return annotation to type the self-reference. **Populated exclusively by the staff CSV import** (matched via the `Manager - Work email` column) — there is **no in-app editor**, so re-imports never fight a manual edit. Displayed read-only as "Reports to …" on the profile. See [ADR 0026](../decisions/0026-staff-manager-self-reference.md) and *Import tools* below.
  - **`skills`** — a `jsonb` column (`StaffSkill[]` = `{ name, level }[]`, **NOT NULL default `[]`**, added in `drizzle/0016_demonic_rick_jones.sql`) — the schema's **first jsonb/array column**. Skills are stored **inline** here, deliberately **not** a normalized `skill` + join table; each entry pairs a skill name from the hardcoded catalogue in `src/lib/skills.ts` with a 3-value proficiency level (`senior`/`intermediate`/`learning`). See [ADR 0018](../decisions/0018-skills-inline-jsonb-catalogue.md) and *Skills* below.
- **`staff_employment`** — the *time-varying* employment facts for a person. **One row per change**, not an update-in-place: each row is keyed by `effectiveFromDate` (notNull) and the **current state is the row with the latest `effectiveFromDate`**. FK `staffId` → `staff.id` (`onDelete: cascade`). Fields: `lineOfBusiness`, `role`, `employmentType`, `isBillable`, `utilizationTarget` (integer percent 0–100), `billableType` (NOT NULL, default `HUB`), **`isManagement`** (bool, default `false`), and **compensation** — `base`, `hourlyRate`, `guaranteedBonus`, `discretionaryBonus` (all **NOT NULL** `numeric(12,2)`, `mode:number`; `discretionaryBonus` defaults 0) + `currency` (**NOT NULL** `currencyEnum`), added nullable in `drizzle/0017_gorgeous_celestials.sql` and made NOT NULL in `drizzle/0018_dry_captain_cross.sql` (which backfills legacy rows to 0 / 'CAD' first). (The external `ripplingId` lives on `staff`, not here.) See [ADR 0007](../decisions/0007-staff-employment-effective-dating.md) for *why* employment is split out and history-as-rows, plus the in-place-correction amendment used by bulk-edit.
  - **Compensation is effective-dated like the rest of the row, but import-only, required, and view-gated.** All five columns are **NOT NULL** (`discretionaryBonus` defaults 0). Populated **only by the Rippling CSV import** — there is **no in-app edit UI** (not the profile, not bulk-edit). Comp is **required on every import row** — except **guaranteed bonus, which is optional** (a blank cell defaults to `0`): the importer skips a row it can't supply `base`/`hourlyRate`/`currency` for (see *Import tools*), so there is **no carry-forward on the import path**. It *is* still carried forward when a *non-comp* change spawns a new row via bulk-edit. `discretionaryBonus` is **not imported** — it always defaults to 0. Visibility requires `staff.viewCompensation` (own comp always visible) — see *Authorization* and [ADR 0020](../decisions/0020-compensation-effective-dated-import-only.md). Values/formatting live in the pure `src/lib/currency.ts` module (mirrors `line-of-business.ts`), the single source shared by the `currencyEnum` pgEnum, the import zod schema, and display.
  - **`isManagement` is orthogonal to `role`.** It is *not* a role value — someone can work a billable role (e.g. `ENGINEER`) and *also* be management for it. It is set **in-app only** (the bulk-edit tool below; never derived from the Rippling CSV), so the staff import **preserves** it across re-syncs (carried forward when a new effective-dated row is written). It does **not** affect billability — `LEADERSHIP` stays non-billable on its own; the flag is independent. It is **not** surfaced on the staff profile view; it's maintained and read via the bulk-edit tool below.
  - **`billableType` is likewise in-app only.** A required `HUB`|`GLOBAL` value (default `HUB`, no longer a "no billable type" state since `drizzle/0010` backfilled nulls and added the NOT NULL constraint). Like `isManagement`, it is never derived from the Rippling CSV, so the import **carries it forward** when writing a new effective-dated row — a re-sync won't reset a `GLOBAL` person back to `HUB`. Set via the bulk-edit tool below.
- **`staff_pto`** — leave records for a person: `startDate`/`endDate` (both date, notNull), `type` (`ptoTypeEnum`), and `isPending` (bool, default `true`). FK `staffId` → `staff.id` (`onDelete: cascade`), plus a **notNull, unique `ripplingId`** — the Rippling "Leave request ID", the PTO import's match key (one record per leave request). Unlike `staff_employment`, this is **not** effective-dated history — each row is a discrete leave span. `isPending` means awaiting approval; it's cleared on approval, or set `false` on insert when the row is synced from Rippling as already-approved. Populated by the PTO import tool (see below).
- **Skills** — capability + proficiency level. **Status: built**, but **not** the many:many table originally proposed: they live **inline** on `staff.skills` (jsonb), picked from the hardcoded catalogue in `src/lib/skills.ts`. See the `skills` column above, *Skills* below, and [ADR 0018](../decisions/0018-skills-inline-jsonb-catalogue.md).

### Enums (`src/lib/db/staff-schema.ts`)

- `lineOfBusiness`: `CORPORATE`, `CORE`, `FINTECH`, `COMMERCE`, `DESIGN` — **shared/global**, not staff-specific (top-level export reused by CRM/allocations).
- `role`: `ENGINEER`, `DESIGNER`, `LEADERSHIP`, `SALES`, `SOLUTIONS`, `OPERATIONS`, `ARCHITECT`, `DELIVERY`, `QA` — the value formerly named `MANAGEMENT` was **renamed in place to `LEADERSHIP`** via `ALTER TYPE ... RENAME VALUE` (`drizzle/0009_purple_exodus.sql`; no data migration, existing rows kept their label). drizzle-kit can't detect an enum-value rename, so the generated destructive drop/recreate was replaced by hand — keep that in mind for future enum renames.
- `employmentType`: `FULL_TIME`, `HOURLY`
- `billableType`: `HUB`, `GLOBAL` (NOT NULL, default `HUB` — every employment row has one)
- `currency`: `CAD`, `USD`, `GBP`, `EUR`, `AED` — the compensation currency (NOT NULL). Values are **derived from the `CURRENCY` tuple in `src/lib/currency.ts`** (`pgEnum("currency", [...CURRENCY])`), not hand-listed, so the pgEnum can't drift from the shared module. **Gotcha — adding an enum value:** `AED` was added by appending to the `CURRENCY` tuple + `drizzle/0027_flawless_arachne.sql` (`ALTER TYPE "public"."currency" ADD VALUE 'AED'`). `ALTER TYPE ... ADD VALUE` **cannot run inside a transaction**, but drizzle-kit's `db:migrate` wraps each migration in one — so it fails there. It was applied out-of-band via **autocommit plus a manual `_journal.json` record** (the same reason the PTO `ADD VALUE`s above are worth flagging). Expect to hand-apply future enum-value additions this way.
- `pto_type`: `VACATION`, `STATUTORY_HOLIDAY`, `SICK_LEAVE`, `UNPAID_LEAVE`, `PARENTAL_LEAVE`, `BEREAVEMENT_LEAVE`, `COMPANY_RETREAT`, `RELIGIOUS_HOLIDAY`, `JURY_DUTY`, `LEAVE_OF_ABSENCE`, `OTHER_LEAVE` — **enum order matters** (it's the on-disk order). New values (`COMPANY_RETREAT`/`RELIGIOUS_HOLIDAY`, later `JURY_DUTY`/`LEAVE_OF_ABSENCE`) were inserted **before** `OTHER_LEAVE` via `ALTER TYPE ... ADD VALUE`. The `PTO_TYPE` tuple in `src/lib/pto-import/types.ts` is **derived directly** from this pgEnum (`ptoTypeEnum.enumValues`) rather than hand-mirrored, so it can never drift from the schema.

**Display labels — `src/lib/staff-enums.ts` (labels-only; values stay on the pgEnum).** Human-readable labels for `role`/`employment_type`/`billable_type`/`pto_type` live in one pure, client-safe module — `ROLE_LABELS`, `EMPLOYMENT_TYPE_LABELS`, `BILLABLE_TYPE_LABELS`, `PTO_TYPE_LABELS` (each `Record<enumUnion, string>`) plus the `Role`/`EmploymentType`/`BillableType`/`PtoType` union types (derived via `import type` from the schema enums). Every staff/admin render site uses these maps instead of `humanizeEnum()`, which **mangled acronyms** (`QA` → "Qa"). **Deliberate deviation from the [ADR 0016](../decisions/0016-junction-table-and-shared-enum-conventions.md) shared-enum pattern:** for these staff enums only the *labels* were extracted — the enum **value tuples stay defined inline on the `pgEnum`s in `staff-schema.ts`** (consumed elsewhere via `*.enumValues`), NOT hoisted into a standalone tuple module the way `line-of-business.ts`/`currency.ts` do it. The union types still key off the pgEnum, so a value added to the schema forces a matching label entry (type error otherwise) — no drift. (One further code-level deviation: the hand-rolled staff filter/table selects keep a `labels` prop rather than switching to the shared `EnumSelect`, because `EnumSelect` lacks an "All" sentinel and the size/aria options those selects need.)

## Nuances / gotchas

- **Read the latest employment row.** Don't assume one row per person — query `staff_employment` ordered by `effectiveFromDate desc` (filtered to `<= today` for as-of reads) to get current facts. Inserting a new row is how you "change" someone's role/billability.
- **`utilizationTarget` defaults to 100** for billable staff. Callers must set it to `0` when `isBillable` is false — the DB does not enforce that coupling.
- **PTO `isPending` defaults to `true`.** Rippling-sync inserts of already-approved leave must explicitly write `isPending: false`. Capacity/availability reads should account for pending vs. approved leave deliberately.
- **`staff.userId` is the auth link, and it's often null.** A logged-in user maps to their staff record via `staff.userId = user.id`, but the link is **null** for anyone synced from Rippling who hasn't signed in yet. Don't assume every staff row has a user (or vice versa). `email` is no longer unique, so do **not** match identity for general queries by email. (The legacy `staff_profile` example that was 1:1 with `user` has been deleted.) The **one** place email matching is allowed is the first-login auto-link below, which writes `userId` so subsequent reads go through the FK.
- **Login is gated on an active staff record with employment.** `getCurrentStaff(user)` (`src/lib/staff.ts`) is the gate run by the `(app)` layout: it returns `ok` only for an `isActive` staff row that has ≥1 `staff_employment` row; `incomplete` if active-but-no-employment; `not_setup` otherwise. So a freshly synced staff row with no employment history can't get its person into the app — employment must be populated first. See the auth flow in [flows.md](../flows.md).
- **First-login auto-link (email → `userId`).** `getCurrentStaff` first looks up by `staff.userId`; if none, it falls back to the active staff row matching `user.email` and writes `staff.userId = user.id`, **guarded on `userId IS NULL`** so it fires at most once and concurrent logins are safe. This is the bridge from email-synced staff to authenticated sessions; it's the only sanctioned email-based identity match.
- **A `staff` row is a tenure, not a permanent person.** Leavers are marked inactive (`isActive = false`, `terminationDate` set), never deleted. A rehire gets a **brand-new `staff` row** (new id, fresh `joinDate`, its own employment history) — we never reactivate the old one. So one human can be several staff rows over time, and there's **no cross-tenure "person" entity** linking them. Don't assume `staff` ↔ human is 1:1; queries that must follow a person across stints have to aggregate multiple rows. See [ADR 0007 — Leavers and rejoiners](../decisions/0007-staff-employment-effective-dating.md).
- **Dates/times are timezone-agnostic.** `joinDate`/`terminationDate`, `effectiveFromDate`, and PTO `startDate`/`endDate` are `date()` strings (`"YYYY-MM-DD"`); `clientIntroUpdatedAt` and other instants are `timestamp` without time zone. Do zone conversion at the UI edge only — see [`.claude/rules/database.md`](../../.claude/rules/database.md).
- **Edits are gated by RBAC (owner, or `staff.edit`) — declared in metadata, not the body.** `updateStaffLinks`/`updateStaffClientIntro`/`updateStaffResume` declare `secureActionClient.metadata({ authorize: authorizeStaffEdit })` (`src/actions/staff/canEditStaff.ts`); the `authorizeStaffEdit` hook reads `clientInput.staffId` and calls `canEditStaff` **before the body runs**: a user may always edit their **own** linked profile; editing anyone else's requires the `staff.edit` permission (manager/admin). The action bodies carry no authz call. This closed the former open-edit gap. See [permissions.md](./permissions.md) and [ADR 0014](../decisions/0014-rbac-better-auth-access-control.md).
- **Cross-person reads are mostly unscoped, with PTO and compensation gated.** `getStaffProfile`/`History`/`Avatar` show other people for the directory; the `(app)` layout gate is their only boundary. Two exceptions require a capability for *other* people (own always visible): (1) **PTO** — another person's aggregated PTO needs `pto.review`, else `getStaffPto` returns `null` and `ProfileView` hides the section; (2) **compensation** — `canViewCompensation` (needs `staff.viewCompensation`) gates both the profile Compensation card and whether `getStaffHistory` appends comp amounts to the employment entries at all. Comp lives on the same `staff_employment` row `getStaffProfile` returns, so the card gate is applied by *ProfileView not rendering the card* rather than by nulling the fields — but the history gate is a true read-side filter (client component). See [ADR 0020](../decisions/0020-compensation-effective-dated-import-only.md). **Inactive staff profiles are readable by direct `/staff/[id]` URL** (the directory hides them; the read doesn't). Consistent with the internal-only posture.
- **`getStaffDirectory` scans the whole `staff_employment` table** and reduces to latest-per-staff in JS. Fine at company scale; if effective-dating history grows large, switch to `DISTINCT ON (staff_id)` / a lateral join. There is **no composite index on `(staff_id, effective_from_date)`** yet.

## Skills

Skills are stored **inline** on `staff.skills` (jsonb `StaffSkill[]`), not a normalized table — see [ADR 0018](../decisions/0018-skills-inline-jsonb-catalogue.md) for *why* (and the tradeoffs).

- **Catalogue in code (`src/lib/skills.ts`).** The single source of truth for the pickable skills — a curated catalogue of ~307 skills across **9 top-level dimensions** (`Type of Work`, `Languages`, `Frameworks`, `Hosting`, `Databases`, `DevOps & CI/CD`, `Testing`, `Tools`, `Compliance`). Several dimensions carry sub-groups in the source (e.g. `Languages` → Systems/Mobile/Web3/Functional) purely to organize it, but the picker groups **one level deep, by dimension**: `SKILL_CATEGORIES` flattens each dimension's sub-groups into a single ordered skill list (order is meaningful and preserved in the picker). Also exports `ALL_SKILLS` (the flat list for validation), the fixed 3-value `PROFICIENCY_LEVELS` (`senior`/`intermediate`/`learning`, ordered most→least) + `PROFICIENCY_LABELS`, and the `StaffSkill` type. It is **client-safe** (no `server-only` import) because it's consumed by the client picker, the server validation schema, *and* the Drizzle schema at once — same discipline as `staff-import/types.ts` and the shared CRM enum module.
- **Filter helpers (pure, unit-tested in `skills.test.ts`).** `meetsMinimumLevel(level, minimum)` — "is `level` at least as proficient as `minimum`", a `<=` compare on `PROFICIENCY_LEVELS` indices (index 0 = most proficient). `matchesSkillFilter(skills, required, minimum?)` — whether a person holds **every** skill in `required` at ≥ `minimum` (**AND** semantics); an empty `required` matches everyone, and `minimum` defaults to the lowest level (`learning`), so omitting it imposes no level constraint. These back the directory's client-side skill filter (see *Directory* under *Key flows* and [ui.md](../ui.md)); nothing queries skills server-side yet ([ADR 0018](../decisions/0018-skills-inline-jsonb-catalogue.md)).
- **Read view** (`src/components/staff/skills-section.tsx`) groups a person's skills by proficiency level as badge chips inside a **Skills card** on `ProfileView` (after Résumé). Empty → "No skills yet."
- **Edit** happens on a **dedicated page** `/staff/[id]/skills` (`src/app/(app)/staff/[id]/skills/page.tsx` → `edit-skills-form.tsx`), *not* a dialog like the other profile cards (the editor is too large). The page `notFound()`s anyone who can't edit; the profile only shows its "Edit" link when `canEdit`. The editor is **level-first**: pick a target level with the "Add as" toggle (the 3 `PROFICIENCY_LEVELS`), then click skills from a **searchable catalogue of chips** (grouped one level deep by dimension, already-chosen skills hidden) to add them at that level — so a batch of same-level skills is a few clicks. Chosen skills show in a **"Your skills"** panel grouped by level, each a chip with per-chip **re-level** (▾ dropdown) and **remove** (✕). Adds **dedup by name** (`chosen` set); the schema caps the list at **200 skills**. Save calls `updateStaffSkills` then returns to the profile.
  - **Two-panel layout (`edit-skills-form.tsx`).** On desktop (≥ md) it's a 2-column grid: the "Add skills" editor is the **left** column and "Your skills" is the **right** (via `order` utilities), so it reads left→right as cause→effect. The "Your skills" column is **`sticky top-6`** with its own max-height/overflow so added skills are seen landing while the long catalogue column scrolls with the page. Inside the editor, the "Add as" toggle + search box are **`sticky top-0`** (pinned while the catalogue scrolls beneath); Save/Cancel are a **`sticky bottom-0`** right-aligned action bar (Save last) so they're always reachable. On narrow screens (< md) it collapses to a single stack with "Your skills" **first**, then the editor. The page container is `max-w-5xl` (widened from `max-w-3xl`) to fit two columns.
- **Mutation** `updateStaffSkills` (`src/actions/staff/updateStaffSkills.ts`, schema in `updateStaffSkills.schema.ts`) **replaces the whole list** and reuses the existing `authorizeStaffEdit` hook / `staff.edit` capability (own always; others need `staff.edit`) — **no new permission was added, the RBAC matrix is unchanged**. The schema validates every name is in `ALL_SKILLS` and rejects duplicates (guards against a crafted payload since the client only ever picks from the catalogue). `getStaffProfile` projects `skills` and `StaffProfile.skills: StaffSkill[]`.

## Manual of Me

A short **survey** a person fills out on their profile — 7 free-text prompts ("The basics you should know about me are…", "What helps me thrive at work…", how to give them feedback, etc.) so teammates and managers can work with them well. It is the **first consumer of the generic `responses` table** — see [ADR 0028](../decisions/0028-generic-responses-table-app-validated-question-ids.md) for *why* one generic table with app-validated question ids (not per-survey tables / a pgEnum).

- **Source of truth in code (`src/lib/manual-of-me.ts`).** A pure, client-importable module (no `db`/drizzle) exporting `MANUAL_OF_ME_QUESTION_IDS` (7 ids, canonical **display order**), `MANUAL_OF_ME_QUESTIONS` (id/title/subtitle/`type: "text"`), and the `manualOfMeQuestionId` zod enum. Same single-source pattern as `@/lib/skills` / `@/lib/line-of-business`: one module feeds the read, the write schema, and the UI. A future survey adds its own defs module and reuses `responses` + `upsertResponse`.
- **Storage grain = current value, keyed by `(staffId, questionId)`.** Answers live in the generic `responses` table (`textResponse` only for this survey), one **upsert** row per question — *not* append-only history. An unanswered question has **no row**; clearing an answer sends `"" → null` and keeps the row. See the `responses` entity in [data-model.md](../data-model.md).
- **Read — not ownership-scoped (`getManualOfMe`).** `src/actions/responses/getManualOfMe.ts` (server-only, `React.cache`) returns **all 7** questions in canonical order with the person's `textResponse` merged in (null if unanswered). Like `getStaffProfile`, it's visible to **any profile viewer** — the `(app)` layout is the only boundary (same posture as the Client intro), so both `/profile` (own) and `/staff/[id]` (others) load it and pass `manualOfMe` to `ProfileView`.
- **Write — reuses `authorizeStaffEdit` (no new permission).** `src/actions/responses/upsertResponse.ts` (`secureActionClient`, `metadata({ authorize: authorizeStaffEdit })`) upserts one answer via `onConflictDoUpdate` on `(staffId, questionId)`; `upsertResponse.schema.ts` validates `questionId` against `manualOfMeQuestionId` (so a crafted id can't write an arbitrary row) and caps `textResponse` at 10,000 chars. Gate is the **same** as every other profile-field edit — own profile always, others need `staff.edit`; **the RBAC matrix is unchanged**. Revalidates `/profile` and `/staff/${staffId}`.
- **Read view on the profile.** A "Manual of Me" card in `ProfileView` (`src/components/staff/manual-of-me-section.tsx`) shows only the **answered** questions (empty state otherwise) plus an "N of 7 answered" line; the "Edit"/"Fill out" link (to the editor page) renders only when `canEdit`.
- **Edit — a dedicated guided page** `/staff/[id]/manual-of-me` (`src/app/(app)/staff/[id]/manual-of-me/page.tsx`, mirrors the skills editor: `notFound()`s any viewer failing `canEditStaff`, since the write action is the real boundary). It renders `src/components/staff/manual-of-me-guide.tsx` — a **one-question-at-a-time** flow with a clickable step rail, **per-question autosave** (on blur, debounced typing ~800 ms, and on navigation/unmount), a subtle inline save indicator (never a toast), and a **"Done"** that flushes and returns to the profile. There is **no big Submit** — deliberately, so a person can answer in any order and come back later (several prompts invite that). Saves are serialized via a `savingRef` (one in flight; re-checks for edits landed mid-save).

## Admin tooling (localhost-only)

This domain is populated and maintained by **local-only** tools in the `admin` area (outside `(app)`, gated on a loopback host): two CSV importers — `/admin/upload-staff` (people) and `/admin/upload-pto` (leave) — and a **bulk employment editor** (`/admin/bulk-edit-roles`). They live outside `(app)` precisely because importing *creates* the staff records the `(app)` layout gates on — see [architecture.md](../architecture.md) (Admin area, [ADR 0008](../decisions/0008-localhost-only-admin-area.md)) and [flows.md](../flows.md) for the gating rationale and end-to-end steps. The importers follow the same shape: pure client transform → server `compute*Plan` diff → preview tables → commit action that **recomputes the plan server-side** and applies it in one transaction (`publicActionClient` + `assertLocalhost()`, never `secureActionClient`).

### Staff import (`/admin/upload-staff`)

Ingests a [Rippling](https://rippling.com) employee export into `staff` + `staff_employment`. The enum tuples the transform validates against (`LINE_OF_BUSINESS`/`ROLE`/`EMPLOYMENT_TYPE`) are **derived directly** from the pgEnums in `staff-schema.ts` (`*.enumValues`) in `src/lib/staff-import/types.ts`, not hand-copied, so they can never drift from the schema.

Code: client `src/components/admin/staff-import.tsx` (PapaParse + preview tables); pure transform `src/lib/staff-import/transform.ts`; server diff `src/lib/staff-import/plan.ts`; shared types/zod `src/lib/staff-import/types.ts`; actions `src/actions/admin/{previewStaffImport,commitStaffImport}.ts` (both `publicActionClient` + `assertLocalhost()` — local seeding must not require auth/staff).

### CSV column mapping

`Employee - ID`→`ripplingId`, `Employee`→`name`, `Work email`→`email`, `Start date`→`joinDate`, `Last day of work`→`terminationDate`, **`Manager - Work email`**→`managerEmail` (resolved to `managerId` at commit — see *Manager resolution* below). **Compensation:** `Annual base remuneration`→`base`, `Hourly Rate`→`hourlyRate`, `Target annual bonus`→`guaranteedBonus`, `Compensation currency`→`currency` (`discretionaryBonus` has no column — it always defaults to 0). Column lookup is whitespace/case-tolerant; dates accept ISO (`YYYY-MM-DD`) or US (`M/D/YYYY`), blank → null; money cells go through `parseNumber` (`src/lib/csv-import/parse.ts`) which strips currency symbols/separators and yields `null` on blank/unparseable/negative. A row missing any of id/name/email, with an unparseable *date*, **or missing a required comp value** (`base`/`hourlyRate`/`currency`) is **skipped** (surfaced, never persisted) under "Missing/invalid compensation: …". Note: because **both** `base` and `hourlyRate` are required, a row supplying only one of them is skipped. **`Target annual bonus` (`guaranteedBonus`) is the exception — it's optional:** a blank/whitespace-only cell defaults to `0` and the row is kept; only a *present-but-unparseable* value (parses to `null`) skips the row.

### Derivation rules (the non-obvious nuance — keep in sync with `transform.ts`)

- **`lineOfBusiness`:** Department=="Design" → `DESIGN`; else first match in Teams: core→`CORE`, commerce→`COMMERCE`, corporate→`CORPORATE`, fintech/crypto→`FINTECH`. **Unmappable → row skipped.**
- **`role`** (first match wins): Dept Design→`DESIGNER`; Dept Talent→`OPERATIONS`; Dept Sales→`SALES`; Dept Solutioning→`SOLUTIONS`; Title contains Delivery→`DELIVERY`; Architect→`ARCHITECT`; QA→`QA`; Engineer→`ENGINEER`; else `OPERATIONS`.
- **`employmentType`:** "Employment type name" contains "Hourly" → `HOURLY`, else `FULL_TIME`.
- **`isActive`** = no `terminationDate`. **`isBillable`** = role NOT in {LEADERSHIP, SALES, SOLUTIONS, OPERATIONS}. **`utilizationTarget`** = `isBillable ? 100 : 0` — the non-billable ⇒ 0 coupling runs through `normalizeEmploymentFacts` in the shared `src/lib/employment.ts`, the single home for the billability invariants (see *Bulk edit roles* below).
- **`isManagement` and `billableType` are never derived.** The transform doesn't set them; the commit **carries forward** the existing row's `isManagement` and `billableType` when writing a new effective-dated row (defaulting to `false`/`HUB` for brand-new people), so a re-sync never resets an in-app management flag or knocks a `GLOBAL` person back to `HUB`. (`LEADERSHIP` roles are likewise import-preserved — `plan.ts` keeps a person already in `LEADERSHIP` in that role rather than overwriting from the CSV-derived role.)
- **Compensation is required (except guaranteed bonus), not carried forward.** The transform reads the comp cells (`parseNumber`/`normalizeCurrency`) and **skips any row missing `base`/`hourlyRate`/`currency`** (see column map above); **`guaranteedBonus` is optional — a blank cell defaults to `0`** (a present-but-unparseable value still skips the row). So every surviving row already carries full comp, and `computeImportPlan` (`plan.ts`) diffs it straight from `incoming` with **no overlay from the current record**. A comp field counts as "changed" (→ new effective-dated row) whenever the CSV's value differs from the stored one. `discretionaryBonus` isn't imported, so it stays at its default (0) unless changed elsewhere. (Carry-forward survives only in **bulk-edit** — see below — where comp genuinely isn't part of the edit.)

### Manager resolution (`Manager - Work email` → `managerId`) — see [ADR 0026](../decisions/0026-staff-manager-self-reference.md)

The manager is named by **email** but people are matched by `ripplingId`, and `email` is **not unique** on `staff` — so the manager is carried through the pipeline as a **stable manager `ripplingId` reference**, resolved to a concrete `staff.id` only at commit (when create-ids exist). Pure logic lives in `src/lib/staff-import/managers.ts` (`MANAGER_EMAIL_HEADER`, `buildManagerEmailIndex`, `resolveManager`; unit-tested in `managers.test.ts`); `transform.ts` extracts `managerEmail` onto `NormalizedStaff`.

- **Two-pass email index (`plan.ts`).** `computeImportPlan` builds `email → ripplingId(s)` from **the incoming batch first, then the DB**: the batch covers a full-org / first import (the manager is a create in the same file), the DB lookup covers a partial import (the manager lives only in the DB). It also loads each matched person's *current* manager as a ripplingId so a manager change is detectable.
- **Manager change is an update.** The manager is a `staff` identity fact but is **not** in the `ComparableField` tuples (the compared value is a resolved ripplingId, not the raw email); it's tracked on its own `managerChanged` flag, and a **manager-only change counts as an update** (shown as a "manager changed" badge in the preview).
- **Three-state rule — blank clears; unresolvable/absent preserve.** `transform.ts` detects whether the CSV even has the `Manager - Work email` column and encodes three states on `managerEmail`, which `plan.ts` honours per existing person (a brand-new person has nothing to preserve, so it just takes the resolved manager, i.e. `null` when blank/unresolvable):
  - **filled + resolves** (`string`) → set/update the manager;
  - **column present + cell blank** (`null`) → **clear** the manager (`managerId = null`) — the *only* case that wipes a link, and it's authoritative;
  - **column entirely absent** (`undefined`) → the import carries **no** manager info, so **every existing link is preserved** (no resolution is even attempted). This is the fix for the old bug where an absent column read as "everyone blank" and wiped every manager.
- **Filled-but-unresolvable PRESERVES the existing link.** The three warning cases — `not_found` (no staff match), `ambiguous` (>1 match), `self` (names one's own email) — leave an existing person's current manager **untouched** and only emit a non-blocking warning, so a typo/ambiguous/self cell can't silently wipe a correct reporting line. (For a brand-new person with no existing manager, the link is simply left unset — there's nothing to preserve.)
- **Flagging is non-blocking (`managerWarnings`).** Those three cases surface as warnings while the person still imports (only the manager link is affected — and even that is preserved for existing people). Distinct from a `SkippedRow` — the row is not skipped. Warning messages state **only the problem** (e.g. `Manager email "x@y" matched no staff.`), since the outcome differs by case (kept for an existing person, unset for a new one); the UI's "Manager issues" section explains existing links are left unchanged and a blank cell clears. The commit result reports `managersLinked`.
- **`managersLinked` counts relationships actually established or changed this import** — creates that got a manager plus updates whose manager changed to a non-null value. A cleared link, or an unchanged pointer carried past an unrelated field change, does **not** count (so it isn't merely "every written row with a manager").
- **No cycle detection beyond self-reference** — intentional for a display-only field (matches `contacts.managerId`).

### Persistence (respects ADR 0007 effective dating)

Matching is by `ripplingId`. The server **recomputes the diff** (`computeImportPlan`) — the client preview is never trusted — and commits in one transaction:
- **Create** → new `staff` row + initial `staffEmployment` (`effectiveFromDate = joinDate ?? today`).
- **Update** → `staff` identity fields written in place; a **new** `staffEmployment` row (`effectiveFromDate = today`) inserted **only when an employment fact changed** (the "latest row wins" pattern), so identity-only edits don't churn employment history.
- **`managerId`** rides the same `staff` upsert's `set` clause (`managerId: sql\`excluded.manager_id\``). `commitStaffImport` resolves manager ripplingIds → staff ids across this batch's creates (freshly minted ids) + updates + a DB lookup for managers only in the DB. Because creates and updates share **one** batched `insert(staff)`, Postgres verifies the self-FK at statement end, so an **intra-batch reference (A managed by B, both new in the same file) resolves** (verified against Postgres).

**Gotcha:** duplicate `ripplingId`s *within one CSV* are not de-duped (a Rippling export shouldn't contain them); a genuine duplicate fails the commit transaction on the `staff.ripplingId` unique constraint.

### PTO import (`/admin/upload-pto`)

Mirrors the staff importer (clone of `staff-import.tsx`), ingesting a Rippling **leave** export into `staff_pto`. Code: client `src/components/admin/pto-import.tsx`; pure transform `src/lib/pto-import/transform.ts` (reuses the same `getField`/`parseDate` helpers as staff-import); server diff `src/lib/pto-import/plan.ts`; shared types/zod `src/lib/pto-import/types.ts`; actions `src/actions/admin/{previewPtoImport,commitPtoImport}.ts` (+ `ptoImport.schema.ts`).

The defining difference from the staff importer: rows can **delete** as well as upsert, and matching is **two-level**. See [ADR 0009](../decisions/0009-pto-import-cancel-as-delete.md) for *why* cancellations delete and what "destructive re-sync" implies.

#### CSV column mapping

`Leave request ID`→`staffPto.ripplingId` (the PTO record key), `Employee - ID`→resolves the staff member (`staff.ripplingId`), `Employee`→display name, `Start date`→`startDate`, `Leave end date`→`endDate`. Same whitespace/case-tolerant lookup and ISO/US date parsing as staff-import. A row missing `Leave request ID` or `Employee - ID` is **skipped**.

#### Derivation rules (keep in sync with `transform.ts`)

- **`Leave request status` → action + `isPending`:** `APPROVED` → upsert, `isPending=false`; `Pending` → upsert, `isPending=true`; `REJECTED`/`CANCELED`/`CANCELLED` → **delete** action (remove the record if it exists). Anything else → row **skipped** (unrecognized status).
- **`Leave policy custom name` → `type`** (case-insensitive substring, first match): "unlimited vacation"→`VACATION`; "family medical leave"/"sick leave"→`SICK_LEAVE`; "statutory holiday"→`STATUTORY_HOLIDAY`; "company retreat"→`COMPANY_RETREAT`; "religious holiday"→`RELIGIOUS_HOLIDAY`; "bereavement leave"→`BEREAVEMENT_LEAVE`; "parental leave"/"pregnancy leave"/"paternity leave"→`PARENTAL_LEAVE`; "jury duty"→`JURY_DUTY`; "leave of absence"→`LEAVE_OF_ABSENCE`. **No match → row skipped** ("Unrecognized leave type").
- For `upsert` rows, a missing/unparseable `Start date` or `Leave end date` **skips** the row. `delete` rows skip date/type derivation entirely (a cancelled request may have neither) — they carry only the leave-request id.

#### Persistence (`computePtoImportPlan` → `commitPtoImport`)

Two-level diff: resolve staff by `Employee - ID`, then match the PTO record by `Leave request ID`. The plan buckets are `creates` / `updates` (changed-field marked) / `deletes` / `unresolved` / `unchanged` / `ignoredCancellations`. Commit recomputes server-side and runs one transaction: insert creates, `update` changed rows by **id**, delete cancelled/rejected rows by **leave-request id**.

**Gotchas:**
- **Unresolved, not skipped, when staff is missing.** An upsert row whose `Employee - ID` matches no `staff` row is surfaced as `unresolved` (it can't be inserted without the staff FK) — distinct from the client-side `skipped` rows. Import staff first.
- **Delete is keyed independently of staff.** A `delete` row removes any existing record by its leave-request id even if the employee can't be resolved; with no matching record it's counted as `ignoredCancellations` (a cancel for leave we never imported is a no-op).
- Same intra-CSV duplicate-`ripplingId` guard as staff-import — the transform skips a second occurrence of a `Leave request ID` to avoid a unique-constraint rollback.

### Bulk edit roles (`/admin/bulk-edit-roles`)

Edits the latest employment row for many staff at once, the only in-app way to maintain employment facts (the importers seed them; this corrects/advances them). A filterable, sortable [TanStack](https://tanstack.com/table) table where each row is one staff member's current employment facts — line of business, role, employment type, `isBillable`, `utilizationTarget`, `billableType`, `isManagement` — editable inline (selects, switches, a clamped 0–100 utilization input with 0/50/100 presets, and a `HUB`/`GLOBAL` segmented control for `billableType`). Toggling `isManagement` **on** auto-sets `isBillable=false` and `utilizationTarget=0`. These are two distinct rules, both defined once in `src/lib/employment.ts` (see below): the **management→non-billable default** is a *client-side convenience* applied only at the moment management is enabled (the admin can re-enable billable afterward — a billable manager is valid), whereas the **non-billable ⇒ `utilizationTarget=0` invariant is a standing rule and IS server-enforced** (the commit's schema `.refine` calls `isEmploymentInvariantSatisfied`, so a crafted payload can't slip a non-zero target onto a non-billable row). A floating save bar shows the changed-staff count and a confirmation dialog summarises per-staff field diffs before committing.

Code: read `src/actions/staff/getStaffEmploymentForEdit.ts` (server-only; latest employment row per staff via the same two-query, latest-per-staff reduction as `getStaffDirectory` — includes inactive staff so the table can filter on "active"; staff with **no** employment row are omitted, there's nothing to edit); client `src/components/admin/bulk-edit-roles.tsx`; commit `src/actions/admin/commitBulkEditEmployment.ts` (+ `bulkEditEmployment.schema.ts`). The commit is `publicActionClient` + `assertLocalhost()` like the importers.

**The save behaviour is the load-bearing part — it deliberately extends ADR 0007:**

- **Effective date left BLANK → UPDATE the latest employment row in place** (semantics: correcting a data-entry mistake — no new historical fact). This is the *new* path; everywhere else, employment changes only ever insert.
- **Effective date provided → INSERT a new effective-dated row per staff** (semantics: a real change effective on that date). The date must be **strictly after** that staff's latest `effectiveFromDate` (re-validated server-side; rejected with the affected names otherwise).

The payload is never trusted: the commit re-reads each staff's latest row, **drops no-op changes** (a row where no editable fact differs), and enforces the billable invariant (`isBillable=false ⇒ utilizationTarget=0`). That invariant now lives in **one** pure, client-importable module — **`src/lib/employment.ts`** (`isEmploymentInvariantSatisfied` + `normalizeEmploymentFacts`), mirroring how `line-of-business.ts` / `currency.ts` are the single source for their enums. It's imported by everywhere that authors or validates billability: `staff-import/transform.ts` and `staff-import/types.ts` (derive-on-import), `bulkEditEmployment.schema.ts` (the server `.refine`), and `bulk-edit-roles.tsx` (as-you-type derivation) — so the definition can't drift. See [ADR 0007 — In-place correction (bulk edit)](../decisions/0007-staff-employment-effective-dating.md).

**Compensation is not editable here (import-only).** When bulk-edit inserts a *new* effective-dated row it reads the latest row's comp and **carries it forward**, so advancing someone's role/LoB never clears their salary; the in-place-correction path leaves the existing row's comp untouched. See [ADR 0020](../decisions/0020-compensation-effective-dated-import-only.md).

## Key flows

- **Login gating** — on first authenticated request `getCurrentStaff` resolves user → staff (auto-linking by email once), and the `(app)` layout admits only `ok`; `not_setup`/`incomplete` are bounced to the single `/profile-setup` block screen. See [flows.md](../flows.md).
- **Employment change** — insert a new `staff_employment` row with a new `effectiveFromDate` rather than mutating the prior row, preserving history. The **one** exception is correcting a wrong latest row in place via bulk-edit with a blank effective date (no new fact) — see *Bulk edit roles* above and [ADR 0007](../decisions/0007-staff-employment-effective-dating.md).
- **Profile read (by id — `/profile` and `/staff/[id]`)** — the per-person reads are now **parameterized by `staffId`** and not ownership-scoped: `getStaffProfile(staffId)` (`src/actions/staff/getStaffProfile.ts`), `getStaffHistory(staffId)`, `getStaffPto(staffId)`, `getStaffAvatar(staffId)`. `getStaffProfile` returns latest employment via the ADR 0007 ordering — **including the compensation columns** (`base`/`hourlyRate`/`guaranteedBonus`/`discretionaryBonus`/`currency`), which `ProfileView` renders in a **Compensation card only when its `canViewCompensation` prop is true** — and is wrapped in `React.cache` so `/staff/[id]`'s `generateMetadata` and page body share one query. Both pages wire the prop: `/profile` hard-codes `true` (own profile), `/staff/[id]` computes `canViewCompensation(user, id)` (alongside `canEditStaff`) and passes it both to `ProfileView` **and** as `getStaffHistory`'s `includeCompensation` arg. See [ADR 0020](../decisions/0020-compensation-effective-dated-import-only.md) and [ui.md](../ui.md). `getStaffProfile` also **self-joins `staff`** (drizzle `alias`) on `managerId` to project `{ managerId, managerName }`; `ProfileView` renders a read-only **"Reports to <link>"** line in the identity header linking to `/staff/[managerId]` (omitted when there's no manager). The manager is set by the import only — no editor (see [ADR 0026](../decisions/0026-staff-manager-self-reference.md)). `getStaffAvatar` reads the linked auth `user.image` (separate query because the avatar lives on `user`, not `staff`; null when unlinked).
  - **Self read** — `getMyProfile`/`getMyHistory`/`getMyPto` are now **thin wrappers**: each resolves the current staff id via `getCurrentStaffId()` (`src/actions/staff/getCurrentStaffId.ts`, `React.cache`d, `staff.userId = user.id` → id or null) and delegates to the `getStaff*` core. `MyProfile` is a back-compat alias of `StaffProfile`. The page never touches `db` (ADR 0010). See [ui.md](../ui.md) (`/profile`).
  - **Why not ownership-scoped:** the directory and `/staff/[id]` deliberately show other people, so profile/history/avatar reads can't filter by `userId`. Access is the `(app)` layout's session+staff gate, plus a capability check where data is sensitive — `getStaffPto` requires `pto.review` for non-owners (returns `null` otherwise). This softens ADR 0010's "inherently self-scoped" property for cross-person data — see *Authorization* below, [permissions.md](./permissions.md), and [ADR 0014](../decisions/0014-rbac-better-auth-access-control.md).
- **Directory ("Browse staff", `/staff`)** — `getStaffDirectory()` (`src/actions/staff/getStaffDirectory.ts`) returns one `StaffDirectoryEntry` per staff member (identity + `isActive` + avatar + **`skills: StaffSkill[]`** + latest employment facts; employment fields null when no history). **Two queries, no N+1:** all staff (left-joined to `user` for the avatar; `skills` comes straight off the `staff` row), then the *entire* `staff_employment` table ordered newest-first and reduced in JS to keep the latest row per `staffId`. Includes inactive staff so the UI's "active only" toggle (defaults ON) can hide/show them client-side. It also exports `staffDirectoryFilterOptions` (line-of-business / role / employment-type enum values) so the page gets filter options without importing the Drizzle schema. All search + filtering is **client-side** over this single server fetch (`StaffDirectory` component) — including the **skills filter** (multi-select, AND semantics, with an optional minimum-proficiency segment), which runs in-memory via the pure `matchesSkillFilter` (see *Skills* below). No URL params, no per-filter server query. See *Authorization* (whole-table scan tradeoff) and [ui.md](../ui.md).
- **Edit (links + client intro + résumé)** — the `secureActionClient` mutations `updateStaffLinks` (`src/actions/staff/updateStaffLinks.ts`), `updateStaffClientIntro`, and `updateStaffResume` (`src/actions/staff/updateStaffResume.ts`) all take an explicit **`staffId`** and edit that row by id, and all declare `metadata({ action, authorize: authorizeStaffEdit })` — the `authorizeStaffEdit` hook (`src/actions/staff/canEditStaff.ts`) reads `clientInput.staffId` and runs `canEditStaff` before the body (owner always; others need `staff.edit` — see *Authorization* and [permissions.md](./permissions.md)). The bodies hold no authz call. `.returning()` confirms a row matched (→ `UserSafeActionError` if not). `updateStaffClientIntro`/`updateStaffResume` stamp their `*UpdatedAt` column. All revalidate `/profile` and `/staff/${staffId}`. Schema rules: URL fields empty-string→null else valid URL; client intro trimmed, max 2000, empty→null; résumé trimmed, **max 50,000 chars**, empty→null; schemas in their own `.schema.ts` files so the dialogs import them for the resolver (never export a schema from a `'use server'` file — `.claude/rules/forms.md`). The edit dialogs (`edit-links-dialog.tsx`, `edit-client-intro-dialog.tsx`, `edit-resume-dialog.tsx`) take a `staffId` prop so `/profile` and `/staff/[id]` share them. UI in [ui.md](../ui.md).
- **Résumé PDF upload (parse only, no storage)** — the résumé dialog has an "Upload PDF" button that reads the file to base64 client-side (guarded to ~6 MB), calls the `parseResumePdf` `secureActionClient` action (`src/actions/staff/parseResumePdf.ts`), which extracts text server-side via the **`unpdf`** dependency and returns it for the user to review in the textarea; **no DB write and the PDF is never stored** (it only parses the caller's own upload, so it needs no `staff.edit` gate — the gated write is `updateStaffResume`). Scanned/image-only PDFs (no extractable text) and unreadable files return a `UserSafeActionError` telling the user to paste text manually. Saving goes through `updateStaffResume` as normal. Base64 inflates payload ~33%, so `next.config.ts` raises `experimental.serverActions.bodySizeLimit` to `"8mb"` to fit a ~6 MB PDF (client and server caps mirror each other). See [ADR 0013](../decisions/0013-resume-pdf-parse-not-store.md).
- **History feed (`/profile` + `/staff/[id]`)** — `getStaffHistory(staffId, includeCompensation = false)` (`src/actions/staff/getStaffHistory.ts`, with `getMyHistory` as its self-resolving wrapper) is a server-only read returning a **category-agnostic** `HistoryEntry[]` (`{ id, date, category, summary }`, `HistoryCategory = "EMPLOYMENT" | "ALLOCATION"`), newest first, for the given staff id (not ownership-scoped — see *Authorization*). **Today only `EMPLOYMENT` entries are mapped** (from `staffEmployment` rows, ordered per ADR 0007), with **compensation folded into the same entry's summary** for authorized viewers (see below); `ALLOCATION` is still a stub. Entries are pushed into one array and merged by a single stable sort (date strings are `"YYYY-MM-DD"`, so lexicographic compare is chronological and equal-date entries keep per-source order). Adding a source needs no change to the consuming UI (`HistorySheet` is purely presentational). Rendered by the `HistorySheet` drawer — see [ui.md](../ui.md). For *why* the feed is category-agnostic and merged server-side, see [ADR 0011](../decisions/0011-category-agnostic-history-feed.md).
  - **Comp amounts are gated, folded into the employment entry.** `includeCompensation` (the caller passes `canViewCompensation`'s result — `true` for own profile) controls whether comp amounts are appended to each employment entry's summary **at all**; because `HistorySheet` is a client component, this gate is at the *read*, so salary never ships to an unauthorized browser. When on, each entry's summary gains `Base … · Hourly … · Guaranteed bonus … · Discretionary bonus …` (zero bonuses omitted). There is **no separate `COMPENSATION` category** — comp rides the employment entry it changed with. See [ADR 0020](../decisions/0020-compensation-effective-dated-import-only.md).
- **Edit (skills)** — a dedicated page `/staff/[id]/skills` (not a dialog); see *Skills* above.
- **Manual of Me** — a guided, autosaving survey on its own page `/staff/[id]/manual-of-me`, stored in the generic `responses` table; readable by any profile viewer, editable via `authorizeStaffEdit`. See *Manual of Me* above and [ADR 0028](../decisions/0028-generic-responses-table-app-validated-question-ids.md).
- **Profile maintenance** — keep skills, availability, and rates current. Links, client intro, résumé, **skills**, and the **Manual of Me** survey have a self-edit UI today; **compensation** is maintained via the Rippling import only (no edit UI — see [ADR 0020](../decisions/0020-compensation-effective-dated-import-only.md)); availability is still proposed.
- **Skill matching** — Allocations querying skills to find suitable People is still proposed. Note the inline-jsonb storage means this can't be an indexed join yet — see [ADR 0018](../decisions/0018-skills-inline-jsonb-catalogue.md) for the tradeoff and the migration path if/when matching lands.

## Connects to

- **Allocations** — role, line of business, and `utilizationTarget` drive staffing and capacity; `staff_pto` spans reduce available capacity over their date range.
- **Timesheets** — `isBillable`/`billableType` turn logged hours into billing; **compensation** (cost side — base/hourly on `staff_employment`) feeds margin. External **charge** rates are still proposed.
- **Performance** — profile is the subject of reviews.

## Open questions

- **External charge rates** (what we bill a client for a person) — still unmodelled. Would follow the same effective-dated pattern; the *cost* side (compensation) now lives on `staff_employment` (see below).
- **Cross-tenure identity:** no `person` entity links a human's multiple `staff` rows across leave/rehire stints (see *Nuances*). If tenure-spanning reporting or "have we worked with them before" becomes a need, this is the gap to fill.

> **Resolved:** compensation model + sensitivity — compensation is **effective-dated columns on `staff_employment`** (not a separate table), populated **import-only** and **required** (rows missing comp are skipped; no carry-forward on import), and **view-gated** by `staff.viewCompensation` (own always visible), shown on the profile Compensation card + folded into the history feed's employment entries. See [ADR 0020](../decisions/0020-compensation-effective-dated-import-only.md).

> **Resolved:** skill taxonomy + proficiency scale — skills are an inline `jsonb` list on `staff` picked from a hardcoded catalogue (`src/lib/skills.ts`), with a fixed 3-value proficiency scale. See *Skills* above and [ADR 0018](../decisions/0018-skills-inline-jsonb-catalogue.md). (Deliberately not the normalized many:many table originally proposed.)

> **Resolved:** linking `staff` ↔ auth `user` is now the nullable, unique `staff.userId` FK (`onDelete: set null`) — an optional 1:1 link, null until first sign-in. Identity is no longer keyed off email (`staff.email` is no longer unique). See Key entities above.
