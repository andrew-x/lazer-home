# Domain: CRM

**Status: growing.** Companies, contacts, and opportunities (data, reads, create, edit, and `/companies` + `/contacts` + `/opportunities` UIs) all exist. **Companies and contacts are now full CRUD (create/read/edit; no delete yet)** — each has a **detail page** at `/companies/[id]` and `/contacts/[id]` — the app's **first drill-down detail views** and its **first tabbed pages** (see the UI section below) — carrying an **Edit** affordance (for `crm.edit` holders) that opens a dialog wired to `updateCompany`/`updateContact`, including an optional **staff owner** assignment (the accountable staff member). Companies and contacts now live on **two separate pages** (`/companies` and `/contacts`) — the old combined "Companies & Contacts" page was split ([design spec](../superpowers/specs/2026-07-09-split-companies-contacts-design.md)). `/opportunities` is now a **drag-and-drop kanban board** (not a table), with 14 leaf statuses organized into ordered pipeline groups — see [ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md). Opportunities now carry a **required line of business** (the shared enum, [ADR 0025](../decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)), and their table + junctions now live in **their own schema file** `src/lib/db/opportunities-schema.ts` (split out of `crm-schema.ts`, which keeps only companies + contacts). Clicking a card opens a **detail drawer** — a wide (`max-w-3xl`) two-tab read-first view (**Info** + **Project plan**) whose Info fields **edit one at a time in place** (per-field pencil → Confirm/Cancel), covering status, line of business, next steps, owners, source, contacts, referral sources — **not** the company. Projects now exist too ([projects.md](./projects.md)), and the **Opportunity → Project handoff is now built**: `projects.opportunityId` is populated when a project is created from an opportunity, and a **delivery-stage rule** requires a linked project before an opportunity can advance to Allocating or later ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)). Manages the companies and people we deal with (clients, partners, prospects) and the pipeline deals we're trying to win, all of which feed delivery.

## Purpose

Track who we sell to and work with, and (eventually) what we're trying to win, so won work flows cleanly into Projects and Allocations.

## Key entities

- **Company** (built) — an organisation we deal with. **`isPartner` is a standalone flag** marking whether the company is a partner (default `false`); it is **not** a client-vs-partner dichotomy — a company may be neither a client nor a partner, and nothing requires it to be either. We deliberately modelled this as *Company* rather than the narrower *Client* the early design assumed — see [ADR 0015](../decisions/0015-crm-company-over-client.md). Fields: `name` (required), `websiteUrl` (optional), `isPartner`, and an optional **`ownerId`** — a nullable FK → **`staff`** (`onDelete: set null`), the staff member accountable for the relationship; cleared (not blocked) if that staff row is removed. This is a CRM company/contact ↔ staff link (mirrors `opportunityOwners.staffId`; see [Connects to](#connects-to)). Table `companies`, id prefix `company`.
- **Contact** (built) — a person, optionally attached to a Company. Fields: `firstName`/`lastName` (required), `email` (required, **unique**), `phone` (optional; shown as a `tel:` link in the contacts table), `companyId` (optional FK → `companies`, `onDelete: set null`), `role` (optional free-text job title, e.g. "CTO"), `linkedinUrl` (optional; validated by the shared `optionalUrl` schema, same as company `websiteUrl`; shown as an external "Profile" link in the contacts table), and **`managerId`** — an optional **self-referential** FK → `contacts.id` (`onDelete: set null`), "managed by" another contact. This is the repo's **first self-referential FK**; by business rule a manager must be an existing contact **at the same company**, enforced app-side (not in the DB, via `assertValidManager` in `contactChecks.ts`) — see [ADR 0022](../decisions/0022-contact-manager-self-reference.md). Also carries an optional **`ownerId`** — a nullable FK → **`staff`** (`onDelete: set null`), the accountable staff member; cleared if that staff row is removed (same set-null convention as `companies.ownerId`). Table `contacts`, id prefix `contact`. Schema in `src/lib/db/crm-schema.ts`; see [../data-model.md](../data-model.md).
- **Opportunity** (built) — a pipeline deal that always belongs to a Company. Fields: `name` (required), required `companyId` (FK → `companies`, **`onDelete: restrict`** — a company with live opportunities can't be deleted, unlike its optional/set-null contacts), `source` + `status` (pgEnums), **required `lineOfBusiness`** (the shared `lineOfBusinessEnum` — which practice the deal belongs to; a project created from it defaults to the same, [ADR 0025](../decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)), optional free-text `nextSteps`, and **`position`** (see below). Table `opportunities`, id prefix `opp` — **now in its own schema file** `src/lib/db/opportunities-schema.ts` (split out of `crm-schema.ts`, which keeps only `companies` + `contacts`; [ADR 0025](../decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)). Its people are modelled with **four junction tables** (see below): related `contacts`, `owners` (→ `staff`), and referral sources split into `source_contacts` and `source_staff`. An Opportunity produces a Project once it reaches delivery stages; the link column lives on the **project** side (`projects.opportunityId`, nullable FK → `opportunities`, `onDelete: restrict`, **at most one project per opportunity** — enforced by a **partial unique index** on the FK, [ADR 0019 amendment](../decisions/0019-project-opportunity-link.md)), now **populated by the handoff flow** ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)), so a delete of an opportunity with its live project is blocked. See [ADR 0019](../decisions/0019-project-opportunity-link.md).
  - **`source`** (`opportunity_source`): `inbound`, `farming`, `extension`, `change_request`, `staff_referral`, `contact_referral`.
  - **`status`** (`opportunity_status`): a **flat 14-value leaf enum in strict pipeline order** — `maturing`, `lead`, `qualifying`, `scoping_awaiting_info`, `scoping`, `scoping_reviewing`, `allocating_awaiting_profiles`, `allocating_introing_profiles`, `negotiating`, `closing_awaiting_contracts`, `closing_redlining`, `closing_awaiting_signatures`, `closed_won`, `closed_lost`. The DB knows only leaves; the two-level **grouping** (below) is derived in code. (Restructured from the old flat 7-value enum; migration `0020` remaps legacy `closing` → `closing_awaiting_contracts`.)
  - **Pipeline groups** — `src/lib/opportunity-pipeline.ts` (a pure, client-importable module, mirroring `line-of-business.ts`) defines `OPPORTUNITY_GROUPS`: ordered groups → their leaf substatuses. In order: **Maturing, Lead, Qualifying** (single-status), **Scoping** (awaiting info / scoping / reviewing), **Allocating** (awaiting profiles / introing profiles), **Negotiating**, **Closing** (awaiting contracts / redlining / awaiting signatures), **Won** (`closed_won`), **Lost** (`closed_lost`). A **module-load assertion** (+ `opportunity-pipeline.test.ts`) requires the flattened groups to equal `OPPORTUNITY_STATUSES` in the same order, so they can't drift. Also exports `resolveTargetStatus` (which leaf a card takes when dropped on a collapsed group), `computePosition` (fractional-index midpoint), and **`requiresProject(status)`** — true from the **Allocating** group onward (Allocating → Negotiating → Closing → Won), Closed – Lost excepted: the delivery-stage rule that an opportunity can't advance without a linked project ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)). See [ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md).
  - **`position`** (`doublePrecision NOT NULL DEFAULT 0`) — a **single global fractional index** for manual kanban ordering. Cards in a column sort by `(position, createdAt)`; a drag writes the midpoint between its new neighbours (one-row update), a new opportunity gets `max(position)+1` (lands at the end of its column). Index `opportunities_status_position_idx` on `(status, position)`. No fractional-index rebalancing — an accepted limitation at this scale ([ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md)).
  - Both enum value tuples (`OPPORTUNITY_SOURCES`/`OPPORTUNITY_STATUSES`), their union types (`OpportunitySource`/`OpportunityStatus`), and their `SOURCE_LABELS`/`STATUS_LABELS` are declared **once** in `src/lib/opportunity.ts` (a pure, client-importable module — no `db`/drizzle, no UI, mirroring `line-of-business.ts`/`currency.ts`), and everything imports them **directly** from there: the `pgEnum` in `opportunities-schema.ts`, the zod schema, and the display components. This fixed the old layering where the tuples lived in `createOpportunity.schema.ts` (imported *upward* by the pgEnum) and the labels in `opportunity-display.ts`. The **back-compat re-export shims are now gone** — `createOpportunity.schema.ts` no longer re-exports the enums, and `opportunity-display.ts` keeps only `STATUS_SELECT_LABELS` (the group-qualified dropdown labels it derives). Per-leaf display labels are `STATUS_LABELS`; group labels live in `opportunity-pipeline.ts`.

## What's built

- **Schema** — `src/lib/db/crm-schema.ts` (`companies`, `contacts`) + `src/lib/db/opportunities-schema.ts` (`opportunities` + four junction tables + the `source`/`status` pgEnums + `lineOfBusiness`), both barrelled by `src/lib/db/schema.ts`. The opportunities file imports `companies`/`contacts` from `./crm-schema` and `staff`/`lineOfBusinessEnum` from `./staff-schema` ([ADR 0025](../decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)). Migrations `drizzle/0012_spotty_mysterio.sql` (companies/contacts), `drizzle/0013_clean_firelord.sql` (adds `contacts.phone`), `drizzle/0014_nostalgic_crusher_hogan.sql` (opportunities + junctions + enums), `drizzle/0020_unique_blue_shield.sql` (**hand-edited**: recreates `opportunity_status` as the 14-value leaf enum, remapping legacy `closing` → `closing_awaiting_contracts`; adds `position` + backfills it via `row_number()` per status; adds `opportunities_status_position_idx`), `drizzle/0021_glamorous_goliath.sql` (adds `contacts.linkedinUrl` + self-referential `contacts.managerId`), `drizzle/0024_harsh_diamondback.sql` (adds `opportunities.line_of_business` — backfilled to `CORE` via a temporary default, then dropped, so it's NOT NULL with no default). The schema split itself is a pure code reorg (no migration).
- **Junction tables** — the repo's **first** many-to-many join tables: `opportunity_contacts`, `opportunity_owners` (→ `staff`), `opportunity_source_contacts`, `opportunity_source_staff` (→ `staff`). All follow one convention (documented in [../data-model.md](../data-model.md#junction-tables--the-first-many-to-many-pattern)): a **surrogate `text` PK** via `generateId` (no composite PKs — repo convention), a **`unique(...)` on the FK pair** for set-semantics (no duplicate links), an **`index(...)` on the non-opportunity FK** for reverse lookups, and **both FKs `onDelete: "cascade"`** (a link is meaningless without both endpoints).
- **Server layer** — `src/actions/crm/`:
  - `getCompaniesPage.ts` / `getContactsPage.ts` — server-only reads (per ADR 0010), **server-side offset/limit pagination** with a `count` and a page clamped into range (out-of-bounds query params can't return an empty page). Both share `src/lib/pagination.ts`, which exports `CRM_PAGE_SIZE` (20), `clampPage` (the range-clamp arithmetic — there's always ≥1 page, even when empty), and the `Page<T>` envelope (`{ rows, total, page, pageSize, pageCount }`) they return. Contacts left-join companies to resolve `companyName` (company is optional) **and self-join `contacts` (drizzle `alias`) to resolve `managerName`** (manager is optional too); `ContactRow` carries `linkedinUrl`, `managerId`, `managerName`.
  - `getCompanyDetail.ts` / `getContactDetail.ts` — server-only reads backing the two detail pages (per ADR 0010; **not** `'use server'`, so pages `await` them directly during SSR). Each fetches the entity's core fields, then its related collections in a `Promise.all`, projected to just the columns the view renders; both return `null` on an unknown id (page → `notFound()`). No pagination (an entity's own footprint is small). Neither adds a permission beyond the `(app)` gate — consistent with all CRM reads being open.
    - `getCompanyDetail` returns core fields + three collections tied to `companyId`: `opportunities`, `projects`, and `contacts` (the company's people directory).
    - `getContactDetail` returns identity + employer + manager (both optional, resolved via **left joins**, with a `contacts` self-join `alias` for `managerName` like `getContactsPage`), plus the contact's CRM footprint split into `referredOpportunities` (via `opportunitySourceContacts`), `involvedOpportunities` (via `opportunityContacts`, **minus** any already in referred so nothing double-lists), and `referredProjects`. **Contacts don't link to projects directly**, so "projects" is *derived*: projects whose `opportunityId` is one of the contact's referred opportunity ids (`inArray`). All opportunity/project rows inner-join their company so the cross-company view can name and link the employer.
  - `getOpportunitiesBoard.ts` — **non-paginated** server-only read backing the kanban board (a consultancy's pipeline is small, so no paging). Returns *all* cards `{ id, name, companyId, companyName, source, status, ownerNames, hasProject, position, createdAt }` ordered by `(position, createdAt)`; inner-joins companies for `companyName` (company is required) and resolves owner names **and** the `hasProject` flag via **two grouped follow-up queries** over all card ids (no N+1). `hasProject` (any linked `projects.opportunityId`) drives the delivery-stage requirement; `companyId` lets the board prefill the create-project prompt. `createdAt` is returned as epoch millis so the client's `(position, createdAt)` tie-break matches the server's. (Replaced the removed `getOpportunitiesPage.ts` — opportunities are no longer paginated.)
  - `getOpportunity.ts` — **server-only** read for the detail drawer: an opportunity's core fields, its company, the four people junctions each as `{ id, name }[]` (shaped for the drawer's comboboxes), and its linked projects. Null if the id is unknown.
  - `loadOpportunityDetail.ts` — a thin **`'use server'`** action wrapping `getOpportunity` for the client-triggered drawer load (the **interactive-read exception** to the server-only rule, like `searchStaff`), gated `crm.edit` — the drawer is edit-only, so this can't leak detail past the write gate. **Named `load*`, not `get*`, deliberately: the `get*` prefix is reserved for server-only reads** (per `.claude/rules/server-actions.md`), so a client-triggered `'use server'` read uses a different verb.
  - `updateOpportunity.ts` (+ `.schema.ts`) — the drawer's **edit** action, gated `crm.edit`. One transaction: updates the row (name/source/status/**lineOfBusiness**/nextSteps — **not** the company) and **replaces the four people junctions wholesale** (delete + re-insert via the shared `writeOpportunityLinks` — see below). Enforces the delivery-stage rule **on status transitions only** (see the delivery-stage enforcement bullet). `revalidatePath("/opportunities")`.
  - `updateOpportunityPosition.ts` (+ `.schema.ts`) — the kanban **move** action, gated `crm.edit`. Input `{ id, status, position }` (an absolute destination the board computes client-side from the drop's neighbours); a **single-row update** with no server-side renumbering, then `revalidatePath("/opportunities")`. See [ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md).
  - **Delivery-stage enforcement** — the invariant is *an opportunity in a delivery stage has a linked project*, enforced server-side at the **entry points** (so it can't be bypassed by hitting an action directly): `createOpportunity` **rejects** creating in a delivery stage (a new opportunity can't have a project yet), and `updateOpportunityPosition`/`updateOpportunity` guard **transitions** — each fetches the current status and fires only when the status is *changing into* a requiring stage without a project (`newStatus !== currentStatus && requiresProject(newStatus) && !opportunityHasProject(id)`), throwing a `UserSafeActionError`. This is **transition-based, not state-based**: editing (rename/owners/next-steps) or reordering within a delivery column no longer trips the guard (the earlier every-edit check over-fired). `opportunityHasProject.ts` is the indexed existence check (server-only). The client block is just UX. See [ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md).
  - `searchCompanies.ts` / `searchContacts.ts` / `searchStaff.ts` — type-ahead pickers, `secureActionClient` gated `crm.edit` (they back CRM entry, itself a write, so the gate stops them being used to enumerate the roster past the page gate). Each returns up to `SEARCH_LIMIT` `{ id, name }` by `ilike` match; **a blank query returns nothing** (search only runs once the user types). All three share the query surface via `src/lib/search.ts`, which exports `SEARCH_LIMIT` (10) and `searchQuerySchema` (a single `query` string **trimmed on parse**, so whitespace collapses to the blank-query short-circuit). The `searchStaff` / `searchCompanies` **query bodies** are further shared with the Projects domain via `searchStaffByName` / `searchCompaniesByName` in `src/actions/shared/entitySearch.ts` — the CRM and projects search actions each wrap the same query with their own permission gate (`crm.edit` vs `projects.edit`); only `searchContacts` keeps its own body (different columns). Nuances: `searchStaff` filters `isActive = true` (**only active staff**), and `searchContacts` matches **first name / last name / email** (not just name) and takes an **optional `companyId` filter** that scopes results to one company — the contact manager picker passes it so a manager can only be a colleague at the same company. All also share the LIKE-metacharacter escaper `escapeLike` in `src/lib/like.ts` (extracted from `searchCompanies`).
  - `createOpportunity.ts` — gated `crm.edit`. **Rejects creating in a delivery stage** (any `requiresProject(status)` — Allocating onward): a brand-new opportunity can't have a linked project yet, so that state is unreachable; the error points the user to create it earlier and add a project as it advances. Otherwise one **transaction**: reads `max(position)` and inserts the opportunity with `position = max+1` (so it lands at the end of its board column), then writes the four people-junctions via the shared `writeOpportunityLinks` (see below). Related entities (company, contacts) are created via their own actions first; this only consumes their ids. `revalidatePath("/opportunities")`.
  - `opportunityLinks.ts` — **`writeOpportunityLinks(tx, opportunityId, ids)`**, a shared helper (not an action) that inserts the four opportunity people-junction rows in one transaction, **deduping each id list** first so a repeat can't trip the junction `unique` index. Used by **both** `createOpportunity` (fresh insert) and `updateOpportunity` (after it deletes the existing rows), so the two always write the identical junction shape — the create/update junction logic is no longer duplicated.
  - `createOpportunity.schema.ts` — the shared zod schema (its field shape is reused by the update schema; it imports the enum tuples **directly** from `@/lib/opportunity` — the old back-compat re-export shim is gone). A `superRefine` enforces the conditional referral rules: `source = staff_referral` ⇒ ≥1 `sourceStaffIds`; `source = contact_referral` ⇒ ≥1 `sourceContactIds`.
  - `createContact.ts` / `updateContact.ts` (+ `.schema.ts`) — create and edit a contact, both gated `crm.edit`. `createContact` is a **single `db.insert(contacts)`** (no transaction) and **returns `{ id }`** (not `{ ok: true }`) so an inline-created contact can be immediately selected in the opportunity form; `updateContact` is a `db.update(...).returning()` by `id` (an empty `returning` → user-safe "That contact no longer exists"). **Two checks are shared between them via `contactChecks.ts`** (so create/update can't drift): `assertValidManager({ managerId, companyId, selfId? })` — a manager must be an existing contact at the *same* company (and, on update, never the contact itself); the UI picker enforces the same rule, but this re-check (a DB lookup) stops a hand-crafted request creating a cross-company, self-referential, or dangling management link ([ADR 0022](../decisions/0022-contact-manager-self-reference.md)) — and `mapContactEmailConflict(error)` — maps the Postgres unique-violation on `contacts_email_unique` to a user-safe "A contact with that email already exists." (returns `never`, used as the whole `catch` body), via the shared **`isUniqueViolation(error, constraint)`** helper in `src/lib/db/unique-violation.ts` (a reusable SQLSTATE-`23505`-on-named-constraint check, extracted from the old inline check in `createContact` so the CRM write actions share it). Both actions also write an optional **`ownerId`** (the owning staff member — added with the edit feature). Creating a company inline from the contact form is **persist-first** — the new company is saved via `createCompany` first (through `CompanyComboboxField`'s inline dialog), then its id is passed here as `companyId`. This mirrors opportunities and replaced an earlier design where `createContact` created the company itself in a nested `db.transaction` (that `newCompany` path and the transaction are gone). `createContact` `revalidatePath("/contacts")`; `updateContact` revalidates `/contacts` and `/contacts/${id}`. Input schemas live in sibling `.schema.ts` files.
  - `createCompany.ts` / `updateCompany.ts` (+ `.schema.ts`) — create and edit a company (name/websiteUrl/`isPartner`/`ownerId`), both gated `crm.edit`. `updateCompany` is a `db.update(...).returning()` by `id` (empty → "That company no longer exists"), revalidating `/companies` and `/companies/${id}`.
- **UI** — three CRM pages: `/companies` ("Companies", `src/app/(app)/companies/page.tsx`) and `/contacts` ("Contacts", `src/app/(app)/contacts/page.tsx`), each its own server-side-paginated table, plus `/opportunities` (`src/app/(app)/opportunities/page.tsx`, a **dnd-kit kanban board** — `opportunity-board.tsx` + `opportunity-board-column.tsx` + `opportunity-card.tsx`; the old `opportunities-table.tsx` was removed). Clicking a card opens `opportunity-detail-sheet.tsx` (a **wide** `max-w-3xl` right-side `Sheet`) that lazy-loads via `loadOpportunityDetail` and re-fetches after every save. It reads as a **detail view with two tabs**:
    - **Info** — each field (Name, Line of business, Owners, Contacts, Source + its referral picker, Status, Next steps) renders **read-only with a per-field pencil**; clicking Edit swaps that one field into an inline control with **Confirm/Cancel**, and Confirm saves **just that field** via the existing full-form `updateOpportunity` (each Confirm rebuilds the whole `UpdateOpportunityInput` from the loaded detail with only that field changed — so the referral rule and the transition guard keep working). **Source + referral edit together as one unit** (the referral picker depends on the source). The client mirror of the delivery-stage guard lives in the Status field. Built on the shared **`InlineEditField`** primitive (see [../ui.md](../ui.md)).
    - **Project plan** — the single linked project: a "Create project" button (hidden once a project exists — at most one per opportunity), the linked-project display, and `AddProjectDialog` (prefilled + company-locked). See [projects.md](./projects.md).
    The drawer only **mounts for `crm.edit`** users (gated on the board), so editing is always allowed once open. **Click-vs-drag** is disambiguated by dnd-kit's 5px pointer activation constraint plus a `suppressClickRef` the board sets after a real drag (so ending a drag on a card doesn't also open the drawer).
  - **Detail pages**: `/companies/[id]` (`company-detail-view.tsx`) and `/contacts/[id]` (`contact-detail-view.tsx`), each a Server Component that `await`s its `get*Detail` read and `notFound()`s on a miss (with a `generateMetadata` titling the tab). Each now carries an **Edit affordance** (for `crm.edit` holders) opening `edit-company-dialog.tsx` / `edit-contact-dialog.tsx`, wired to `updateCompany` / `updateContact`. Both dialogs assign the optional **owner** via the searchable **`OwnerComboboxField`** (`owner-combobox-field.tsx`) — a clearable, debounced single-select of **active** staff backed by the `crm.edit`-gated `searchStaff` action, mirroring `ManagerComboboxField` (the contact dialog also carries the same manager picker). The owner name shown on each read view comes from a `leftJoin` to `staff` in `getCompanyDetail`/`getContactDetail`. Company detail = identity header + **tabs** (Opportunities / Projects / Contacts), each a table; the Contacts tab links each person to their contact page (it *is* the company's people directory). Contact detail = identity header + a **Details card** (email/phone/LinkedIn/company/manager, all optional; company & manager link out) + **tabs** (Opportunities — two labelled groups "Referred by this contact" / "Also involved in"; Projects — the derived referred-projects). Cross-links weave the two pages together (company ↔ contacts, opportunity/project rows → their company). These are the app's **first tabbed pages** (first use of the vendored `tabs` primitive) and share `opportunity-status-badge.tsx` (renders a status as its pipeline **group** label). See `src/components/crm/**` and [../ui.md](../ui.md).
  - **List tables now link into the detail pages:** `companies-table.tsx` links the name cell to `/companies/[id]`; `contacts-table.tsx` links the name cell to `/contacts/[id]` and its company cell to `/companies/[id]`.

## Authorization

**Reads are open** — any signed-in user can browse all companies, contacts, and opportunities (the `(app)` gate is the boundary). **All CRM writes** are gated by a single flat capability (no ownership dimension): **`crm.edit`** (renamed from `contacts.edit` when opportunities landed), granted to `sales`, `manager`, `admin`. It covers creating/editing companies, contacts, *and* opportunities — including the inline company/contact creation, the opportunity detail drawer's edit/move, `loadOpportunityDetail` (edit-only read), and every type-ahead picker (`searchCompanies`/`searchContacts`/`searchStaff`) that backs entry. Declared in action metadata; each page hides its "Add" dialog(s) (via one `canEdit` flag) for users without the capability, the opportunities board renders **static (no drag-and-drop) with no clickable drawer** for non-editors.

**Cross-domain RBAC nuance (delivery-stage handoff).** Creating a project needs **`projects.edit`**, which `sales` does *not* have (only `crm.edit`). The board is passed a separate `canCreateProject` flag (`projects.edit`): a user who *can* create projects gets the prefilled create-project prompt when they move a card into a delivery stage without one; a user who can't (e.g. `sales`) gets a **toast telling them a delivery manager must create the project first**. No permission was widened — the division of labour is deliberate ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)). See [domains/permissions.md](./permissions.md).

## Key flows

- **Pipeline management** (built) — create Opportunities and move them through the pipeline (`maturing` → `lead` → … → `closed_won`/`closed_lost`) by **dragging cards** across the kanban board (`updateOpportunityPosition`). Multi-substatus groups (Scoping/Allocating/Closing) collapse into one column or expand into subcolumns; dropping onto a collapsed group picks the first substatus (if dragged from earlier in the pipeline) or the last (from later). Non-editors (`!crm.edit`) get a static, read-only board.
- **Edit a company / contact** (built) — from the detail page (`/companies/[id]`, `/contacts/[id]`), a `crm.edit` holder opens an edit dialog covering the core fields, the optional **staff owner** (`OwnerComboboxField`), and — for contacts — company and manager. Saved via `updateCompany` / `updateContact` (a single `db.update(...).returning()` by id; an empty `returning` → user-safe "That … no longer exists"). `updateContact` re-runs the same-company **manager check** (`assertValidManager`, now guarding against self-management too) and the **email-uniqueness** map that `createContact` uses. Both revalidate the list path *and* the detail path.
- **Edit an opportunity** (built) — clicking a card opens the **detail drawer** (`opportunity-detail-sheet.tsx`), which lazy-loads via `loadOpportunityDetail`. Its **Info** tab edits name, status, line of business, next steps, source, owners, contacts, and referral sources **one field at a time in place** (pencil → Confirm/Cancel); each Confirm still calls the full-form `updateOpportunity` (the four people junctions replaced wholesale) with only that field changed. **The company is not editable there.** This is CRM's first field-edit flow beyond the drag-move, and the first use of the shared `InlineEditField` primitive.
- **Opportunity → Project handoff + delivery-stage requirement** (built) — **transitioning** an opportunity into the **Allocating** group or later (`requiresProject`, Closed – Lost excepted) requires a linked project; creating one directly in a delivery stage is forbidden. Dragging a card into a delivery stage without a project is **blocked** (only on a genuine stage change — an intra-column reorder never prompts); a `projects.edit` holder is shown the create-project dialog **prefilled and company-locked** for that opportunity, and on creation the pending move completes; a user without `projects.edit` (e.g. `sales`) gets a toast pointing to a delivery manager. Enforced server-side at the entry points (creation ban + transition guard in both status-mutating actions). See [projects.md](./projects.md), [ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md), and [flows.md](../flows.md).

## Connects to

- **Staff** — an opportunity's `owners` and referral `source_staff` are `staff` rows (FK, cascade). This is the first CRM ↔ staff link. A **company and a contact also each carry an optional `ownerId` → `staff`** (`onDelete: set null`) — the accountable owner. Deliberately **set-null, not cascade** (unlike the opportunity junctions): an owner is a soft pointer, so removing a staff member should only unassign them, never delete the company/contact.
- **Projects** — a project belongs to a `companies` row (required FK, `restrict`) and may optionally link back to an `opportunities` row (`projects.opportunityId`, nullable FK, `restrict`, **at most one project per opportunity** via a partial unique index). The Opportunity → Project handoff now **populates** that column, and `requiresProject` makes a project a precondition for advancing an opportunity to a delivery stage. See [projects.md](./projects.md), [ADR 0019](../decisions/0019-project-opportunity-link.md), [ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md).
- **Allocations / Timesheets** via the Project a won Opportunity will create (proposed).
- See [../data-model.md](../data-model.md) and [../flows.md](../flows.md).

## Open questions

- Probability/weighting per status and pipeline revenue forecasting (value × probability) — not modelled; opportunities carry no monetary value field yet.
- **Edit/delete** — **companies, contacts, and opportunities now all have edit** (companies/contacts via a whole-dialog `updateCompany`/`updateContact`; the opportunity detail drawer via per-field `updateOpportunity` + drag-move), so a contact's `companyId`/`linkedinUrl`/`managerId`/`ownerId` (and a company's owner) can now be changed after creation (the old create-only limitation is gone). **No *delete* for any CRM entity yet.** Note the `onDelete: restrict` on `opportunities.companyId` means a company-delete flow must handle live opportunities, and the `restrict` on `projects.opportunityId` means an **opportunity**-delete flow must handle the (at most one) live project linked to it.
