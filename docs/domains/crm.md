# Domain: CRM

**Status: growing.** Companies, contacts, and opportunities (data, reads, create, and `/companies` + `/contacts` + `/opportunities` UIs) all exist. Companies and contacts now also have **read-only detail pages** at `/companies/[id]` and `/contacts/[id]` — the app's **first drill-down detail views** and its **first tabbed pages** (see the UI section below). Companies and contacts now live on **two separate pages** (`/companies` and `/contacts`) — the old combined "Companies & Contacts" page was split ([design spec](../superpowers/specs/2026-07-09-split-companies-contacts-design.md)). `/opportunities` is now a **drag-and-drop kanban board** (not a table), with 14 leaf statuses organized into ordered pipeline groups — see [ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md). Clicking a card opens a **detail drawer** that **edits** the opportunity (status, next steps, owners, source, contacts, referral sources — not the company). Projects now exist too ([projects.md](./projects.md)), and the **Opportunity → Project handoff is now built**: `projects.opportunityId` is populated when a project is created from an opportunity, and a **delivery-stage rule** requires a linked project before an opportunity can advance to Allocating or later ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)). Manages the companies and people we deal with (clients, partners, prospects) and the pipeline deals we're trying to win, all of which feed delivery.

## Purpose

Track who we sell to and work with, and (eventually) what we're trying to win, so won work flows cleanly into Projects and Allocations.

## Key entities

- **Company** (built) — an organisation we deal with. **`isPartner` is a standalone flag** marking whether the company is a partner (default `false`); it is **not** a client-vs-partner dichotomy — a company may be neither a client nor a partner, and nothing requires it to be either. We deliberately modelled this as *Company* rather than the narrower *Client* the early design assumed — see [ADR 0015](../decisions/0015-crm-company-over-client.md). Fields: `name` (required), `websiteUrl` (optional), `isPartner`. Table `companies`, id prefix `company`.
- **Contact** (built) — a person, optionally attached to a Company. Fields: `firstName`/`lastName` (required), `email` (required, **unique**), `phone` (optional; shown as a `tel:` link in the contacts table), `companyId` (optional FK → `companies`, `onDelete: set null`), `role` (optional free-text job title, e.g. "CTO"), `linkedinUrl` (optional; validated by the shared `optionalUrl` schema, same as company `websiteUrl`; shown as an external "Profile" link in the contacts table), and **`managerId`** — an optional **self-referential** FK → `contacts.id` (`onDelete: set null`), "managed by" another contact. This is the repo's **first self-referential FK**; by business rule a manager must be an existing contact **at the same company**, enforced app-side (not in the DB) — see [ADR 0022](../decisions/0022-contact-manager-self-reference.md). Table `contacts`, id prefix `contact`. Schema in `src/lib/db/crm-schema.ts`; see [../data-model.md](../data-model.md).
- **Opportunity** (built) — a pipeline deal that always belongs to a Company. Fields: `name` (required), required `companyId` (FK → `companies`, **`onDelete: restrict`** — a company with live opportunities can't be deleted, unlike its optional/set-null contacts), `source` + `status` (pgEnums), optional free-text `nextSteps`, and **`position`** (see below). Table `opportunities`, id prefix `opp`. Its people are modelled with **four junction tables** (see below): related `contacts`, `owners` (→ `staff`), and referral sources split into `source_contacts` and `source_staff`. An Opportunity produces a Project once it reaches delivery stages; the link column lives on the **project** side (`projects.opportunityId`, nullable FK → `opportunities`, `onDelete: restrict`, **1:N** — one opportunity → many projects), now **populated by the handoff flow** ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)), so a delete of an opportunity with live projects is blocked. See [ADR 0019](../decisions/0019-project-opportunity-link.md).
  - **`source`** (`opportunity_source`): `inbound`, `farming`, `extension`, `change_request`, `staff_referral`, `contact_referral`.
  - **`status`** (`opportunity_status`): a **flat 14-value leaf enum in strict pipeline order** — `maturing`, `lead`, `qualifying`, `scoping_awaiting_info`, `scoping`, `scoping_reviewing`, `allocating_awaiting_profiles`, `allocating_introing_profiles`, `negotiating`, `closing_awaiting_contracts`, `closing_redlining`, `closing_awaiting_signatures`, `closed_won`, `closed_lost`. The DB knows only leaves; the two-level **grouping** (below) is derived in code. (Restructured from the old flat 7-value enum; migration `0020` remaps legacy `closing` → `closing_awaiting_contracts`.)
  - **Pipeline groups** — `src/lib/opportunity-pipeline.ts` (a pure, client-importable module, mirroring `line-of-business.ts`) defines `OPPORTUNITY_GROUPS`: ordered groups → their leaf substatuses. In order: **Maturing, Lead, Qualifying** (single-status), **Scoping** (awaiting info / scoping / reviewing), **Allocating** (awaiting profiles / introing profiles), **Negotiating**, **Closing** (awaiting contracts / redlining / awaiting signatures), **Won** (`closed_won`), **Lost** (`closed_lost`). A **module-load assertion** (+ `opportunity-pipeline.test.ts`) requires the flattened groups to equal `OPPORTUNITY_STATUSES` in the same order, so they can't drift. Also exports `resolveTargetStatus` (which leaf a card takes when dropped on a collapsed group), `computePosition` (fractional-index midpoint), and **`requiresProject(status)`** — true from the **Allocating** group onward (Allocating → Negotiating → Closing → Won), Closed – Lost excepted: the delivery-stage rule that an opportunity can't advance without a linked project ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)). See [ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md).
  - **`position`** (`doublePrecision NOT NULL DEFAULT 0`) — a **single global fractional index** for manual kanban ordering. Cards in a column sort by `(position, createdAt)`; a drag writes the midpoint between its new neighbours (one-row update), a new opportunity gets `max(position)+1` (lands at the end of its column). Index `opportunities_status_position_idx` on `(status, position)`. No fractional-index rebalancing — an accepted limitation at this scale ([ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md)).
  - Both enum value tuples are declared **once** in `src/actions/crm/createOpportunity.schema.ts` (a pure, client-importable module — no `db`/drizzle) and imported by *both* the `pgEnum` in `crm-schema.ts` and the zod schema, so there's one source of truth. Per-leaf display labels live in `src/components/crm/opportunity-display.ts` (`STATUS_LABELS`); group labels live in `opportunity-pipeline.ts`.

## What's built

- **Schema** — `src/lib/db/crm-schema.ts` (`companies`, `contacts`, `opportunities` + four junction tables), barrelled by `src/lib/db/schema.ts`. Migrations `drizzle/0012_spotty_mysterio.sql` (companies/contacts), `drizzle/0013_clean_firelord.sql` (adds `contacts.phone`), `drizzle/0014_nostalgic_crusher_hogan.sql` (opportunities + junctions + enums), `drizzle/0020_unique_blue_shield.sql` (**hand-edited**: recreates `opportunity_status` as the 14-value leaf enum, remapping legacy `closing` → `closing_awaiting_contracts`; adds `position` + backfills it via `row_number()` per status; adds `opportunities_status_position_idx`), `drizzle/0021_glamorous_goliath.sql` (adds `contacts.linkedinUrl` + self-referential `contacts.managerId`).
- **Junction tables** — the repo's **first** many-to-many join tables: `opportunity_contacts`, `opportunity_owners` (→ `staff`), `opportunity_source_contacts`, `opportunity_source_staff` (→ `staff`). All follow one convention (documented in [../data-model.md](../data-model.md#junction-tables--the-first-many-to-many-pattern)): a **surrogate `text` PK** via `generateId` (no composite PKs — repo convention), a **`unique(...)` on the FK pair** for set-semantics (no duplicate links), an **`index(...)` on the non-opportunity FK** for reverse lookups, and **both FKs `onDelete: "cascade"`** (a link is meaningless without both endpoints).
- **Server layer** — `src/actions/crm/`:
  - `getCompaniesPage.ts` / `getContactsPage.ts` — server-only reads (per ADR 0010), **server-side offset/limit pagination** with a `count` and a page clamped into range (out-of-bounds query params can't return an empty page). Both share `src/lib/pagination.ts`, which exports `CRM_PAGE_SIZE` (20), `clampPage` (the range-clamp arithmetic — there's always ≥1 page, even when empty), and the `Page<T>` envelope (`{ rows, total, page, pageSize, pageCount }`) they return. Contacts left-join companies to resolve `companyName` (company is optional) **and self-join `contacts` (drizzle `alias`) to resolve `managerName`** (manager is optional too); `ContactRow` carries `linkedinUrl`, `managerId`, `managerName`.
  - `getCompanyDetail.ts` / `getContactDetail.ts` — server-only reads backing the two detail pages (per ADR 0010; **not** `'use server'`, so pages `await` them directly during SSR). Each fetches the entity's core fields, then its related collections in a `Promise.all`, projected to just the columns the view renders; both return `null` on an unknown id (page → `notFound()`). No pagination (an entity's own footprint is small). Neither adds a permission beyond the `(app)` gate — consistent with all CRM reads being open.
    - `getCompanyDetail` returns core fields + three collections tied to `companyId`: `opportunities`, `projects`, and `contacts` (the company's people directory).
    - `getContactDetail` returns identity + employer + manager (both optional, resolved via **left joins**, with a `contacts` self-join `alias` for `managerName` like `getContactsPage`), plus the contact's CRM footprint split into `referredOpportunities` (via `opportunitySourceContacts`), `involvedOpportunities` (via `opportunityContacts`, **minus** any already in referred so nothing double-lists), and `referredProjects`. **Contacts don't link to projects directly**, so "projects" is *derived*: projects whose `opportunityId` is one of the contact's referred opportunity ids (`inArray`). All opportunity/project rows inner-join their company so the cross-company view can name and link the employer.
  - `getOpportunitiesBoard.ts` — **non-paginated** server-only read backing the kanban board (a consultancy's pipeline is small, so no paging). Returns *all* cards `{ id, name, companyId, companyName, source, status, ownerNames, hasProject, position, createdAt }` ordered by `(position, createdAt)`; inner-joins companies for `companyName` (company is required) and resolves owner names **and** the `hasProject` flag via **two grouped follow-up queries** over all card ids (no N+1). `hasProject` (any linked `projects.opportunityId`) drives the delivery-stage requirement; `companyId` lets the board prefill the create-project prompt. `createdAt` is returned as epoch millis so the client's `(position, createdAt)` tie-break matches the server's. (Replaced the removed `getOpportunitiesPage.ts` — opportunities are no longer paginated.)
  - `getOpportunity.ts` — **server-only** read for the detail drawer: an opportunity's core fields, its company, the four people junctions each as `{ id, name }[]` (shaped for the drawer's comboboxes), and its linked projects. Null if the id is unknown.
  - `getOpportunityDetail.ts` — a thin **`'use server'`** action wrapping `getOpportunity` for the client-triggered drawer load (the **interactive-read exception** to the server-only rule, like `searchStaff`), gated `crm.edit` — the drawer is edit-only, so this can't leak detail past the write gate.
  - `updateOpportunity.ts` (+ `.schema.ts`) — the drawer's **edit** action, gated `crm.edit`. One transaction: updates the row (name/source/status/nextSteps — **not** the company) and **replaces the four people junctions wholesale** (delete + re-insert deduped id sets, same shape `createOpportunity` writes). Enforces the delivery-stage rule **on status transitions only** (see the delivery-stage enforcement bullet). `revalidatePath("/opportunities")`.
  - `updateOpportunityPosition.ts` (+ `.schema.ts`) — the kanban **move** action, gated `crm.edit`. Input `{ id, status, position }` (an absolute destination the board computes client-side from the drop's neighbours); a **single-row update** with no server-side renumbering, then `revalidatePath("/opportunities")`. See [ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md).
  - **Delivery-stage enforcement** — the invariant is *an opportunity in a delivery stage has a linked project*, enforced server-side at the **entry points** (so it can't be bypassed by hitting an action directly): `createOpportunity` **rejects** creating in a delivery stage (a new opportunity can't have a project yet), and `updateOpportunityPosition`/`updateOpportunity` guard **transitions** — each fetches the current status and fires only when the status is *changing into* a requiring stage without a project (`newStatus !== currentStatus && requiresProject(newStatus) && !opportunityHasProject(id)`), throwing a `UserSafeActionError`. This is **transition-based, not state-based**: editing (rename/owners/next-steps) or reordering within a delivery column no longer trips the guard (the earlier every-edit check over-fired). `opportunityHasProject.ts` is the indexed existence check (server-only). The client block is just UX. See [ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md).
  - `searchCompanies.ts` / `searchContacts.ts` / `searchStaff.ts` — type-ahead pickers, `secureActionClient` gated `crm.edit` (they back CRM entry, itself a write, so the gate stops them being used to enumerate the roster past the page gate). Each returns up to `SEARCH_LIMIT` `{ id, name }` by `ilike` match; **a blank query returns nothing** (search only runs once the user types). All three share the query surface via `src/lib/search.ts`, which exports `SEARCH_LIMIT` (10) and `searchQuerySchema` (a single `query` string **trimmed on parse**, so whitespace collapses to the blank-query short-circuit). The `searchStaff` / `searchCompanies` **query bodies** are further shared with the Projects domain via `searchStaffByName` / `searchCompaniesByName` in `src/actions/shared/entitySearch.ts` — the CRM and projects search actions each wrap the same query with their own permission gate (`crm.edit` vs `projects.edit`); only `searchContacts` keeps its own body (different columns). Nuances: `searchStaff` filters `isActive = true` (**only active staff**), and `searchContacts` matches **first name / last name / email** (not just name) and takes an **optional `companyId` filter** that scopes results to one company — the contact manager picker passes it so a manager can only be a colleague at the same company. All also share the LIKE-metacharacter escaper `escapeLike` in `src/lib/like.ts` (extracted from `searchCompanies`).
  - `createOpportunity.ts` — gated `crm.edit`. **Rejects creating in a delivery stage** (any `requiresProject(status)` — Allocating onward): a brand-new opportunity can't have a linked project yet, so that state is unreachable; the error points the user to create it earlier and add a project as it advances. Otherwise one **transaction**: reads `max(position)` and inserts the opportunity with `position = max+1` (so it lands at the end of its board column), then bulk-inserts junction rows, **deduping each id array** first so a repeat can't trip the junction `unique` index. Related entities (company, contacts) are created via their own actions first; this only consumes their ids. `revalidatePath("/opportunities")`.
  - `createOpportunity.schema.ts` — the shared zod schema (+ the enum tuples). A `superRefine` enforces the conditional referral rules: `source = staff_referral` ⇒ ≥1 `sourceStaffIds`; `source = contact_referral` ⇒ ≥1 `sourceContactIds`.
  - `createCompany.ts` — gated `crm.edit`.
  - `createContact.ts` — gated `crm.edit`; a **single `db.insert(contacts)`** (no transaction) that maps the Postgres unique-violation (SQLSTATE `23505`) on `email` to a user-safe "A contact with that email already exists." **Returns `{ id }`** (not `{ ok: true }`) so an inline-created contact can be immediately selected in the opportunity form. **Manager validation is server-side:** before inserting, if `managerId` is set it re-checks (a DB lookup) that a company is set and that the manager is a contact at the *same* company, rejecting with a user-safe error otherwise — the UI picker enforces the same rule, but a hand-crafted request can't create a cross-company or dangling management link ([ADR 0022](../decisions/0022-contact-manager-self-reference.md)). Creating a company inline from the contact form is **persist-first** — the new company is saved via `createCompany` first (through `CompanyComboboxField`'s inline dialog), then its id is passed here as `companyId`. This mirrors opportunities and replaced an earlier design where `createContact` created the company itself in a nested `db.transaction` (that `newCompany` path and the transaction are gone). `createContact` now `revalidatePath("/contacts")` (its own page); `createCompany` `revalidatePath("/companies")`. Input schemas live in sibling `.schema.ts` files.
- **UI** — three CRM pages: `/companies` ("Companies", `src/app/(app)/companies/page.tsx`) and `/contacts` ("Contacts", `src/app/(app)/contacts/page.tsx`), each its own server-side-paginated table, plus `/opportunities` (`src/app/(app)/opportunities/page.tsx`, a **dnd-kit kanban board** — `opportunity-board.tsx` + `opportunity-board-column.tsx` + `opportunity-card.tsx`; the old `opportunities-table.tsx` was removed). Clicking a card opens `opportunity-detail-sheet.tsx` (a right-side `Sheet`) that lazy-loads via `getOpportunityDetail` and edits through `updateOpportunity`. **Click-vs-drag** is disambiguated by dnd-kit's 5px pointer activation constraint plus a `suppressClickRef` the board sets after a real drag (so ending a drag on a card doesn't also open the drawer).
  - **Detail pages** (**read-only**, no edit affordances yet): `/companies/[id]` (`company-detail-view.tsx`) and `/contacts/[id]` (`contact-detail-view.tsx`), each a Server Component that `await`s its `get*Detail` read and `notFound()`s on a miss (with a `generateMetadata` titling the tab). Company detail = identity header + **tabs** (Opportunities / Projects / Contacts), each a table; the Contacts tab links each person to their contact page (it *is* the company's people directory). Contact detail = identity header + a **Details card** (email/phone/LinkedIn/company/manager, all optional; company & manager link out) + **tabs** (Opportunities — two labelled groups "Referred by this contact" / "Also involved in"; Projects — the derived referred-projects). Cross-links weave the two pages together (company ↔ contacts, opportunity/project rows → their company). These are the app's **first tabbed pages** (first use of the vendored `tabs` primitive) and share `opportunity-status-badge.tsx` (renders a status as its pipeline **group** label). See `src/components/crm/**` and [../ui.md](../ui.md).
  - **List tables now link into the detail pages:** `companies-table.tsx` links the name cell to `/companies/[id]`; `contacts-table.tsx` links the name cell to `/contacts/[id]` and its company cell to `/companies/[id]`.

## Authorization

**Reads are open** — any signed-in user can browse all companies, contacts, and opportunities (the `(app)` gate is the boundary). **All CRM writes** are gated by a single flat capability (no ownership dimension): **`crm.edit`** (renamed from `contacts.edit` when opportunities landed), granted to `sales`, `manager`, `admin`. It covers creating/editing companies, contacts, *and* opportunities — including the inline company/contact creation, the opportunity detail drawer's edit/move, `getOpportunityDetail` (edit-only read), and every type-ahead picker (`searchCompanies`/`searchContacts`/`searchStaff`) that backs entry. Declared in action metadata; each page hides its "Add" dialog(s) (via one `canEdit` flag) for users without the capability, the opportunities board renders **static (no drag-and-drop) with no clickable drawer** for non-editors.

**Cross-domain RBAC nuance (delivery-stage handoff).** Creating a project needs **`projects.edit`**, which `sales` does *not* have (only `crm.edit`). The board is passed a separate `canCreateProject` flag (`projects.edit`): a user who *can* create projects gets the prefilled create-project prompt when they move a card into a delivery stage without one; a user who can't (e.g. `sales`) gets a **toast telling them a delivery manager must create the project first**. No permission was widened — the division of labour is deliberate ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)). See [domains/permissions.md](./permissions.md).

## Key flows

- **Pipeline management** (built) — create Opportunities and move them through the pipeline (`maturing` → `lead` → … → `closed_won`/`closed_lost`) by **dragging cards** across the kanban board (`updateOpportunityPosition`). Multi-substatus groups (Scoping/Allocating/Closing) collapse into one column or expand into subcolumns; dropping onto a collapsed group picks the first substatus (if dragged from earlier in the pipeline) or the last (from later). Non-editors (`!crm.edit`) get a static, read-only board.
- **Edit an opportunity** (built) — clicking a card opens the **detail drawer** (`opportunity-detail-sheet.tsx`), which lazy-loads via `getOpportunityDetail` and edits name, status, next steps, source, owners, contacts, and referral sources through `updateOpportunity` (the four people junctions replaced wholesale). **The company is not editable there.** This is CRM's first field-edit flow beyond the drag-move.
- **Opportunity → Project handoff + delivery-stage requirement** (built) — **transitioning** an opportunity into the **Allocating** group or later (`requiresProject`, Closed – Lost excepted) requires a linked project; creating one directly in a delivery stage is forbidden. Dragging a card into a delivery stage without a project is **blocked** (only on a genuine stage change — an intra-column reorder never prompts); a `projects.edit` holder is shown the create-project dialog **prefilled and company-locked** for that opportunity, and on creation the pending move completes; a user without `projects.edit` (e.g. `sales`) gets a toast pointing to a delivery manager. Enforced server-side at the entry points (creation ban + transition guard in both status-mutating actions). See [projects.md](./projects.md), [ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md), and [flows.md](../flows.md).

## Connects to

- **Staff** — an opportunity's `owners` and referral `source_staff` are `staff` rows (FK, cascade). This is the first CRM ↔ staff link.
- **Projects** — a project belongs to a `companies` row (required FK, `restrict`) and may optionally link back to an `opportunities` row (`projects.opportunityId`, nullable FK, `restrict`, 1:N). The Opportunity → Project handoff now **populates** that column, and `requiresProject` makes a project a precondition for advancing an opportunity to a delivery stage. See [projects.md](./projects.md), [ADR 0019](../decisions/0019-project-opportunity-link.md), [ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md).
- **Allocations / Timesheets** via the Project a won Opportunity will create (proposed).
- See [../data-model.md](../data-model.md) and [../flows.md](../flows.md).

## Open questions

- Probability/weighting per status and pipeline revenue forecasting (value × probability) — not modelled; opportunities carry no monetary value field yet.
- **Edit/delete** — companies and contacts are still **create + read only** (they now have read-only **detail pages**, but no edit affordance and no delete); **opportunities now have full field edit** (the detail drawer, `updateOpportunity`) plus the drag-move, but no *delete* for any CRM entity yet. Because contacts are create-only, a contact's `linkedinUrl` and `managerId` can only be set at creation: a contact created **without** a company can never get a manager, and a manager can't be reassigned or added later. Building the contact edit flow is the way to fix this — see [ADR 0022](../decisions/0022-contact-manager-self-reference.md). Note the `onDelete: restrict` on `opportunities.companyId` means a company-delete flow must handle live opportunities, and the `restrict` on `projects.opportunityId` means an **opportunity**-delete flow must handle live projects linked to it.
