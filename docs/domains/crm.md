# Domain: CRM

**Status: growing.** Companies, contacts, and opportunities (data, reads, create, edit, and `/companies` + `/contacts` + `/opportunities` UIs) all exist. **Companies and contacts are now full CRUD (create/read/edit; no delete yet)** — each has a **detail page** at `/companies/[id]` and `/contacts/[id]` — the app's **first drill-down detail views** (see the UI section below) — carrying an **Edit** affordance (for `crm.edit` holders) that opens a dialog wired to `updateCompany`/`updateContact` for the core fields. The optional **staff owner** (the accountable staff member) is **edited in place on the detail page** (`InlineOwnerField`, mirroring the opportunity drawer's per-field edits), not in that dialog. Companies and contacts now live on **two separate pages** (`/companies` and `/contacts`) — the old combined "Companies & Contacts" page was split ([design spec](../superpowers/specs/2026-07-09-split-companies-contacts-design.md)). `/opportunities` now toggles between a **drag-and-drop kanban board** (default) and a **paginated, filterable list view** (`?view=list`), with 14 leaf statuses organized into ordered pipeline groups — see [ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md) and [Board + list views](#board--list-views). Opportunities now carry a **required line of business** (the shared enum, [ADR 0025](../decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)), and their table + junctions now live in **their own schema file** `src/lib/db/opportunities-schema.ts` (split out of `crm-schema.ts`, which keeps only companies + contacts). Clicking a card (board) or a name (list) opens a **detail drawer** — a wide (`sm:max-w-[80rem]`, widened for the planner + the two-column Details layout) two-tab read-first view (**Details** + **Project plan**). The **Details** tab is a responsive two-column layout: the Info fields (line of business, source, company, contacts, owners) sit in a left meta column — each **edits one at a time in place** (per-field pencil → Confirm/Cancel) — while the Next steps + Notes logs fill the right column (columns stack on small screens). The **Project plan** tab is a **weekly Gantt-like staffing planner** over the linked project's roles ([ADR 0031](../decisions/0031-opportunity-project-planner-and-role-status.md)). Both **contacts and opportunities** also carry two running logs of timestamped, authored free-text entries — **notes** ("what happened") and **next steps** ("what's planned") — see [Timestamped entries](#timestamped-entries-notes--next-steps) and [ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md); the scalar `opportunities.nextSteps` column that predated them is gone. Projects now exist too ([projects.md](./projects.md)), and the **Opportunity → Project link inverted**: it lives on **`opportunities.projectId`** (many opps → one project), populated when a project is created from or associated to an opportunity, and a **delivery-stage rule** requires a linked project before an opportunity can advance to Allocating or later ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md), [ADR 0031](../decisions/0031-opportunity-project-planner-and-role-status.md)). Manages the companies and people we deal with (clients, partners, prospects) and the pipeline deals we're trying to win, all of which feed delivery.

## Purpose

Track who we sell to and work with, and (eventually) what we're trying to win, so won work flows cleanly into Projects and Allocations.

## Key entities

- **Company** (built) — an organisation we deal with. **`isPartner` is a standalone flag** marking whether the company is a partner (default `false`); it is **not** a client-vs-partner dichotomy — a company may be neither a client nor a partner, and nothing requires it to be either. We deliberately modelled this as *Company* rather than the narrower *Client* the early design assumed — see [ADR 0015](../decisions/0015-crm-company-over-client.md). Fields: `name` (required), `websiteUrl` (optional), an optional free-text **`location`** (`"City, CC"`; see [Location](#location)), `isPartner`, and an optional **`ownerId`** — a nullable FK → **`staff`** (`onDelete: set null`), the staff member accountable for the relationship; cleared (not blocked) if that staff row is removed. This is a CRM company/contact ↔ staff link (mirrors `opportunityOwners.staffId`; see [Connects to](#connects-to)). Table `companies`, id prefix `company`. A company's **status is *derived*, not stored** — a set of tags (**Partner** = `isPartner`; **Client** = has a confirmed project; **Prospect** = has an open opportunity) computed from its relationships, so `isPartner` remains the only stored/manual relationship signal ([ADR 0034](../decisions/0034-company-status-derived-tags.md); see [Company status tags](#company-status-tags)). A company also carries a running **notes log** (`company_entries` — the same authored/timestamped entry pattern contacts & opportunities use, but companies use only the `note` kind in the UI; see [Timestamped entries](#timestamped-entries-notes--next-steps)).
- **Contact** (built) — a person, optionally attached to a Company. Fields: `firstName`/`lastName` (required), `email` (required, **unique**), `phone` (optional; shown as a `tel:` link on the detail page), `companyId` (optional FK → `companies`, `onDelete: set null`), `role` (optional free-text job title, e.g. "CTO"), `linkedinUrl` (optional; validated by the shared `optionalUrl` schema, same as company `websiteUrl`; shown as an external "Profile" link on the detail page), an optional free-text **`location`** (`"City, CC"`; see [Location](#location)), and **`managerId`** — an optional **self-referential** FK → `contacts.id` (`onDelete: set null`), "managed by" another contact. This is the repo's **first self-referential FK**; by business rule a manager must be an existing contact **at the same company**, enforced app-side (not in the DB, via `assertValidManager` in `contactChecks.ts`) — see [ADR 0022](../decisions/0022-contact-manager-self-reference.md). Also carries an optional **`ownerId`** — a nullable FK → **`staff`** (`onDelete: set null`), the accountable staff member; cleared if that staff row is removed (same set-null convention as `companies.ownerId`) — and an optional **`relationshipStrength`** — a plain **`integer`** on a **1–5** scale (1 New/Unestablished, 2 Weak, 3 Developing, 4 Strong Supporter, 5 Champion/Trusted Partner), **null = not yet rated**; level meanings live in the pure module `src/lib/crm/relationship-strength.ts` (mirrors `@/lib/performance/feedback-rating`, [ADR 0016](../decisions/0016-junction-table-and-shared-enum-conventions.md)), and it's **edited inline as a star rating** on the contact page (see [Relationship strength](#relationship-strength) below). Table `contacts`, id prefix `contact`. A contact also carries two running logs of **notes** and **next steps** (`contact_entries` — see [Timestamped entries](#timestamped-entries-notes--next-steps)). Schema in `src/lib/db/crm-schema.ts`; see [../data-model.md](../data-model.md).
- **Opportunity** (built) — a pipeline deal that always belongs to a Company. Fields: `name` (required), required `companyId` (FK → `companies`, **`onDelete: restrict`** — a company with live opportunities can't be deleted, unlike its optional/set-null contacts), `source` + `status` (pgEnums), **required `lineOfBusiness`** (the shared `lineOfBusinessEnum` — which practice the deal belongs to; a project created from it defaults to the same, [ADR 0025](../decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)), and **`position`** (see below). **No scalar `nextSteps` column** — the old single next-step field was migrated into the `opportunity_entries` log and dropped ([ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md); see [Timestamped entries](#timestamped-entries-notes--next-steps)). Table `opportunities`, id prefix `opp` — **now in its own schema file** `src/lib/db/opportunities-schema.ts` (split out of `crm-schema.ts`, which keeps only `companies` + `contacts`; [ADR 0025](../decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)). Its people are modelled with **four junction tables** (see below): related `contacts`, `owners` (→ `staff`), and referral sources split into `source_contacts` and `source_staff`. An Opportunity produces a Project once it reaches delivery stages; the link column now lives on the **opportunity** side — **`opportunities.projectId`** (nullable FK → `projects`, `onDelete: restrict`, indexed `opportunities_project_idx`), **many opportunities → one project** (a project can be built up from several deals; an opportunity has at most one project). The old `projects.opportunityId` column + its partial unique index were **removed** in the inversion. `restrict` blocks deleting a project any opportunity references. Populated by the handoff/planner flow ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md), [ADR 0031](../decisions/0031-opportunity-project-planner-and-role-status.md)). See [ADR 0019](../decisions/0019-project-opportunity-link.md).
  - **`source`** (`opportunity_source`): `inbound`, `farming`, `extension`, `change_request`, `staff_referral`, `contact_referral`.
  - **`status`** (`opportunity_status`): a **flat 14-value leaf enum in strict pipeline order** — `maturing`, `lead`, `qualifying`, `scoping_awaiting_info`, `scoping`, `scoping_reviewing`, `allocating_awaiting_profiles`, `allocating_introing_profiles`, `negotiating`, `closing_awaiting_contracts`, `closing_redlining`, `closing_awaiting_signatures`, `closed_won`, `closed_lost`. The DB knows only leaves; the two-level **grouping** (below) is derived in code. (Restructured from the old flat 7-value enum; migration `0020` remaps legacy `closing` → `closing_awaiting_contracts`.)
  - **Pipeline groups** — `src/lib/crm/opportunity-pipeline.ts` (a pure, client-importable module, mirroring `line-of-business.ts`) defines `OPPORTUNITY_GROUPS`: ordered groups → their leaf substatuses. In order: **Maturing, Lead, Qualifying** (single-status), **Scoping** (awaiting info / scoping / reviewing), **Allocating** (awaiting profiles / introing profiles), **Negotiating**, **Closing** (awaiting contracts / redlining / awaiting signatures), **Won** (`closed_won`), **Lost** (`closed_lost`). A **module-load assertion** requires the flattened groups to equal `OPPORTUNITY_STATUSES` in the same order, so they can't drift. Also exports `resolveTargetStatus` (which leaf a card takes when dropped on a collapsed group), `computePosition` (fractional-index midpoint), **`opportunityGroupById(id)`** (a group by its id — how the list view's `stage` filter param resolves to its leaf statuses), **`requiresProject(status)`** — true from the **Allocating** group onward (Allocating → Negotiating → Closing → Won), Closed – Lost excepted: the delivery-stage rule that an opportunity can't advance without a linked project ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)) — and the board-capping constants **`CAPPED_BOARD_STATUSES`** (`maturing`, `closed_won`, `closed_lost`) + **`BOARD_COLUMN_CAP`** (20), see [Board + list views](#board--list-views). See [ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md).
  - **`position`** (`doublePrecision NOT NULL DEFAULT 0`) — a **single global fractional index** for manual kanban ordering. Cards in a column sort by `(position, createdAt)`; a drag writes the midpoint between its new neighbours (one-row update), a new opportunity gets `max(position)+1` (lands at the end of its column). Index `opportunities_status_position_idx` on `(status, position)`. No fractional-index rebalancing — an accepted limitation at this scale ([ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md)).
  - Both enum value tuples (`OPPORTUNITY_SOURCES`/`OPPORTUNITY_STATUSES`), their union types (`OpportunitySource`/`OpportunityStatus`), and their `SOURCE_LABELS`/`STATUS_LABELS` are declared **once** in `src/lib/crm/opportunity.ts` (a pure, client-importable module — no `db`/drizzle, no UI, mirroring `line-of-business.ts`/`currency.ts`), and everything imports them **directly** from there: the `pgEnum` in `opportunities-schema.ts`, the zod schema, and the display components. This fixed the old layering where the tuples lived in `createOpportunity.schema.ts` (imported *upward* by the pgEnum) and the labels in `opportunity-display.ts`. The **back-compat re-export shims are now gone** — `createOpportunity.schema.ts` no longer re-exports the enums, and `opportunity-display.ts` keeps only `STATUS_SELECT_LABELS` (the group-qualified dropdown labels it derives). Per-leaf display labels are `STATUS_LABELS`; group labels live in `opportunity-pipeline.ts`.

## What's built

- **Schema** — `src/lib/db/crm-schema.ts` (`companies`, `contacts`, **`contact_entries`**, **`company_entries`**, and the shared **`crm_entry_kind`** pgEnum) + `src/lib/db/opportunities-schema.ts` (`opportunities` + four junction tables + **`opportunity_entries`** + the `source`/`status` pgEnums + `lineOfBusiness`), both barrelled by `src/lib/db/schema.ts`. The three `*_entries` tables (notes & next steps) share the `crm_entry_kind` enum — see [Timestamped entries](#timestamped-entries-notes--next-steps) and [ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md). `company_entries` mirrors `contact_entries` exactly (`id`, `companyId` FK → `companies` cascade, `kind`, `body`, `authorStaffId` FK → `staff` set-null, timestamps, index `company_entries_company_kind_created_idx` on `(companyId, kind, createdAt)`; row type `CompanyEntry`) — the DB shape carries both kinds for consistency even though the company UI only writes `note`. Migration `drizzle/0004_blue_terrax.sql`. The opportunities file imports `companies`/`contacts` from `./crm-schema` and `staff`/`lineOfBusinessEnum` from `./staff-schema` ([ADR 0025](../decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)). Migrations `drizzle/0012_spotty_mysterio.sql` (companies/contacts), `drizzle/0013_clean_firelord.sql` (adds `contacts.phone`), `drizzle/0014_nostalgic_crusher_hogan.sql` (opportunities + junctions + enums), `drizzle/0020_unique_blue_shield.sql` (**hand-edited**: recreates `opportunity_status` as the 14-value leaf enum, remapping legacy `closing` → `closing_awaiting_contracts`; adds `position` + backfills it via `row_number()` per status; adds `opportunities_status_position_idx`), `drizzle/0021_glamorous_goliath.sql` (adds `contacts.linkedinUrl` + self-referential `contacts.managerId`), `drizzle/0024_harsh_diamondback.sql` (adds `opportunities.line_of_business` — backfilled to `CORE` via a temporary default, then dropped, so it's NOT NULL with no default). The schema split itself is a pure code reorg (no migration).
- **Junction tables** — the repo's **first** many-to-many join tables: `opportunity_contacts`, `opportunity_owners` (→ `staff`), `opportunity_source_contacts`, `opportunity_source_staff` (→ `staff`). All follow one convention (documented in [../data-model.md](../data-model.md#junction-tables--the-first-many-to-many-pattern)): a **surrogate `text` PK** via `generateId` (no composite PKs — repo convention), a **`unique(...)` on the FK pair** for set-semantics (no duplicate links), an **`index(...)` on the non-opportunity FK** for reverse lookups, and **both FKs `onDelete: "cascade"`** (a link is meaningless without both endpoints).
- **Server layer** — `src/actions/crm/`:
  - `getCompaniesPage.ts` / `getContactsPage.ts` — server-only reads (per ADR 0010), **server-side offset/limit pagination** with a `count` and a page clamped into range (out-of-bounds query params can't return an empty page). Both share `src/lib/core/pagination.ts`, which exports `CRM_PAGE_SIZE` (20), `clampPage` (the range-clamp arithmetic — there's always ≥1 page, even when empty), and the `Page<T>` envelope (`{ rows, total, page, pageSize, pageCount }`) they return. `CompanyRow` is `{ id, name, location, isPartner, isClient, isProspect }` (`location` a read-only column projection for the table's Location column) — the three **derived status flags** ([Company status tags](#company-status-tags), [ADR 0034](../decisions/0034-company-status-derived-tags.md)): `isPartner` is the stored column, while `isClient` (a confirmed project, correlated `opportunities.companyId → projectId → projects.status = 'confirmed'`) and `isProspect` (an open opportunity, `status NOT IN CLOSED_OPPORTUNITY_STATUSES`) are computed inline as **correlated `EXISTS` subqueries** so the page stays one query. `getCompaniesPage` also takes a **`filters` arg** (`{ query?, status? }`) — a case-insensitive **name search** (`ilike`) and a **single derived-status filter** (partner/client/prospect); the `where` is applied to **both** the count and the row query so the total/page-count reflect the filtered set. The two `EXISTS` expressions (`hasConfirmedProject`/`hasOpenOpportunity`) are pulled out as reusable condition consts, reused **both** as the selected boolean (via `.mapWith(Boolean)`) and as the status filter's `where` (partner filters on the stored `isPartner` flag). Contacts left-join companies to resolve `companyName` (company is optional). `ContactRow` is **what the list table renders** — `id`, `firstName`/`lastName`, `role`, `companyId`/`companyName`, the read-only `location` label, and the **latest next step** (`nextStep` body + `nextStepAt` epoch millis) resolved via a **`DISTINCT ON (contactId) … WHERE kind = 'next_step' ORDER BY contactId, createdAt DESC`** subquery left-joined on (so contacts with no next step still appear). Email/phone/linkedin/manager dropped from the row (they live on the detail read); see [ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md).
  - `getCompanyDetail.ts` / `getContactDetail.ts` — server-only reads backing the two detail pages (per ADR 0010; **not** `'use server'`, so pages `await` them directly during SSR). Each fetches the entity's core fields, then its related collections in a `Promise.all`, projected to just the columns the view renders; both return `null` on an unknown id (page → `notFound()`). No pagination (an entity's own footprint is small). Neither adds a permission beyond the `(app)` gate — consistent with all CRM reads being open.
    - `getCompanyDetail` returns core fields + **five** collections. Three are tied directly to `companyId`: `opportunities`, `projects` (projects this company *owns* — `projects.companyId`), and `contacts` (the company's people directory — each `CompanyContact` now also carries its **latest next step**, `nextStep` body + `nextStepAt` epoch millis, via the same **`DISTINCT ON (contactId) … WHERE kind = 'next_step'`** subquery left-joined as `getContactsPage`, so contacts with no next step still list). The other two are keyed off the **referral, not ownership** — company-level aggregations of the matching per-contact collections on `getContactDetail`, both following `opportunitySourceContacts → contacts` (filtered to this company's contacts) so the referred entity's own client company (`opportunities.companyId` / `projects.companyId`) is often a **different** company than the one viewed (hence each carries `clientCompanyId`/`clientCompanyName`, distinct from the ownership-based collections):
      - **`referredOpportunities`** (`CompanyReferredOpportunity[]`) — join `opportunitySourceContacts → contacts → opportunities → companies`, one row per (opportunity, referring contact) ordered by opportunity name. Unlike `referredProjects`, it carries the opportunity's **`status`** (rendered as a pipeline-group badge).
      - **`referredProjects`** (`CompanyReferredProject[]`) — the same shape one hop further (`→ projects → companies` via `opportunities.projectId = projects.id`, the inverted link), one row per (project, referring contact) ordered by project name. No `status`.

      Both collapse their (entity, referrer) rows via a shared generic helper **`groupReferrals<T extends { id; referrerId; referrerName }>(rows)`** — one entry per entity id, distinct `referrers` gathered, input order (a Map) preserved so a name-ordered query yields a name-ordered result (a deal with several source contacts here lists once with multiple referrers). This replaced the previously-inline per-collection grouping.
    - `getContactDetail` returns identity + employer + manager (both optional, resolved via **left joins**, with a `contacts` self-join `alias` for `managerName` like `getContactsPage`), plus the contact's CRM footprint split into `referredOpportunities` (via `opportunitySourceContacts`), `involvedOpportunities` (via `opportunityContacts`, **minus** any already in referred so nothing double-lists), and `referredProjects` — the contact's **`relationshipStrength`** (1–5 or null), and the contact's two entry logs, `notes[]` + `nextSteps[]` (newest-first `EntryView`s from the shared `entryViews.ts`). **Contacts don't link to projects directly**, so "projects" is *derived*: the projects those referred opportunities point at (`inArray` on `opportunities.id`, joining `opportunities.projectId = projects.id` — the inverted link — and **`selectDistinct`** since two referred deals could now share one project). All opportunity/project rows inner-join their company so the cross-company view can name and link the employer.
  - `getOpportunitiesBoard.ts` — server-only read backing the kanban board. Returns `{ cards, cappedTotals }` (**not** a bare array — the shape changed when column capping landed). `cards` is `{ id, name, companyId, companyName, source, status, lineOfBusiness, ownerNames, hasProject, position, createdAt, nextStep, nextStepAt }[]` ordered by `(position, createdAt)`; inner-joins companies for `companyName` (company is required). `lineOfBusiness` is a plain column read backing the board's client-side LOB filter (see [Board + list views](#board--list-views)), left-joins the latest next step via the same **`DISTINCT ON`** subquery as `getContactsPage` (`nextStep` body + `nextStepAt` epoch millis, kanban-card "Next:" line), and resolves owner names via the shared **`resolveOwnerNames`** helper (one grouped query, no N+1). `hasProject` is a **plain column read** (`opportunities.projectId != null`) since the link lives on the opportunity — no separate query — and drives the delivery-stage requirement; `companyId` lets the board prefill the create-project prompt. `createdAt` is epoch millis so the client's `(position, createdAt)` tie-break matches the server's. **Board column capping** — the three high-volume columns (`CAPPED_BOARD_STATUSES` = `maturing`, `closed_won`, `closed_lost`, cap `BOARD_COLUMN_CAP` = 20) return only their 20 most-recently-updated cards; every other column returns in full. Implemented with `row_number()`/`count()` window functions partitioned by `status` in a subquery (windows can't be filtered inline), then a `WHERE` keeping all uncapped-status rows plus `rn ≤ 20` of each capped one. `cappedTotals: Partial<Record<OpportunityStatus, number>>` carries the *full* per-status count for each capped column so the board knows when to render its "Show N more" link (deep-linking into the list view). See [Board + list views](#board--list-views).
  - `getOpportunitiesPage.ts` — server-only read backing the **list view** (re-introduced; an earlier version was removed when the board landed, so this is a fresh implementation, not that one revived). Returns `Page<OpportunityRow>` (`{ id, name, companyId, companyName, status, lineOfBusiness, ownerNames, updatedAt }`), **the app's first filtered/searchable paginated read**: server-side offset/limit + `count` (shares `src/lib/core/pagination.ts` / `clampPage` with `getCompaniesPage`), ordered **`updatedAt desc`** (`id desc` tie-break for stable paging). Accepts `OpportunitiesPageFilters { group?, lineOfBusiness?, query? }` — **stage** (a kanban `OpportunityGroupId`, expanded to the group's leaf statuses via `opportunityGroupById`), **line of business** (`eq`), and a case-insensitive **name search** (`ilike` on opportunity **or** company name). Owner names via the shared `resolveOwnerNames`. No capability gate beyond the `(app)` layout — read parity with `getCompaniesPage`.
  - `opportunityOwnerNames.ts` (`resolveOwnerNames`, server-only, **not** an action) — the grouped owner-name query (`opportunityId → names[]`, alphabetical, absent when none) shared by both `getOpportunitiesBoard` and `getOpportunitiesPage` so the `opportunity_owners → staff` join lives in one place (no N+1).
  - `getOpportunity.ts` — **server-only** read for the detail drawer: an opportunity's core fields, its company, the four people junctions each as `{ id, name }[]` (shaped for the drawer's comboboxes), its **single linked project** (`project: EntityRef | null`, left-joined off `opportunities.projectId` — was `projects: EntityRef[]` before the inversion), and its two entry logs `notes[]` + `nextSteps[]` (via `getOpportunityEntries`). Null if the id is unknown.
  - `loadOpportunityDetail.ts` — a thin **`'use server'`** action wrapping `getOpportunity` for the client-triggered drawer load (the **interactive-read exception** to the server-only rule, like `searchStaff`), gated `crm.edit` — the drawer is edit-only, so this can't leak detail past the write gate. **Named `load*`, not `get*`, deliberately: the `get*` prefix is reserved for server-only reads** (per `.claude/rules/server-actions.md`), so a client-triggered `'use server'` read uses a different verb.
  - `updateOpportunityField.ts` (+ `.schema.ts`) — **the drawer's edit action**, gated `crm.edit`. A **discriminated union on `field`**: each Confirm writes **only the slice that changed** (name / companyId / lineOfBusiness / status / contacts / owners / source+its two referral junctions) rather than re-sending the whole record — so a concurrent edit to another field isn't clobbered and unrelated people junctions aren't rewritten (mirrors `updateContactOwner`). Junction variants run in a transaction that also touches the row (`.returning()`-guarded, bumping `updatedAt`) so a row deleted mid-edit surfaces cleanly. The `status` variant routes through the shared `assertOpportunityTransitionAllowed`, and when it transitions into `closed_won` it runs `confirmRolesOnWon` in the same transaction (see below). `revalidatePath("/opportunities")`.
  - `updateOpportunity.ts` (+ `.schema.ts`) — the **former** whole-record drawer edit (one transaction: updates the row + replaces the four people junctions wholesale via `writeOpportunityLinks`). **Now UI-dead** — `updateOpportunityField` replaced it in the drawer. Kept for now pending a decision; don't treat it as the drawer's write path. Still runs `assertOpportunityTransitionAllowed` for its status guard, and `confirmRolesOnWon` on a win.
  - `updateOpportunityPosition.ts` (+ `.schema.ts`) — the kanban **move** action, gated `crm.edit`. Input `{ id, status, position }` (an absolute destination the board computes client-side from the drop's neighbours); a **single-row update** with no server-side renumbering (wrapped in a transaction when the move wins the deal, so `confirmRolesOnWon` commits atomically), then `revalidatePath("/opportunities")`. See [ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md).
  - `confirmRolesOnWon.ts` (server-only) — on a **genuine transition into `closed_won`** (no-op if already won), flips every `tentative` role tagged with the opportunity to `confirmed`. Called inside each status-mutating action's transaction so the status change and the role flip commit together. This is the CRM half of the tentative → confirmed lifecycle ([ADR 0031](../decisions/0031-opportunity-project-planner-and-role-status.md)); the planner role CRUD lives in `src/actions/projects/` (see [projects.md](./projects.md)). `associateOpportunityProject.ts` (in `src/actions/crm/`, gated **`projects.edit`**) links an opportunity to an existing project with the same-company invariant enforced.
  - `deleteOpportunity.ts` (+ `.schema.ts`) — **delete an opportunity**, gated **`crm.edit`** (the app's **first delete flow**). In one transaction: reads the opportunity's `projectId` and, if set, calls the shared **`detachProjectFromOpportunity`** helper (from `src/actions/projects/`) **before** deleting the opportunity row — so a sole-owned project is deleted (cascading roles + DMs) or, if shared, only this opportunity's roles are removed and the project unlinked, all while `project_roles.opportunityId` provenance is still intact. The opportunity's people/entry junctions cascade on the row delete. Revalidates `/opportunities` + `/projects`. See [projects.md](./projects.md#delete--detach) and [ADR 0033](../decisions/0033-line-of-business-on-role-derived-project-status.md).
  - **Entry actions** (`addContactEntry` / `updateContactEntry` / `deleteContactEntry` + the three `*OpportunityEntry` and three `*CompanyEntry` counterparts) — the notes & next-steps CRUD, **all gated `crm.edit`, no per-entry ownership** (any editor may amend any entry, by product decision — [ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md)). `add*` resolves the author server-side via **`resolveAuthorStaffId.ts`** (wraps `getCurrentStaffAccess`; null when the user has no staff record) and inserts with a `centry`/`oentry`/`coentry` id (guarding the parent FK → user-safe "That … no longer exists"); `update*` writes only `body` (`.returning()`-guarded so a mid-edit delete surfaces cleanly); `delete*` removes the row. Each revalidates the parent's list + detail path. They share **`entries.schema.ts`** (the `crm_entry_kind` enum, per-kind length caps via `requiredText` + a `refineEntryBody` superRefine; `addCompanyEntrySchema` mirrors the contact/opportunity add schemas) and the read helper **`entryViews.ts`** (`getContactEntries`/`getOpportunityEntries`/`getCompanyEntries` → newest-first `EntryView`s with author name resolved and timestamps as epoch millis; `editedAt` set only when `updatedAt > createdAt`). See [Timestamped entries](#timestamped-entries-notes--next-steps).
  - **Delivery-stage enforcement** — the invariant is *an opportunity in a delivery stage has a linked project*, enforced server-side at the **entry points** (so it can't be bypassed by hitting an action directly): `createOpportunity` **rejects** creating in a delivery stage (a new opportunity can't have a project yet), and every status-mutating action (`updateOpportunityPosition`, `updateOpportunityField`, `updateOpportunity`) guards **transitions** through the **single shared `assertOpportunityTransitionAllowed.ts`** (server-only) — it fetches the current status and fires only when the status is *changing into* a requiring stage without a project (`newStatus !== currentStatus && requiresProject(newStatus) && !opportunityHasProject(id)`), throwing a `UserSafeActionError`. Extracting this into one helper removed the copy that was duplicated across the update actions. This is **transition-based, not state-based**: editing (rename/owners) or reordering within a delivery column no longer trips the guard (the earlier every-edit check over-fired). `opportunityHasProject.ts` is the indexed existence check (server-only). The client block is just UX. See [ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md).

Each returns up to `SEARCH_LIMIT` `{ id, name }` by `ilike` match; **a blank query returns nothing** (search only runs once the user types). All three share the query surface via `src/lib/core/search.ts`, which exports `SEARCH_LIMIT` (10) and `searchQuerySchema` (a single `query` string **trimmed on parse**, so whitespace collapses to the blank-query short-circuit). The `searchStaff` / `searchCompanies` **query bodies** are further shared with the Projects domain via `searchStaffByName` / `searchCompaniesByName` in `src/actions/shared/entitySearch.ts` — the CRM and projects search actions each wrap the same query with their own permission gate (`crm.edit` vs `projects.edit`); only `searchContacts` keeps its own body (different columns). Input schemas live in sibling `.schema.ts` files.

Each writes **just `ownerId`** (a staff id, or `null` to unassign — schema is `createUpdateSchema(Table).pick({ ownerId }).extend({ id, ownerId: id.nullable().default(null) })`), then `revalidatePath`s the list + detail routes; a `.returning()` empty set → user-safe "That … no longer exists." Precedent: `updateOpportunityPosition` (a narrow single-column write). See the inline owner field in the UI section.
  - `updateContactRelationshipStrength.ts` (+ `.schema.ts`) — the **relationship-strength-only** contact update, gated `crm.edit`, backing the inline star rating on the contact page. The **narrow counterpart** to `updateContact` on the exact model of `updateContactOwner`: it writes **just `relationshipStrength`** (so a rating change can't clobber a concurrently-edited field and **skips `assertValidManager`**), `.returning()`-guards the row, then `revalidatePath`s `/contacts` + `/contacts/${id}`. Schema lives in its own `.schema.ts` (never export schemas from a `"use server"` file) so the client star component can import it.

## Board + list views

`/opportunities` renders **two views of the same pipeline**, chosen by a **`?view=list` URL param** (default = board) — see [ADR 0038](../decisions/0038-opportunities-list-view-and-board-column-capping.md). The switch (`opportunity-view-toggle.tsx`) is **link-based, not local state** — deliberately, so the choice lives in the URL and the board's "Show more" links can deep-link straight into a filtered list. Both views mount for everyone; edit affordances (drag, drawer open) gate on `crm.edit` as before.

- **Board** (`getOpportunitiesBoard`) — the kanban, unchanged in spirit but now **column-capped**: the three high-volume columns (`CAPPED_BOARD_STATUSES` = Maturing, Won, Lost) show only their 20 (`BOARD_COLUMN_CAP`) most-recently-updated cards, since those columns grow unbounded (Maturing accrues early-stage deals; the closed columns accrue every decided deal forever). Every other column shows its full contents. When a capped column holds more than 20, `opportunity-board-column.tsx` renders a **"Show N more"** link to `?view=list&stage=<group>` — the exhaustive browse lives in the list. The board carries **two client-side filters** over the already-loaded cards — a **name search** and a **line-of-business select** (`SelectFilter`, driven by the card's `lineOfBusiness`) — so both deliberately **cannot surface cards beyond a column's cap** (that's the list view's job), and the "Show N more" count stays the *unfiltered* hidden remainder. When a **LOB filter is active**, the deep-link **carries it through** (`?view=list&stage=<group>&lob=<LOB>`) so the exhaustive list applies both filters. `cappedTotals` (from the read) drives the count; the board maps each capped single-status column's `total − loaded` to the link.
- **List** (`getOpportunitiesPage`) — a **paginated, server-filtered table** (`opportunities-table.tsx`), the fallback for browsing the capped/closed columns and searching the whole pipeline. Columns: Name · Company · Stage · Line of business · Owners · Last updated (ordered `updatedAt desc`). The **name cell opens the same `OpportunityDetailSheet`** the board uses (edit-gated, so it only mounts for `crm.edit` users; others see plain text) — **there is no standalone opportunity detail route**, the drawer is the detail affordance in both views. Pagination reuses `PaginationControls` with `paramKey="oppPage"`.
- **List filters** (`opportunities-list-filters.tsx`) — **stage** (by the 9 kanban `OPPORTUNITY_GROUPS`, expanded server-side to leaf statuses), **line of business**, and a **debounced name/company search**, all **URL-synced** (params `stage`, `lob`, `q`, `oppPage`) via `router.replace` so back-navigation restores filters and the server page re-fetches. Reads its current values from the `params` the page already parsed (no `useSearchParams`, so no Suspense boundary). Any filter change resets `oppPage` to 1. The page validates raw params against the known groups / LOB enum before querying (unknown → no filter). Selects reuse the shared `SelectFilter`/`ALL` primitives in `src/components/form/filters.tsx`.

## Company status tags

A company's **status is derived, not stored** — a set of tags computed from its relationships, so nothing is persisted beyond the manual `isPartner` flag. Full rationale in [ADR 0034](../decisions/0034-company-status-derived-tags.md); the essentials:

- **Three independent tags, in canonical order** — **Partner** (the manual `companies.isPartner` boolean), **Client** (has ≥1 **confirmed** project), **Prospect** (has ≥1 **open** opportunity, i.e. `status NOT IN (closed_won, closed_lost)`). A company can carry several at once, or none. There's no direct company→project link, so Client correlates through `opportunities.companyId → opportunities.projectId`; and since a project no longer stores a status (it's **derived from its roles** — [ADR 0033](../decisions/0033-line-of-business-on-role-derived-project-status.md), `deriveProjectStatus`), "confirmed" is expressed in SQL as *the project has ≥1 `confirmed` role and no `tentative`/`paused` role* (the "least-committed wins" rule, matched inline in `getCompaniesPage`).
- **Pure shared module** `src/lib/crm/company-status.ts` (`companyStatusTags`, `COMPANY_STATUS_TAGS`, `COMPANY_STATUS_LABELS`) — no `db`/drizzle, no UI — mirrors the derived-field pattern in `project-derived.ts` so the read computing the flags and the table rendering the badges share one definition of the tags/order/labels.
- **`CLOSED_OPPORTUNITY_STATUSES`** (`closed_won`, `closed_lost`) in `src/lib/crm/opportunity.ts` is the single source of truth for terminal statuses; the prospect check negates it.
- **Computed once, in `getCompaniesPage`** as correlated `EXISTS` subqueries (`isClient`/`isProspect`) so the page stays a single query; `isPartner` is a plain column read. Rendered by `companies-table.tsx` as `secondary` badges (`EmptyCell` when none apply).
- **A "Referrer" tag was considered and dropped for now** — referrals are tracked per-contact/per-staff (the opportunity source junctions), not per-company; no clean company→referral link exists yet.

## Timestamped entries (notes & next steps)

**Contacts and opportunities** carry two running logs of authored, timestamped free-text entries — **notes** (longer prose, "what happened", ≤5000 chars) and **next steps** (short reminders, "what's planned", ≤500); **companies** carry the same log but **only surface `note`** in the UI (no next-step composer). Full rationale in [ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md); the essentials:

- **Three concrete tables, not a polymorphic one** — `contact_entries` + `company_entries` (`crm-schema.ts`) and `opportunity_entries` (`opportunities-schema.ts`), each a real FK to its parent (`onDelete: cascade`). Same shape: `id`, `<parent>Id`, `kind`, `body`, `authorStaffId` (FK → `staff`, `onDelete: set null` — attribution, not ownership), `createdAt`/`updatedAt`, index on `(<parent>Id, kind, createdAt)`. Row types `ContactEntry` / `CompanyEntry` / `OpportunityEntry`. `company_entries` carries the `next_step` kind at the DB level for shape consistency even though the UI writes only `note`.
- **One shared pg enum `crm_entry_kind` = `('note', 'next_step')`** distinguishes the kinds within a table (declared in `crm-schema.ts`, imported by `opportunities-schema.ts`). Notes and next steps differ **only by `kind` + validation length**.
- **No per-entry ownership** — any `crm.edit` holder may add/edit/delete any entry (product decision: shared team logs). All nine actions (add/update/delete × contact/opportunity/company) gate solely on `crm.edit`. The author is resolved server-side (`resolveAuthorStaffId.ts`), never trusted from the client.
- **The scalar `opportunities.nextSteps` column is gone** — migrated into `opportunity_entries` as the first `next_step` entry, then dropped, and removed from the create/update schemas, actions, `updateOpportunityField` union, and the add-opportunity form.
- **The latest next step surfaces in list + detail views** — `getContactsPage` (`ContactRow`), `getOpportunitiesBoard` (`OpportunityBoardCard`), and `getCompanyDetail` (each `CompanyContact`) each left-join a `DISTINCT ON (parentId) … WHERE kind = 'next_step'` subquery, so the newest next step rides along in one query (no N+1) and drives the contacts-table "Next steps" column, the kanban card's "Next:" line, and the company detail page's Contacts "Next steps" column. The **companies list** table stays out of this on purpose — a company owns no entries, and rolling up contact/opportunity next steps was declined.
- **Shared plumbing** — schema `entries.schema.ts`, read helper `entryViews.ts` (`EntryView`, epoch-millis timestamps, `editedAt` only when edited), UI `entry-log.tsx` (one component, `variant`-switched across contact/opportunity/company; composer sized per kind — `Input`+Enter-submits for next steps, `Textarea` for notes). Entry timestamps render with `formatShortDate` (from `src/lib/format/format.ts`).

## Relationship strength

A contact carries an optional **relationship strength** — how strong our relationship with them is, on a **1–5** scale, **null = not yet rated**. It's a plain `integer` on `contacts` (migration `drizzle/0001_past_agent_zero.sql`); the level meanings live in the pure, client-importable module **`src/lib/crm/relationship-strength.ts`** — **1** New/Unestablished, **2** Weak, **3** Developing, **4** Strong Supporter, **5** Champion/Trusted Partner — the single source shared by the column comment, the zod schemas, and the star UI (same pattern as `feedback-rating.ts`, [ADR 0016](../decisions/0016-junction-table-and-shared-enum-conventions.md)).

- **Edited inline as stars, no confirm step.** The contact detail page renders `InlineRelationshipStrengthField` (a reusable `StarRating`, `src/components/form/star-rating.tsx`) in the meta sidebar above the owner; **clicking a star writes immediately** via the narrow **`updateContactRelationshipStrength`** action (writes only that column, skips `assertValidManager` — mirrors `updateContactOwner`), with optimistic local state that reverts on error. Read-only stars for non-`crm.edit` viewers.
- **Round-trips through the full record too.** `updateContact` accepts and re-writes `relationshipStrength`, and the edit dialog keeps it in `defaultValues` (no rendered field) so a full-record save leaves it unchanged — exactly like `ownerId`.
- **Surfaced only on the contact detail page** (as the sidebar rating). Not shown on the contacts list table or company detail Contacts table.
- **Seed** assigns a random 1–5 to ~85% of contacts (`scripts/seed/crm.ts`).

## Location

Companies and contacts each carry an optional **location** — where they're based, a **free-text `"City, CC"` label** (e.g. `"Toronto, CA"`), **null when unknown**. It's a plain nullable `text()` column, **not a FK or enum** (migration `drizzle/0005_secret_zarek.sql`); the full rationale and the shared dataset (which `staff` also uses) live in [../data-model.md](../data-model.md#location-free-text).

- **Picked from a static world-cities list, not typed freely.** The value is chosen through a searchable `EntityCombobox` backed by `searchCities` (`src/actions/cities/searchCities.ts` → the pure `@/lib/cities/cities` module over `world-cities.json`). The label string itself is stored — there's no city id (the combobox's `EntityOption` uses the label for both `id` and `name`).
- **Edited inline on the detail page**, mirroring the owner field: `InlineLocationField` (`inline-location-field.tsx`) reads as text ("Unknown" when unset) until the pencil, then swaps in the city picker with Confirm/Cancel; Confirm calls the location-only action (`updateCompanyLocation`/`updateContactLocation`, `crm.edit`-gated), whose `revalidatePath` refreshes the server-rendered display.
- **`searchCities` is auth-only, deliberately un-gated by capability** — it exposes only static public reference data (see [Authorization](#authorization)).
- **Edited on the two detail pages** (sidebar), **displayed and now *filterable* on both list tables.** The companies and contacts lists each carry a **location filter** — a city picker + a **"Search nearby"** switch that widens the match to cities within `NEARBY_RADIUS_KM` (~100 km) of the picked one. Both filter **server-side, URL-driven** (params `city` + `nearby=1`), mirroring the companies name/status filter pattern: the pages parse the params and pass `{ city, nearby }` into `getCompaniesPage`/`getContactsPage`, whose `where` pushes `inArray(location, nearby ? citiesNear(city) : [city])` (companies AND-combined with its name/status conditions). **This gave contacts its first-ever list filter** (a new `ContactsListFilters` bar + `ContactListFilters`/`contactsWhere` on the read; the companies filter bar just gained a third control). The shared control is `location-filter-control.tsx`; `citiesNear`/`NEARBY_RADIUS_KM` live in `@/lib/cities/cities` — see [../data-model.md](../data-model.md#location-filters). **Seed** assigns a US/CA city to ~80% of companies and ~70% of contacts (`scripts/seed/crm.ts`).

## Authorization

**Reads are open** — any signed-in user can browse all companies, contacts, and opportunities (the `(app)` gate is the boundary). **All CRM writes** are gated by a single flat capability (no ownership dimension): **`crm.edit`** (renamed from `contacts.edit` when opportunities landed), granted to `sales`, `manager`, `admin`. It covers creating/editing companies, contacts, *and* opportunities — including the inline company/contact creation, the inline owner, **relationship-strength** (`updateContactRelationshipStrength`), and **location** (`updateCompanyLocation`/`updateContactLocation`) writes, the opportunity detail drawer's edit/move, **deleting an opportunity** (`deleteOpportunity`, which also detaches/cleans up its project), `loadOpportunityDetail` (edit-only read), the six notes/next-steps entry actions (add/update/delete on both parents, **no per-entry ownership**), and every type-ahead picker (`searchCompanies`/`searchContacts`/`searchStaff`) that backs entry. (**Exception:** `searchCities` (the location picker's type-ahead) and `nearbyCityLabels` (the "search nearby" resolver) are **auth-only with no capability gate** — both serve only static, public world-cities reference data, never user/tenant data.) Declared in action metadata; each page hides its "Add" dialog(s) (via one `canEdit` flag) for users without the capability, the opportunities board renders **static (no drag-and-drop) with no clickable drawer** for non-editors, and the opportunities list view renders **plain-text names (no drawer)** for them (the list read itself, `getOpportunitiesPage`, is ungated like all CRM reads).

**Cross-domain RBAC nuance (delivery-stage handoff).** Creating a project needs **`projects.edit`**, which `sales` does *not* have (only `crm.edit`). The board is passed a separate `canCreateProject` flag (`projects.edit`): a user who *can* create projects gets the prefilled create-project prompt when they move a card into a delivery stage without one; a user who can't (e.g. `sales`) gets a **toast telling them a delivery manager must create the project first**. No permission was widened — the division of labour is deliberate ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)). See [domains/permissions.md](./permissions.md).

## Key flows

- **Pipeline management** (built) — create Opportunities and move them through the pipeline (`maturing` → `lead` → … → `closed_won`/`closed_lost`) by **dragging cards** across the kanban board (`updateOpportunityPosition`). Multi-substatus groups (Scoping/Allocating/Closing) collapse into one column or expand into subcolumns; dropping onto a collapsed group picks the first substatus (if dragged from earlier in the pipeline) or the last (from later). Non-editors (`!crm.edit`) get a static, read-only board.
- **Browse opportunities (board + list)** (built) — the board caps its three high-volume columns (Maturing/Won/Lost) at 20 cards and offers a **"Show N more"** deep-link into the **list view** (`?view=list`), a paginated table filterable by stage / line of business / name search. The board itself also carries **client-side name + line-of-business filters** over its loaded cards (loaded-only by design — an active LOB filter rides through the "Show more" deep-link so the list applies it too). The list's name cell opens the same detail drawer; there's no standalone detail route. See [Board + list views](#board--list-views).
- **Edit a company / contact** (built) — from the detail page (`/companies/[id]`, `/contacts/[id]`), a `crm.edit` holder opens an edit dialog covering the core fields and — for contacts — company and manager. Saved via `updateCompany` / `updateContact` (a single `db.update(...).returning()` by id; an empty `returning` → user-safe "That … no longer exists"). `updateContact` re-runs the same-company **manager check** (`assertValidManager`, now guarding against self-management too) and the **email-uniqueness** map that `createContact` uses. Both revalidate the list path *and* the detail path.
- **Reassign a company / contact owner** (built) — the optional **staff owner** is edited **in place** on the detail page (`InlineOwnerField`), separate from the edit dialog — matching the opportunity drawer's per-field pattern. Confirming saves through the dedicated **owner-only** action (`updateCompanyOwner` / `updateContactOwner`), which writes just `ownerId` (or `null` to unassign) so it neither clobbers concurrently-edited fields nor re-runs the contact manager rule; both revalidate the list + detail paths. The edit dialogs still carry `ownerId` in their form defaults so a full-record save round-trips the current owner unchanged.
- **Edit an opportunity** (built) — clicking a card opens the **detail drawer** (`opportunity-detail-sheet.tsx`), which lazy-loads via `loadOpportunityDetail`. Its **Details** tab edits line of business, source, company, contacts, owners (the meta column) **one field at a time in place** (pencil → Confirm/Cancel), plus name and status from the header; each Confirm calls the field-scoped `updateOpportunityField`, which writes only that field's slice. This is CRM's first field-edit flow beyond the drag-move, and the first use of the shared `InlineEditField` primitive. Notes & next steps share the Details tab's right column (see next flow), not the meta fields.
- **Log notes & next steps** (built) — on a contact detail page or the opportunity drawer's **Details** tab, a `crm.edit` holder appends timestamped **notes** ("what happened", ≤5000 chars) and **next steps** ("what's planned", ≤500), each shown newest-first with author + timestamp and inline edit/delete. **Any editor may amend any entry** (no per-entry ownership). The author is stamped server-side from the session. The **most recent next step** surfaces on the contacts list table and on kanban cards. This replaced the opportunity's old scalar `nextSteps` field, whose value was migrated in as the first next-step entry. See [ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md).
- **Opportunity → Project handoff + delivery-stage requirement** (built) — **transitioning** an opportunity into the **Allocating** group or later (`requiresProject`, Closed – Lost excepted) requires a linked project; creating one directly in a delivery stage is forbidden. Dragging a card into a delivery stage without a project is **blocked** (only on a genuine stage change — an intra-column reorder never prompts); a `projects.edit` holder is shown a **one-click confirm dialog** (no form — `createProjectFromOpportunity` inherits the opportunity's name + company), and on creation the pending move completes; a user without `projects.edit` (e.g. `sales`) gets a toast pointing to a delivery manager. Enforced server-side at the entry points (creation ban + transition guard in both status-mutating actions). See [projects.md](./projects.md), [ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md), and [flows.md](../flows.md).

## Connects to

- **Staff** — an opportunity's `owners` and referral `source_staff` are `staff` rows (FK, cascade). This is the first CRM ↔ staff link. A **company and a contact also each carry an optional `ownerId` → `staff`** (`onDelete: set null`) — the accountable owner. Deliberately **set-null, not cascade** (unlike the opportunity junctions): an owner is a soft pointer, so removing a staff member should only unassign them, never delete the company/contact.
- **Projects** — a project belongs to a `companies` row (required FK, `restrict`). The opportunity link now lives on the **opportunity** side (`opportunities.projectId`, nullable FK, `restrict`, **many opportunities → one project**), populated by the handoff/planner flow, and `requiresProject` makes a project a precondition for advancing an opportunity to a delivery stage. `project_roles.opportunityId` (set-null) records which deal created each role, and roles auto-confirm on the win. See [projects.md](./projects.md), [ADR 0019](../decisions/0019-project-opportunity-link.md), [ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md), [ADR 0031](../decisions/0031-opportunity-project-planner-and-role-status.md).
- **Allocations / Timesheets** via the Project a won Opportunity will create (proposed).
- See [../data-model.md](../data-model.md) and [../flows.md](../flows.md).

## Open questions

- Probability/weighting per status and pipeline revenue forecasting (value × probability) — not modelled; opportunities carry no monetary value field yet.
- **Edit/delete** — **companies, contacts, and opportunities now all have edit** (companies/contacts via a whole-dialog `updateCompany`/`updateContact`; the opportunity detail drawer via per-field `updateOpportunityField` + drag-move), so a contact's `companyId`/`linkedinUrl`/`managerId`/`ownerId` (and a company's owner) can now be changed after creation. **Opportunities now also have delete** (`deleteOpportunity`, gated `crm.edit` — the app's **first delete flow**): it cleans up the linked project first (via the shared `detachProjectFromOpportunity` — a sole-owned project is deleted, else this opportunity's roles are removed and it's unlinked), *then* deletes the opportunity row (its people/entry junctions cascade). Running the detach **before** the row delete keeps `project_roles.opportunityId` provenance intact — see [projects.md](./projects.md#delete--detach). **Companies and contacts still have no delete** — the `onDelete: restrict` on `opportunities.companyId` means a company-delete flow must handle live opportunities.
