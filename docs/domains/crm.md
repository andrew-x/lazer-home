# Domain: CRM

**Status: growing.** Companies, contacts, and opportunities (data, reads, create, edit, and `/companies` + `/contacts` + `/opportunities` UIs) all exist. **Companies and contacts are now full CRUD (create/read/edit; no delete yet)** — each has a **detail page** at `/companies/[id]` and `/contacts/[id]` — the app's **first drill-down detail views** (see the UI section below) — carrying an **Edit** affordance (for `crm.edit` holders) that opens a dialog wired to `updateCompany`/`updateContact` for the core fields. The optional **staff owner** (the accountable staff member) is **edited in place on the detail page** (`InlineOwnerField`, mirroring the opportunity drawer's per-field edits), not in that dialog. Companies and contacts now live on **two separate pages** (`/companies` and `/contacts`) — the old combined "Companies & Contacts" page was split ([design spec](../superpowers/specs/2026-07-09-split-companies-contacts-design.md)). `/opportunities` is now a **drag-and-drop kanban board** (not a table), with 14 leaf statuses organized into ordered pipeline groups — see [ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md). Opportunities now carry a **required line of business** (the shared enum, [ADR 0025](../decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)), and their table + junctions now live in **their own schema file** `src/lib/db/opportunities-schema.ts` (split out of `crm-schema.ts`, which keeps only companies + contacts). Clicking a card opens a **detail drawer** — a wide (`max-w-3xl`) three-tab read-first view (**Info** + **Notes** + **Project plan**) whose Info fields **edit one at a time in place** (per-field pencil → Confirm/Cancel), covering company, status, line of business, owners, source, contacts, referral sources. Both **contacts and opportunities** also carry two running logs of timestamped, authored free-text entries — **notes** ("what happened") and **next steps** ("what's planned") — see [Timestamped entries](#timestamped-entries-notes--next-steps) and [ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md); the scalar `opportunities.nextSteps` column that predated them is gone. Projects now exist too ([projects.md](./projects.md)), and the **Opportunity → Project handoff is now built**: `projects.opportunityId` is populated when a project is created from an opportunity, and a **delivery-stage rule** requires a linked project before an opportunity can advance to Allocating or later ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)). Manages the companies and people we deal with (clients, partners, prospects) and the pipeline deals we're trying to win, all of which feed delivery.

## Purpose

Track who we sell to and work with, and (eventually) what we're trying to win, so won work flows cleanly into Projects and Allocations.

## Key entities

- **Company** (built) — an organisation we deal with. **`isPartner` is a standalone flag** marking whether the company is a partner (default `false`); it is **not** a client-vs-partner dichotomy — a company may be neither a client nor a partner, and nothing requires it to be either. We deliberately modelled this as *Company* rather than the narrower *Client* the early design assumed — see [ADR 0015](../decisions/0015-crm-company-over-client.md). Fields: `name` (required), `websiteUrl` (optional), `isPartner`, and an optional **`ownerId`** — a nullable FK → **`staff`** (`onDelete: set null`), the staff member accountable for the relationship; cleared (not blocked) if that staff row is removed. This is a CRM company/contact ↔ staff link (mirrors `opportunityOwners.staffId`; see [Connects to](#connects-to)). Table `companies`, id prefix `company`.
- **Contact** (built) — a person, optionally attached to a Company. Fields: `firstName`/`lastName` (required), `email` (required, **unique**), `phone` (optional; shown as a `tel:` link on the detail page), `companyId` (optional FK → `companies`, `onDelete: set null`), `role` (optional free-text job title, e.g. "CTO"), `linkedinUrl` (optional; validated by the shared `optionalUrl` schema, same as company `websiteUrl`; shown as an external "Profile" link on the detail page), and **`managerId`** — an optional **self-referential** FK → `contacts.id` (`onDelete: set null`), "managed by" another contact. This is the repo's **first self-referential FK**; by business rule a manager must be an existing contact **at the same company**, enforced app-side (not in the DB, via `assertValidManager` in `contactChecks.ts`) — see [ADR 0022](../decisions/0022-contact-manager-self-reference.md). Also carries an optional **`ownerId`** — a nullable FK → **`staff`** (`onDelete: set null`), the accountable staff member; cleared if that staff row is removed (same set-null convention as `companies.ownerId`). Table `contacts`, id prefix `contact`. A contact also carries two running logs of **notes** and **next steps** (`contact_entries` — see [Timestamped entries](#timestamped-entries-notes--next-steps)). Schema in `src/lib/db/crm-schema.ts`; see [../data-model.md](../data-model.md).
- **Opportunity** (built) — a pipeline deal that always belongs to a Company. Fields: `name` (required), required `companyId` (FK → `companies`, **`onDelete: restrict`** — a company with live opportunities can't be deleted, unlike its optional/set-null contacts), `source` + `status` (pgEnums), **required `lineOfBusiness`** (the shared `lineOfBusinessEnum` — which practice the deal belongs to; a project created from it defaults to the same, [ADR 0025](../decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)), and **`position`** (see below). **No scalar `nextSteps` column** — the old single next-step field was migrated into the `opportunity_entries` log and dropped ([ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md); see [Timestamped entries](#timestamped-entries-notes--next-steps)). Table `opportunities`, id prefix `opp` — **now in its own schema file** `src/lib/db/opportunities-schema.ts` (split out of `crm-schema.ts`, which keeps only `companies` + `contacts`; [ADR 0025](../decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)). Its people are modelled with **four junction tables** (see below): related `contacts`, `owners` (→ `staff`), and referral sources split into `source_contacts` and `source_staff`. An Opportunity produces a Project once it reaches delivery stages; the link column lives on the **project** side (`projects.opportunityId`, nullable FK → `opportunities`, `onDelete: restrict`, **at most one project per opportunity** — enforced by a **partial unique index** on the FK, [ADR 0019 amendment](../decisions/0019-project-opportunity-link.md)), now **populated by the handoff flow** ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)), so a delete of an opportunity with its live project is blocked. See [ADR 0019](../decisions/0019-project-opportunity-link.md).
  - **`source`** (`opportunity_source`): `inbound`, `farming`, `extension`, `change_request`, `staff_referral`, `contact_referral`.
  - **`status`** (`opportunity_status`): a **flat 14-value leaf enum in strict pipeline order** — `maturing`, `lead`, `qualifying`, `scoping_awaiting_info`, `scoping`, `scoping_reviewing`, `allocating_awaiting_profiles`, `allocating_introing_profiles`, `negotiating`, `closing_awaiting_contracts`, `closing_redlining`, `closing_awaiting_signatures`, `closed_won`, `closed_lost`. The DB knows only leaves; the two-level **grouping** (below) is derived in code. (Restructured from the old flat 7-value enum; migration `0020` remaps legacy `closing` → `closing_awaiting_contracts`.)
  - **Pipeline groups** — `src/lib/opportunity-pipeline.ts` (a pure, client-importable module, mirroring `line-of-business.ts`) defines `OPPORTUNITY_GROUPS`: ordered groups → their leaf substatuses. In order: **Maturing, Lead, Qualifying** (single-status), **Scoping** (awaiting info / scoping / reviewing), **Allocating** (awaiting profiles / introing profiles), **Negotiating**, **Closing** (awaiting contracts / redlining / awaiting signatures), **Won** (`closed_won`), **Lost** (`closed_lost`). A **module-load assertion** (+ `opportunity-pipeline.test.ts`) requires the flattened groups to equal `OPPORTUNITY_STATUSES` in the same order, so they can't drift. Also exports `resolveTargetStatus` (which leaf a card takes when dropped on a collapsed group), `computePosition` (fractional-index midpoint), and **`requiresProject(status)`** — true from the **Allocating** group onward (Allocating → Negotiating → Closing → Won), Closed – Lost excepted: the delivery-stage rule that an opportunity can't advance without a linked project ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)). See [ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md).
  - **`position`** (`doublePrecision NOT NULL DEFAULT 0`) — a **single global fractional index** for manual kanban ordering. Cards in a column sort by `(position, createdAt)`; a drag writes the midpoint between its new neighbours (one-row update), a new opportunity gets `max(position)+1` (lands at the end of its column). Index `opportunities_status_position_idx` on `(status, position)`. No fractional-index rebalancing — an accepted limitation at this scale ([ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md)).
  - Both enum value tuples (`OPPORTUNITY_SOURCES`/`OPPORTUNITY_STATUSES`), their union types (`OpportunitySource`/`OpportunityStatus`), and their `SOURCE_LABELS`/`STATUS_LABELS` are declared **once** in `src/lib/opportunity.ts` (a pure, client-importable module — no `db`/drizzle, no UI, mirroring `line-of-business.ts`/`currency.ts`), and everything imports them **directly** from there: the `pgEnum` in `opportunities-schema.ts`, the zod schema, and the display components. This fixed the old layering where the tuples lived in `createOpportunity.schema.ts` (imported *upward* by the pgEnum) and the labels in `opportunity-display.ts`. The **back-compat re-export shims are now gone** — `createOpportunity.schema.ts` no longer re-exports the enums, and `opportunity-display.ts` keeps only `STATUS_SELECT_LABELS` (the group-qualified dropdown labels it derives). Per-leaf display labels are `STATUS_LABELS`; group labels live in `opportunity-pipeline.ts`.

## What's built

- **Schema** — `src/lib/db/crm-schema.ts` (`companies`, `contacts`, **`contact_entries`**, and the shared **`crm_entry_kind`** pgEnum) + `src/lib/db/opportunities-schema.ts` (`opportunities` + four junction tables + **`opportunity_entries`** + the `source`/`status` pgEnums + `lineOfBusiness`), both barrelled by `src/lib/db/schema.ts`. The two `*_entries` tables (notes & next steps) share the `crm_entry_kind` enum — see [Timestamped entries](#timestamped-entries-notes--next-steps) and [ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md). The opportunities file imports `companies`/`contacts` from `./crm-schema` and `staff`/`lineOfBusinessEnum` from `./staff-schema` ([ADR 0025](../decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)). Migrations `drizzle/0012_spotty_mysterio.sql` (companies/contacts), `drizzle/0013_clean_firelord.sql` (adds `contacts.phone`), `drizzle/0014_nostalgic_crusher_hogan.sql` (opportunities + junctions + enums), `drizzle/0020_unique_blue_shield.sql` (**hand-edited**: recreates `opportunity_status` as the 14-value leaf enum, remapping legacy `closing` → `closing_awaiting_contracts`; adds `position` + backfills it via `row_number()` per status; adds `opportunities_status_position_idx`), `drizzle/0021_glamorous_goliath.sql` (adds `contacts.linkedinUrl` + self-referential `contacts.managerId`), `drizzle/0024_harsh_diamondback.sql` (adds `opportunities.line_of_business` — backfilled to `CORE` via a temporary default, then dropped, so it's NOT NULL with no default). The schema split itself is a pure code reorg (no migration).
- **Junction tables** — the repo's **first** many-to-many join tables: `opportunity_contacts`, `opportunity_owners` (→ `staff`), `opportunity_source_contacts`, `opportunity_source_staff` (→ `staff`). All follow one convention (documented in [../data-model.md](../data-model.md#junction-tables--the-first-many-to-many-pattern)): a **surrogate `text` PK** via `generateId` (no composite PKs — repo convention), a **`unique(...)` on the FK pair** for set-semantics (no duplicate links), an **`index(...)` on the non-opportunity FK** for reverse lookups, and **both FKs `onDelete: "cascade"`** (a link is meaningless without both endpoints).
- **Server layer** — `src/actions/crm/`:
  - `getCompaniesPage.ts` / `getContactsPage.ts` — server-only reads (per ADR 0010), **server-side offset/limit pagination** with a `count` and a page clamped into range (out-of-bounds query params can't return an empty page). Both share `src/lib/pagination.ts`, which exports `CRM_PAGE_SIZE` (20), `clampPage` (the range-clamp arithmetic — there's always ≥1 page, even when empty), and the `Page<T>` envelope (`{ rows, total, page, pageSize, pageCount }`) they return. Contacts left-join companies to resolve `companyName` (company is optional). `ContactRow` was **slimmed to what the list table renders** — `id`, `firstName`/`lastName`, `role`, `companyId`/`companyName`, and the **latest next step** (`nextStep` body + `nextStepAt` epoch millis) resolved via a **`DISTINCT ON (contactId) … WHERE kind = 'next_step' ORDER BY contactId, createdAt DESC`** subquery left-joined on (so contacts with no next step still appear). Email/phone/linkedin/manager dropped from the row (they live on the detail read); see [ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md).
  - `getCompanyDetail.ts` / `getContactDetail.ts` — server-only reads backing the two detail pages (per ADR 0010; **not** `'use server'`, so pages `await` them directly during SSR). Each fetches the entity's core fields, then its related collections in a `Promise.all`, projected to just the columns the view renders; both return `null` on an unknown id (page → `notFound()`). No pagination (an entity's own footprint is small). Neither adds a permission beyond the `(app)` gate — consistent with all CRM reads being open.
    - `getCompanyDetail` returns core fields + **five** collections. Three are tied directly to `companyId`: `opportunities`, `projects` (projects this company *owns* — `projects.companyId`), and `contacts` (the company's people directory — each `CompanyContact` now also carries its **latest next step**, `nextStep` body + `nextStepAt` epoch millis, via the same **`DISTINCT ON (contactId) … WHERE kind = 'next_step'`** subquery left-joined as `getContactsPage`, so contacts with no next step still list). The other two are keyed off the **referral, not ownership** — company-level aggregations of the matching per-contact collections on `getContactDetail`, both following `opportunitySourceContacts → contacts` (filtered to this company's contacts) so the referred entity's own client company (`opportunities.companyId` / `projects.companyId`) is often a **different** company than the one viewed (hence each carries `clientCompanyId`/`clientCompanyName`, distinct from the ownership-based collections):
      - **`referredOpportunities`** (`CompanyReferredOpportunity[]`) — join `opportunitySourceContacts → contacts → opportunities → companies`, one row per (opportunity, referring contact) ordered by opportunity name. Unlike `referredProjects`, it carries the opportunity's **`status`** (rendered as a pipeline-group badge).
      - **`referredProjects`** (`CompanyReferredProject[]`) — the same shape one hop further (`→ projects → companies`), one row per (project, referring contact) ordered by project name. No `status`.

      Both collapse their (entity, referrer) rows via a shared generic helper **`groupReferrals<T extends { id; referrerId; referrerName }>(rows)`** — one entry per entity id, distinct `referrers` gathered, input order (a Map) preserved so a name-ordered query yields a name-ordered result (a deal with several source contacts here lists once with multiple referrers). This replaced the previously-inline per-collection grouping.
    - `getContactDetail` returns identity + employer + manager (both optional, resolved via **left joins**, with a `contacts` self-join `alias` for `managerName`), plus the contact's CRM footprint split into `referredOpportunities` (via `opportunitySourceContacts`), `involvedOpportunities` (via `opportunityContacts`, **minus** any already in referred so nothing double-lists), and `referredProjects` — and the contact's two entry logs, `notes[]` + `nextSteps[]` (newest-first `EntryView`s from the shared `entryViews.ts`). **Contacts don't link to projects directly**, so "projects" is *derived*: projects whose `opportunityId` is one of the contact's referred opportunity ids (`inArray`). All opportunity/project rows inner-join their company so the cross-company view can name and link the employer.
  - `getOpportunitiesBoard.ts` — **non-paginated** server-only read backing the kanban board (a consultancy's pipeline is small, so no paging). Returns *all* cards `{ id, name, companyId, companyName, source, status, ownerNames, hasProject, position, createdAt, nextStep, nextStepAt }` ordered by `(position, createdAt)`; inner-joins companies for `companyName` (company is required), left-joins the latest next step via the same **`DISTINCT ON`** subquery as `getContactsPage` (`nextStep` body + `nextStepAt` epoch millis, kanban-card "Next:" line), and resolves owner names **and** the `hasProject` flag via **two grouped follow-up queries** over all card ids (no N+1). `hasProject` (any linked `projects.opportunityId`) drives the delivery-stage requirement; `companyId` lets the board prefill the create-project prompt. `createdAt` is returned as epoch millis so the client's `(position, createdAt)` tie-break matches the server's. (Replaced the removed `getOpportunitiesPage.ts` — opportunities are no longer paginated.)
  - `getOpportunity.ts` — **server-only** read for the detail drawer: an opportunity's core fields, its company, the four people junctions each as `{ id, name }[]` (shaped for the drawer's comboboxes), its linked projects, and its two entry logs `notes[]` + `nextSteps[]` (via `getOpportunityEntries`). Null if the id is unknown. (The old scalar `nextSteps` selection is gone — [ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md).)
  - `loadOpportunityDetail.ts` — a thin **`'use server'`** action wrapping `getOpportunity` for the client-triggered drawer load (the **interactive-read exception** to the server-only rule, like `searchStaff`), gated `crm.edit` — the drawer is edit-only, so this can't leak detail past the write gate. **Named `load*`, not `get*`, deliberately: the `get*` prefix is reserved for server-only reads** (per `.claude/rules/server-actions.md`), so a client-triggered `'use server'` read uses a different verb.
  - `updateOpportunityField.ts` (+ `.schema.ts`) — **the drawer's edit action**, gated `crm.edit`. A **discriminated union on `field`**: each Confirm writes **only the slice that changed** (name / companyId / lineOfBusiness / status / contacts / owners / source+its two referral junctions) rather than re-sending the whole record — so a concurrent edit to another field isn't clobbered and unrelated people junctions aren't rewritten (mirrors `updateContactOwner`). Junction variants run in a transaction that also touches the row (`.returning()`-guarded, bumping `updatedAt`) so a row deleted mid-edit surfaces cleanly. The `status` variant routes through the shared `assertOpportunityTransitionAllowed`. `revalidatePath("/opportunities")`.
  - `updateOpportunity.ts` (+ `.schema.ts`) — the **former** whole-record drawer edit (one transaction: updates the row + replaces the four people junctions wholesale via `writeOpportunityLinks`). **Now UI-dead** — `updateOpportunityField` replaced it in the drawer. Kept for now pending a decision; don't treat it as the drawer's write path. Still runs `assertOpportunityTransitionAllowed` for its status guard.
  - `updateOpportunityPosition.ts` (+ `.schema.ts`) — the kanban **move** action, gated `crm.edit`. Input `{ id, status, position }` (an absolute destination the board computes client-side from the drop's neighbours); a **single-row update** with no server-side renumbering, then `revalidatePath("/opportunities")`. See [ADR 0021](../decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md).
  - **Delivery-stage enforcement** — the invariant is *an opportunity in a delivery stage has a linked project*, enforced server-side at the **entry points** (so it can't be bypassed by hitting an action directly): `createOpportunity` **rejects** creating in a delivery stage (a new opportunity can't have a project yet), and every status-mutating action (`updateOpportunityPosition`, `updateOpportunityField`, `updateOpportunity`) guards **transitions** through the **single shared `assertOpportunityTransitionAllowed.ts`** (server-only) — it fetches the current status and fires only when the status is *changing into* a requiring stage without a project (`newStatus !== currentStatus && requiresProject(newStatus) && !opportunityHasProject(id)`), throwing a `UserSafeActionError`. Extracting this into one helper removed the copy that was duplicated across the update actions. This is **transition-based, not state-based**: editing (rename/owners) or reordering within a delivery column no longer trips the guard (the earlier every-edit check over-fired). `opportunityHasProject.ts` is the indexed existence check (server-only). The client block is just UX. See [ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md).
  - `searchCompanies.ts` / `searchContacts.ts` / `searchStaff.ts` — type-ahead pickers, `secureActionClient` gated `crm.edit` (they back CRM entry, itself a write, so the gate stops them being used to enumerate the roster past the page gate). Each returns up to `SEARCH_LIMIT` `{ id, name }` by `ilike` match; **a blank query returns nothing** (search only runs once the user types). All three share the query surface via `src/lib/search.ts`, which exports `SEARCH_LIMIT` (10) and `searchQuerySchema` (a single `query` string **trimmed on parse**, so whitespace collapses to the blank-query short-circuit). The `searchStaff` / `searchCompanies` **query bodies** are further shared with the Projects domain via `searchStaffByName` / `searchCompaniesByName` in `src/actions/shared/entitySearch.ts` — the CRM and projects search actions each wrap the same query with their own permission gate (`crm.edit` vs `projects.edit`); only `searchContacts` keeps its own body (different columns). Nuances: `searchStaff` filters `isActive = true` (**only active staff**), and `searchContacts` matches **first name / last name / email** (not just name) and takes an **optional `companyId` filter** that scopes results to one company — the contact manager picker passes it so a manager can only be a colleague at the same company. All also share the LIKE-metacharacter escaper `escapeLike` in `src/lib/like.ts` (extracted from `searchCompanies`).
  - `createOpportunity.ts` — gated `crm.edit`. **Rejects creating in a delivery stage** (any `requiresProject(status)` — Allocating onward): a brand-new opportunity can't have a linked project yet, so that state is unreachable; the error points the user to create it earlier and add a project as it advances. Otherwise one **transaction**: reads `max(position)` and inserts the opportunity with `position = max+1` (so it lands at the end of its board column), then writes the four people-junctions via the shared `writeOpportunityLinks` (see below). Related entities (company, contacts) are created via their own actions first; this only consumes their ids. `revalidatePath("/opportunities")`.
  - `opportunityLinks.ts` — shared junction helpers (not actions), each **deduping its id list** first so a repeat can't trip the junction `unique` index: **`writeOpportunityLinks(tx, opportunityId, ids)`** inserts all four people-junction rows at once (used by `createOpportunity`, and by `updateOpportunity` after it deletes the existing rows); and the per-junction **`replaceOpportunity{Contacts,Owners,SourceStaff,SourceContacts}(tx, id, ids)`** (delete + re-insert one junction), used by the field-scoped `updateOpportunityField` so each people-edit rewrites only its own junction.
  - `createOpportunity.schema.ts` — the shared zod schema. Its field shape lives in the exported **`opportunityBaseFields`** block, which **both** create and update build a `z.object` from (update just adds `id`), so the two can't drift; **required `companyId` sits in that shared base** (it moved in when company became drawer-editable — create and update now enforce the same "Company is required." rule). Imports the enum tuples **directly** from `@/lib/opportunity` (the old back-compat re-export shim is gone). A `superRefine` (`refineReferral`, also shared) enforces the conditional referral rules: `source = staff_referral` ⇒ ≥1 `sourceStaffIds`; `source = contact_referral` ⇒ ≥1 `sourceContactIds`.
  - `createContact.ts` / `updateContact.ts` (+ `.schema.ts`) — create and edit a contact, both gated `crm.edit`. `createContact` is a **single `db.insert(contacts)`** (no transaction) and **returns `{ id }`** (not `{ ok: true }`) so an inline-created contact can be immediately selected in the opportunity form; `updateContact` is a `db.update(...).returning()` by `id` (an empty `returning` → user-safe "That contact no longer exists"). **Two checks are shared between them via `contactChecks.ts`** (so create/update can't drift): `assertValidManager({ managerId, companyId, selfId? })` — a manager must be an existing contact at the *same* company (and, on update, never the contact itself); the UI picker enforces the same rule, but this re-check (a DB lookup) stops a hand-crafted request creating a cross-company, self-referential, or dangling management link ([ADR 0022](../decisions/0022-contact-manager-self-reference.md)) — and `mapContactEmailConflict(error)` — maps the Postgres unique-violation on `contacts_email_unique` to a user-safe "A contact with that email already exists." (returns `never`, used as the whole `catch` body), via the shared **`isUniqueViolation(error, constraint)`** helper in `src/lib/db/unique-violation.ts` (a reusable SQLSTATE-`23505`-on-named-constraint check, extracted from the old inline check in `createContact` so the CRM write actions share it). Both actions accept and write the optional **`ownerId`** (the owning staff member): `createContact` sets it on create, and `updateContact` — as a **full-record** update — round-trips it. The edit dialog no longer renders an owner picker, but keeps `ownerId` in its `defaultValues` (with no rendered field) so this full-record save re-sends the current owner **unchanged** (RHF v7 includes unregistered defaults in the submit payload); a dialog save must not wipe the owner. Reassigning the owner now goes through the dedicated **`updateContactOwner`** action (below) via the inline field. Creating a company inline from the contact form is **persist-first** — the new company is saved via `createCompany` first (through `CompanyComboboxField`'s inline dialog), then its id is passed here as `companyId`. This mirrors opportunities and replaced an earlier design where `createContact` created the company itself in a nested `db.transaction` (that `newCompany` path and the transaction are gone). `createContact` `revalidatePath("/contacts")`; `updateContact` revalidates `/contacts` and `/contacts/${id}`. Input schemas live in sibling `.schema.ts` files.
  - `createCompany.ts` / `updateCompany.ts` (+ `.schema.ts`) — create and edit a company (name/websiteUrl/`isPartner`/`ownerId`), both gated `crm.edit`. `updateCompany` is a `db.update(...).returning()` by `id` (empty → "That company no longer exists"), revalidating `/companies` and `/companies/${id}`. Like the contact dialog, the company edit dialog no longer renders an owner picker but keeps `ownerId` in its `defaultValues`, so this full-record save round-trips the current owner unchanged; owner reassignment goes through `updateCompanyOwner` (below).
  - `updateCompanyOwner.ts` / `updateContactOwner.ts` (+ `.schema.ts`) — **owner-only** updates, both gated `crm.edit`, backing the inline owner field on each detail page. Each writes **just `ownerId`** (a staff id, or `null` to unassign — schema is `createUpdateSchema(Table).pick({ ownerId }).extend({ id, ownerId: id.nullable().default(null) })`), then `revalidatePath`s the list + detail routes; a `.returning()` empty set → user-safe "That … no longer exists." Deliberately **narrow counterparts** to the full-record `updateCompany`/`updateContact`: writing only `ownerId` means an owner change can't clobber a field being edited concurrently elsewhere, and — for contact — it **skips `assertValidManager`** (a full-record save would wrongly re-run the manager rule on an owner-only change). Precedent: `updateOpportunityPosition` (a narrow single-column write). See the inline owner field in the UI section.
  - **Entry actions** (`addContactEntry` / `updateContactEntry` / `deleteContactEntry` + the three `*OpportunityEntry` counterparts) — the notes & next-steps CRUD, **all gated `crm.edit`, no per-entry ownership** (any editor may amend any entry, by product decision — [ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md)). `add*` resolves the author server-side via **`resolveAuthorStaffId.ts`** (wraps `getCurrentStaffAccess`; null when the user has no staff record) and inserts with a `centry`/`oentry` id; `update*` writes only `body` (`.returning()`-guarded so a mid-edit delete surfaces cleanly); `delete*` removes the row. Each revalidates the parent's list + detail path. They share **`entries.schema.ts`** (the `crm_entry_kind` enum, per-kind length caps via `requiredText` + a `refineEntryBody` superRefine) and the read helper **`entryViews.ts`** (`getContactEntries`/`getOpportunityEntries` → newest-first `EntryView`s with author name resolved and timestamps as epoch millis; `editedAt` set only when `updatedAt > createdAt`). See [Timestamped entries](#timestamped-entries-notes--next-steps).
- **UI** — three CRM pages: `/companies` ("Companies", `src/app/(app)/companies/page.tsx`) and `/contacts` ("Contacts", `src/app/(app)/contacts/page.tsx`), each its own server-side-paginated table, plus `/opportunities` (`src/app/(app)/opportunities/page.tsx`, a **dnd-kit kanban board** — `opportunity-board.tsx` + `opportunity-board-column.tsx` + `opportunity-card.tsx`; the old `opportunities-table.tsx` was removed). Clicking a card opens `opportunity-detail-sheet.tsx` (a **wide** `max-w-3xl` right-side `Sheet`) that lazy-loads via `loadOpportunityDetail` and re-fetches after every save. It reads as a **detail view with three tabs**:
    - **Info** — each field (Name, Company, Line of business, Owners, Contacts, Source + its referral picker, Status) renders **read-only with a per-field pencil**; clicking Edit swaps that one field into an inline control with **Confirm/Cancel**, and Confirm saves **just that field** via **`updateOpportunityField`** (a discriminated union on `field` — the action writes only the changed slice, not the whole record). **Source + referral edit together as one unit** (the referral picker depends on the source). The client mirror of the delivery-stage guard lives in the Status field. Built on the shared **`InlineEditField`** primitive (see [../ui.md](../ui.md)). The **Company** field (`CompanyField`, sits directly above Contacts) reuses `CompanyCombobox`; clearing it fails the schema's required rule inline. Because company is now an editable Info field, it was **dropped from the drawer header** — the `SheetDescription` no longer shows the company name and is now a visually-hidden generic string (`"Opportunity details for {name}"`) purely satisfying the dialog's description/a11y slot.
    - **Notes** — the opportunity's two entry logs (Next steps, then Notes), each an `EntryLog` (`entry-log.tsx`, `canEdit` since the drawer is edit-only) whose mutations call `onChanged={refresh}` to re-fetch the drawer. Replaced the old scalar `NextStepsField` (deleted). See [Timestamped entries](#timestamped-entries-notes--next-steps).
    - **Project plan** — the single linked project: a "Create project" button (hidden once a project exists — at most one per opportunity), the linked-project display, and `AddProjectDialog` (prefilled + company-locked). See [projects.md](./projects.md).
    The drawer only **mounts for `crm.edit`** users (gated on the board), so editing is always allowed once open. **Click-vs-drag** is disambiguated by dnd-kit's 5px pointer activation constraint plus a `suppressClickRef` the board sets after a real drag (so ending a drag on a card doesn't also open the drawer).
  - **Detail pages**: `/companies/[id]` (`company-detail-view.tsx`) and `/contacts/[id]` (`contact-detail-view.tsx`), each a Server Component that `await`s its `get*Detail` read and `notFound()`s on a miss (with a `generateMetadata` titling the tab). Each now carries an **Edit affordance** (for `crm.edit` holders) opening `edit-company-dialog.tsx` / `edit-contact-dialog.tsx`, wired to `updateCompany` / `updateContact` — covering the **core fields only** (the contact dialog also carries the manager picker via `ManagerComboboxField`). The optional **owner** is **edited in place**, not in that dialog, via the shared **`InlineOwnerField`** (`inline-owner-field.tsx`) — placed on both detail pages as its own hairline-divided section at the bottom of the meta sidebar (below the scalar fields). It reads as text (owner name, or "Unassigned") until the pencil is clicked, then swaps in a single-select staff picker (`EntityCombobox` + the `crm.edit`-gated `searchStaff`, **active** staff only) with Confirm/Cancel, and confirming calls the entity's owner-only action (`updateCompanyOwner`/`updateContactOwner`), whose `revalidatePath` refreshes the server-rendered display. This is the single-select counterpart of the opportunity drawer's (multi) owners field, built on the same `InlineEditField` primitive (see [../ui.md](../ui.md)). *(The old dialog-based `OwnerComboboxField` / `owner-combobox-field.tsx` was deleted once both edit dialogs dropped their owner picker.)* The owner name shown on each read view comes from a `leftJoin` to `staff` in `getCompanyDetail`/`getContactDetail`.

    Nuance: `InlineOwnerField` calls **both** `useAction(updateCompanyOwner)` and `useAction(updateContactOwner)` unconditionally (picking the `active` one by `kind`) and branches its `execute` in a `kind` ternary — a single ternary-derived `execute` can't be typed against either action's input, so both hooks are instantiated and the concrete one is called. **Both pages share one two-column layout skeleton** (`detail-parts.tsx`), so they read as visually consistent siblings: a fixed-width meta sidebar (`md:w-64`) beside a `max-w-5xl` main column, collapsing to a single stacked column (sidebar first) on narrow screens. The sidebar holds identity (icon/avatar + name + Edit-dialog button) at top, then the scalar meta fields, then the owner as its own divided section; the main column is a stack of titled sections (**no tabs**). Company main column = **Opportunities / Projects / Referred opportunities / Referred projects / Contacts** sections, each a table. The first four are **omitted entirely when their collection is empty** (gated on `length > 0`, no empty-state row); **Contacts is the only always-present section** — it links each person to their contact page (it *is* the company's people directory), keeps its "No contacts at this company yet." empty state, and its columns are **Name · Role · Email · Phone · Next steps** (the last renders each contact's latest next step — line-clamped body + `formatShortDate` timestamp, `EmptyCell` when none — the only place a company view surfaces contact-level next steps; the companies *list* table deliberately does not, see below). The two referral sections are both **referral, not ownership** — the referred entity's client company is often a *different* company than the one viewed — and share a local `ReferrerLinks` component for the referrer cell (each referrer an `InternalLink` to their contact page, comma-separated when several). **Referred opportunities** lists opportunities that contacts here brought us: Opportunity, Client (`InternalLink` → `/companies/[clientCompanyId]`), Stage (`OpportunityStatusBadge` — the pipeline **group** label), Referred by. **Referred projects** lists projects they brought us: Project (plain text — projects have no detail page), Client (`InternalLink`), Referred by (no Stage column — the collection carries no status). Contact sidebar carries the contact methods (email/phone/LinkedIn/company/manager, all optional; company & manager link out); its main column leads with **Next steps** and **Notes** `DetailSection`s (each an `EntryLog`, `canEdit` = `crm.edit`), then Opportunities (two labelled subsections "Referred by this contact" / "Also involved in") + Projects (the derived referred-projects). Cross-links weave the two pages together (company ↔ contacts, opportunity/project rows → their company). Both share `opportunity-status-badge.tsx` (renders a status as its pipeline **group** label). See `src/components/crm/**` and [../ui.md](../ui.md).
  - **List tables now link into the detail pages:** `companies-table.tsx` links the name cell to `/companies/[id]`; `contacts-table.tsx` links the name cell to `/contacts/[id]` and its company cell to `/companies/[id]`. The contacts table was **slimmed to Name · Company · Role · Next steps** — the last column renders the latest next-step body (line-clamped) with its `formatShortDate` timestamp; Email/Phone/Manager/LinkedIn were dropped from the table (still on the detail page). The **companies list table deliberately does NOT show next steps** — a company has no entries of its own, and rolling them up from its contacts/opportunities was intentionally declined; contact-level next steps surface only on the company *detail* page's Contacts section. See [ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md).
  - **`entry-log.tsx`** (`EntryLog`) — the shared client component behind every notes / next-steps log (contact detail, opportunity drawer). One `kind` per instance; a `variant` (`contact` | `opportunity`) picks the action set. Renders a composer (for `canEdit`) plus a newest-first list with inline edit/delete on each entry, author + `formatDateTime` timestamp, and an "edited" marker. See [Timestamped entries](#timestamped-entries-notes--next-steps).

## Timestamped entries (notes & next steps)

Both **contacts and opportunities** carry two running logs of authored, timestamped free-text entries — **notes** (longer prose, "what happened", ≤5000 chars) and **next steps** (short reminders, "what's planned", ≤500). Full rationale in [ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md); the essentials:

- **Two concrete tables, not a polymorphic one** — `contact_entries` (`crm-schema.ts`) and `opportunity_entries` (`opportunities-schema.ts`), each a real FK to its parent (`onDelete: cascade`). Same shape: `id`, `<parent>Id`, `kind`, `body`, `authorStaffId` (FK → `staff`, `onDelete: set null` — attribution, not ownership), `createdAt`/`updatedAt`, index on `(<parent>Id, kind, createdAt)`. Row types `ContactEntry` / `OpportunityEntry`.
- **One shared pg enum `crm_entry_kind` = `('note', 'next_step')`** distinguishes the kinds within a table (declared in `crm-schema.ts`, imported by `opportunities-schema.ts`). Notes and next steps differ **only by `kind` + validation length**.
- **No per-entry ownership** — any `crm.edit` holder may add/edit/delete any entry (product decision: shared team logs). All six actions gate solely on `crm.edit`. The author is resolved server-side (`resolveAuthorStaffId.ts`), never trusted from the client.
- **The scalar `opportunities.nextSteps` column is gone** — migrated into `opportunity_entries` as the first `next_step` entry, then dropped, and removed from the create/update schemas, actions, `updateOpportunityField` union, and the add-opportunity form.
- **The latest next step surfaces in list + detail views** — `getContactsPage` (`ContactRow`), `getOpportunitiesBoard` (`OpportunityBoardCard`), and `getCompanyDetail` (each `CompanyContact`) each left-join a `DISTINCT ON (parentId) … WHERE kind = 'next_step'` subquery, so the newest next step rides along in one query (no N+1) and drives the contacts-table "Next steps" column, the kanban card's "Next:" line, and the company detail page's Contacts "Next steps" column. The **companies list** table stays out of this on purpose — a company owns no entries, and rolling up contact/opportunity next steps was declined.
- **Shared plumbing** — schema `entries.schema.ts`, read helper `entryViews.ts` (`EntryView`, epoch-millis timestamps, `editedAt` only when edited), UI `entry-log.tsx` (one component, `variant`-switched). Date formatters `formatShortDate` / `formatDateTime` live in `src/lib/format.ts`.

## Authorization

**Reads are open** — any signed-in user can browse all companies, contacts, and opportunities (the `(app)` gate is the boundary). **All CRM writes** are gated by a single flat capability (no ownership dimension): **`crm.edit`** (renamed from `contacts.edit` when opportunities landed), granted to `sales`, `manager`, `admin`. It covers creating/editing companies, contacts, *and* opportunities — including the inline company/contact creation, the opportunity detail drawer's edit/move, `loadOpportunityDetail` (edit-only read), the six notes/next-steps entry actions (add/update/delete on both parents, **no per-entry ownership**), and every type-ahead picker (`searchCompanies`/`searchContacts`/`searchStaff`) that backs entry. Declared in action metadata; each page hides its "Add" dialog(s) (via one `canEdit` flag) for users without the capability, the opportunities board renders **static (no drag-and-drop) with no clickable drawer** for non-editors.

**Cross-domain RBAC nuance (delivery-stage handoff).** Creating a project needs **`projects.edit`**, which `sales` does *not* have (only `crm.edit`). The board is passed a separate `canCreateProject` flag (`projects.edit`): a user who *can* create projects gets the prefilled create-project prompt when they move a card into a delivery stage without one; a user who can't (e.g. `sales`) gets a **toast telling them a delivery manager must create the project first**. No permission was widened — the division of labour is deliberate ([ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)). See [domains/permissions.md](./permissions.md).

## Key flows

- **Pipeline management** (built) — create Opportunities and move them through the pipeline (`maturing` → `lead` → … → `closed_won`/`closed_lost`) by **dragging cards** across the kanban board (`updateOpportunityPosition`). Multi-substatus groups (Scoping/Allocating/Closing) collapse into one column or expand into subcolumns; dropping onto a collapsed group picks the first substatus (if dragged from earlier in the pipeline) or the last (from later). Non-editors (`!crm.edit`) get a static, read-only board.
- **Edit a company / contact** (built) — from the detail page (`/companies/[id]`, `/contacts/[id]`), a `crm.edit` holder opens an edit dialog covering the core fields and — for contacts — company and manager. Saved via `updateCompany` / `updateContact` (a single `db.update(...).returning()` by id; an empty `returning` → user-safe "That … no longer exists"). `updateContact` re-runs the same-company **manager check** (`assertValidManager`, now guarding against self-management too) and the **email-uniqueness** map that `createContact` uses. Both revalidate the list path *and* the detail path.
- **Reassign a company / contact owner** (built) — the optional **staff owner** is edited **in place** on the detail page (`InlineOwnerField`), separate from the edit dialog — matching the opportunity drawer's per-field pattern. Confirming saves through the dedicated **owner-only** action (`updateCompanyOwner` / `updateContactOwner`), which writes just `ownerId` (or `null` to unassign) so it neither clobbers concurrently-edited fields nor re-runs the contact manager rule; both revalidate the list + detail paths. The edit dialogs still carry `ownerId` in their form defaults so a full-record save round-trips the current owner unchanged.
- **Edit an opportunity** (built) — clicking a card opens the **detail drawer** (`opportunity-detail-sheet.tsx`), which lazy-loads via `loadOpportunityDetail`. Its **Info** tab edits company, name, status, line of business, source, owners, contacts, and referral sources **one field at a time in place** (pencil → Confirm/Cancel); each Confirm calls the field-scoped `updateOpportunityField`, which writes only that field's slice. This is CRM's first field-edit flow beyond the drag-move, and the first use of the shared `InlineEditField` primitive. Notes & next steps live on their own **Notes** tab (see next flow), not as Info fields.
- **Log notes & next steps** (built) — on a contact detail page or the opportunity drawer's **Notes** tab, a `crm.edit` holder appends timestamped **notes** ("what happened", ≤5000 chars) and **next steps** ("what's planned", ≤500), each shown newest-first with author + timestamp and inline edit/delete. **Any editor may amend any entry** (no per-entry ownership). The author is stamped server-side from the session. The **most recent next step** surfaces on the contacts list table and on kanban cards. This replaced the opportunity's old scalar `nextSteps` field, whose value was migrated in as the first next-step entry. See [ADR 0030](../decisions/0030-crm-timestamped-entries-notes-next-steps.md).
- **Opportunity → Project handoff + delivery-stage requirement** (built) — **transitioning** an opportunity into the **Allocating** group or later (`requiresProject`, Closed – Lost excepted) requires a linked project; creating one directly in a delivery stage is forbidden. Dragging a card into a delivery stage without a project is **blocked** (only on a genuine stage change — an intra-column reorder never prompts); a `projects.edit` holder is shown the create-project dialog **prefilled and company-locked** for that opportunity, and on creation the pending move completes; a user without `projects.edit` (e.g. `sales`) gets a toast pointing to a delivery manager. Enforced server-side at the entry points (creation ban + transition guard in both status-mutating actions). See [projects.md](./projects.md), [ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md), and [flows.md](../flows.md).

## Connects to

- **Staff** — an opportunity's `owners` and referral `source_staff` are `staff` rows (FK, cascade). This is the first CRM ↔ staff link. A **company and a contact also each carry an optional `ownerId` → `staff`** (`onDelete: set null`) — the accountable owner. Deliberately **set-null, not cascade** (unlike the opportunity junctions): an owner is a soft pointer, so removing a staff member should only unassign them, never delete the company/contact.
- **Projects** — a project belongs to a `companies` row (required FK, `restrict`) and may optionally link back to an `opportunities` row (`projects.opportunityId`, nullable FK, `restrict`, **at most one project per opportunity** via a partial unique index). The Opportunity → Project handoff now **populates** that column, and `requiresProject` makes a project a precondition for advancing an opportunity to a delivery stage. See [projects.md](./projects.md), [ADR 0019](../decisions/0019-project-opportunity-link.md), [ADR 0024](../decisions/0024-opportunity-project-handoff-and-placeholder-roles.md).
- **Allocations / Timesheets** via the Project a won Opportunity will create (proposed).
- See [../data-model.md](../data-model.md) and [../flows.md](../flows.md).

## Open questions

- Probability/weighting per status and pipeline revenue forecasting (value × probability) — not modelled; opportunities carry no monetary value field yet.
- **Edit/delete** — **companies, contacts, and opportunities now all have edit** (companies/contacts via a whole-dialog `updateCompany`/`updateContact`; the opportunity detail drawer via per-field `updateOpportunityField` + drag-move), so a contact's `companyId`/`linkedinUrl`/`managerId`/`ownerId` (and a company's owner) can now be changed after creation (the old create-only limitation is gone). **No *delete* for any CRM entity yet.** Note the `onDelete: restrict` on `opportunities.companyId` means a company-delete flow must handle live opportunities, and the `restrict` on `projects.opportunityId` means an **opportunity**-delete flow must handle the (at most one) live project linked to it.
