# 0020 — Compensation as effective-dated facts on `staff_employment`; import-only, required, view-gated

**Status:** accepted · 2026-07-09

## Context

The system needs to hold each person's compensation (base salary, hourly rate, bonuses, currency). Two prior decisions frame this:

- [ADR 0007](./0007-staff-employment-effective-dating.md) split staff into durable identity (`staff`) + **effective-dated employment facts** (`staff_employment`, one row per change, latest `effectiveFromDate` = current), and explicitly said "rates/cost are expected to follow" the same pattern.
- Compensation is the most **sensitive** data in the domain — `data-model.md` and `architecture.md` both flagged that rates/salaries need strict role-based access.

Two shaping constraints came from *how comp is sourced*: it arrives only via the Rippling CSV import (there is no HR data-entry UI here), and compensation is **required data** — every active staff member has pay on file — so a row that can't supply it isn't a valid import, rather than something to fill in later.

## Decision

**Store compensation as five NOT NULL columns on `staff_employment`, not a new table** — `base`, `hourlyRate`, `guaranteedBonus`, `discretionaryBonus` (`numeric(12,2)`, `mode: "number"`; `discretionaryBonus` defaults to `0`), and `currency` (new `currencyEnum`, values from the pure `src/lib/currency.ts` module). It is thus effective-dated for free: a comp change spawns a new employment row, exactly like a role change. (Added nullable in `drizzle/0017_gorgeous_celestials.sql`; made NOT NULL in `drizzle/0018_dry_captain_cross.sql`, which first backfills any pre-comp legacy rows to `0` / `'CAD'` placeholders so the constraint can be added.)

- **Import-only.** Compensation has **no in-app edit UI** — not on the profile, not in the bulk-edit-roles tool. The Rippling importer is the only writer. Column map: "Annual base remuneration"→`base`, "Hourly Rate"→`hourlyRate`, "Target annual bonus"→`guaranteedBonus`, "Compensation currency"→`currency`. `discretionaryBonus` is **not imported** — it always defaults to `0`.
- **Required on import — skip the row if missing.** Comp is not optional: `normalizedStaffSchema` (`staff-import/types.ts`) makes all five fields non-nullable, and `transform.ts` **skips** any row missing `base`, `hourlyRate`, or `currency` (surfaced under "Missing/invalid compensation: …"). `parseNumber` (`csv-import/parse.ts`) still tolerates currency symbols / thousands separators and yields `null` on a blank/unparseable/negative cell — but a `null` value for one of those three now **fails the row** rather than being tolerated. **`guaranteedBonus` is the exception — it's optional:** a blank/whitespace-only "Target annual bonus" cell defaults to `0` and the row is kept (a *present-but-unparseable* value still fails), since many people simply have no guaranteed bonus and shouldn't be dropped from the import over it. **Consequence:** because *both* `base` and `hourlyRate` are required, a person the export gives only one of (e.g. a salaried employee with no hourly rate) is skipped until the CSV carries both. There is **no carry-forward on the import path** — comp flows straight from `incoming` for every row (`plan.ts` no longer overlays it from the current record). **Bulk-edit still carries comp forward** when it inserts a new effective-dated row (comp isn't one of its editable `FACT_FIELDS`), so advancing a role/LoB never clears salary.
- **View-gated via `staff.viewCompensation`; own comp always visible.** New capability granted to `finance` (previously an empty role — this is its first perm), `manager`, and `admin`. Decision helper `canViewCompensation` (`src/actions/staff/canViewCompensation.ts`) mirrors `canEditStaff`: own → always; others → require the permission. It gates both the profile Compensation card (a UI prop) **and** the comp amounts folded into the history feed's employment entries — the latter at the *read* (`getStaffHistory(staffId, includeCompensation)`), because history renders in a client component, so the amounts never reach an unauthorized browser.
- **Shown on profile + history.** A Compensation card (`compensation-section.tsx`) on `ProfileView`, gated by the `canViewCompensation` prop (base / hourly rate / currency always; the guaranteed & discretionary bonus rows only when non-zero; currency shown as its bare code, e.g. `CAD`); and, for authorized viewers, the comp amounts **appended to each employment entry's summary line** in the history feed (e.g. "Engineer · Core · Full time · Billable · Base CA$150,000.00 · Hourly CA$0.00 · Guaranteed bonus CA$20,000.00"; zero bonuses omitted). Compensation is **not** a separate history category — `HistoryCategory` stays `"EMPLOYMENT" | "ALLOCATION"`; comp rides the employment entry it belongs to rather than lighting up the stub category [ADR 0011](./0011-category-agnostic-history-feed.md) reserved.

## Consequences

- **No new table, no new join.** Current comp is already on the latest employment row every reader fetches; history is already the employment row series. Adding comp to a read is adding columns to the existing projection.
- **A comp-only change still writes a full new employment row** (all facts carried forward) — consistent with ADR 0007's history-as-rows model; there is no separate comp-history granularity.
- **Comp can only be corrected by re-importing.** With no edit UI, a wrong figure is fixed by another CSV sync (or a manual DB/`db:studio` write). Bulk-edit deliberately can't touch it, so an admin editing roles can never accidentally alter salary.
- **Sensitivity is enforced at the read boundary**, not just the UI — the history gate is the load-bearing case (client component). Any future comp surface must pass `canViewCompensation` before comp leaves the server.
- **`numeric` with `mode: "number"`** returns JS numbers (fine at these magnitudes); revisit if precise decimal arithmetic on money is ever needed app-side.

## Alternatives considered

- **A separate `staff_compensation` effective-dated table** (the shape `data-model.md` originally floated) — rejected for now: comp changes and employment changes overwhelmingly coincide (both come from the same Rippling sync on the same effective date), so a second effective-dated table would duplicate the row-per-change machinery and force a join for every profile read, with no current query that needs comp history at a different cadence than employment. Promote to its own table only if comp needs to vary independently of employment facts or carry its own attributes (e.g. approval, source).
- **An in-app comp editor (bulk-edit or a profile dialog)** — rejected: Rippling is the system of record for pay; a second write path invites drift and widens the sensitive-write surface. Import-only keeps one source.
- **Nullable comp with carry-forward-on-blank import semantics** — the original decision, now **reversed**. Comp started nullable and carried forward on a blank cell (so a partial export wouldn't zero out salaries). We now treat comp as **required**: the columns are NOT NULL and the importer skips a row it can't supply comp for. Rationale: every active staff member does have pay on file, "unknown salary" isn't a real state, and required-on-import keeps the model honest (no nullable comp to defend against downstream) at the cost of skipping under-specified rows until the export is complete. Carry-forward survives only where there's genuinely nothing to re-supply — bulk-edit inserting a new row for a *non-comp* change.
- **Filtering comp out of the history feed in the UI** — rejected: history renders in a client component, so the data would still ship to the browser. Gate at the server read instead.
