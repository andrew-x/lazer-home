## Component library

shadcn on **Base UI** primitives (`base-nova` preset, **Tabler icons**, Geist font). Config: `components.json` (`style: "base-nova"`, `baseColor: "neutral"`, RSC on, aliases `@/components`, `@/lib`, `@/components/ui`, `@/hooks`).

- **Icons: `@tabler/icons-react`** (`Icon*` components, e.g. `IconHome`, `IconSettings`). Lucide was **removed** (`lucide-react` uninstalled) — all app + vendored `src/components/ui/**` icons use Tabler, and `nav.ts` types icons via Tabler's `Icon` type. Don't add other icon libraries.
- **Add components, don't hand-write them:** `bunx --bun shadcn@latest add <name>` (e.g. `table`, `dialog`, `tabs`, `badge`). Re-add with `--overwrite` to update. **Gotcha:** the shadcn registry emits Lucide imports — after re-adding any `ui/**` component, re-swap its icons to Tabler.
  - **Exception — `dialog.tsx` and `textarea.tsx` were vendored by hand**, not via the CLI. The `shadcn add dialog` run wanted to **overwrite our hand-edited `button.tsx`** (which carries the deliberate `loading` prop) and pulls Lucide, so `dialog.tsx` was instead written to mirror the existing `sheet.tsx`: a centered modal on `@base-ui/react/dialog` with a Tabler `IconX` close. Treat both as vendored primitives all the same (don't hand-edit beyond the icon swap). The `<Dialog>`/`<DialogTrigger render={…}>`/`DialogContent`/`DialogHeader`/`DialogFooter`/`DialogClose` API matches the sheet's.
- **Vendored primitives** live in `src/components/ui/**` and are treated as generated code: don't hand-edit, and Biome skips linting them via an `overrides` block in `biome.json` (turns off a11y/exhaustive-deps/document-cookie rules that the registry code trips — extended with `useSemanticElements`/`useKeyWithClickEvents` off, still needed by `input-group`). Current set also includes `select`, `switch`, `toggle`/`toggle-group`, `popover`, `calendar`, a composed `date-picker`, `tabs` (originally added for the CRM detail pages; those were **de-tabbed** in the shared-layout refactor and now back the **opportunity detail drawer** — Info / Notes / Project plan, see [domains/crm.md](./domains/crm.md)), and (for the CRM company picker) `combobox` + `input-group` (icons swapped Lucide → Tabler as usual). The `combobox` is built on **Base UI's Combobox** — the right fit for the Base UI preset; it replaced the cmdk-based `command` primitive (now deleted), and `input-group` (formerly a `command` dependency) is now a `combobox` dependency. The **`date-picker`** (`src/components/ui/date-picker.tsx`) is our own composition of the vendored `popover` + `calendar` (built on `react-day-picker`, added for it) over the project's Base UI primitives, with Lucide icons swapped to Tabler; it takes/returns a `"YYYY-MM-DD"` string (timezone-agnostic, per `.claude/rules/database.md`) and formats the trigger label via `formatDate` from `@/lib/format` (**`date-fns` was removed** — no longer a dependency). Used by the bulk-edit save bar. **Exception — `Button`'s `loading` prop:** `button.tsx` carries a deliberate hand-edit. `<Button loading>` renders a leading `IconLoader2` spinner (inherits variant color/size), auto-disables (`disabled || loading`), and sets `aria-busy` — the canonical way to satisfy the forms rule's "drive button loading state from `isPending`/`isExecuting`" (used at the auth buttons, the shared `FormDialogFooter` Save button, and the shared `CsvImport` harness's confirm button). **Gotcha:** re-adding Button via `shadcn --overwrite` drops this prop; re-apply it after any overwrite.
- **Polymorphism uses a `render` prop, NOT Radix's `asChild`.** Base UI is not Radix — your training data is likely wrong here. To render a button as a link: `<Button render={<Link href="/x" />}>` (see `not-found.tsx`).

### EditableTable — don't route draft state through TanStack `table.options.meta` (gotcha)

The shared `EditableTable` (`src/components/admin/editable-table.tsx`) hands each cell editor the per-row **draft** (read the current value, push a patch) via **React Context** — the `EditableDraftContext.Provider` wrapping the render, read with the **`useEditableDraft<TValues>()`** hook. This is deliberate and load-bearing; the obvious approach — passing the draft through TanStack's `table.options.meta` and reading it off `cell.getContext()` — is **broken** and caused a real, hard-to-see bug.

**Why `meta` fails.** TanStack memoizes `cell.getContext()` on `[table, column, row, cell]` — all referentially stable while the row's `data` is unchanged. `flexRender` passes that memoized context object straight into `React.createElement(cellComponent, context)`. The draft lives **outside** the table's row `data` (it's separate draft state in `useEditableRows`), so when only the draft changes, the cell element's props are referentially identical to the previous render and **React bails out of re-rendering the cell entirely** — the editor keeps showing the pre-change value. The floating save bar still updated (it's rendered directly by `EditableTable`, not through a table cell), which made it look like state was flowing when the cell alone was frozen. Confirmed at runtime: `onValueChange` fired with the new value and the draft updated (save bar appeared), yet the cell's own value read stayed stale.

**The fix (in code):** deliver the draft through context, not `meta`. A **context read re-renders its consumers even when a parent bails out**, so cells re-render on every draft change. `useEditableDraft()` is the API cells use (the old `editDraft(table)` helper and its `declare module "@tanstack/react-table"` augmentation were removed); cells no longer take a `table` prop. Applied across all three editable tables: `performance/edit-levels.tsx` (`LevelCell`), `admin/bulk-edit-roles.tsx` (`EnumCell`/`SwitchCell`/`UtilizationCell`/`BillableTypeCell`), `admin/manage-users.tsx` (`RoleCell`/`BannedCell`).

**General rule:** never route frequently-changing external state (state that lives outside the table's row `data`) through `table.options.meta` — the memoized `getContext` lets React skip the cell re-render. Use context (a subscribing hook like `useEditableDraft`) instead. (Incidental: these cells render the trigger label directly from the draft value, `<SelectValue>{label(value)}</SelectValue>`, which is a fine simplification — but that was **not** the fix; it doesn't matter if the cell never re-renders.)

## Theming & tokens

- Tokens are oklch CSS variables in `src/app/globals.css` (`:root`), surfaced to Tailwind v4 via `@theme inline`. Imports: `tailwindcss`, `tw-animate-css`, `shadcn/tailwind.css`.
- `--font-heading` maps to Geist sans.
- **Light mode only.** A `.dark` block and the `dark` custom variant exist (carried in from the registry) but are **unused** — there is no theme toggle. Don't wire one up without a decision.
- Use **semantic classes** (`bg-primary`, `text-muted-foreground`, `bg-sidebar-accent`, `border`), never raw hex or `indigo-600`. Conditional classes via `cn()` from `@/lib/utils`.

### Design language (current source of truth)

A deliberate, editorial look that **avoids the generic AI-app aesthetic** (rounded + shadowed + accent-heavy). This refines the theme described in [decisions/0005](./decisions/0005-ui-stack.md) — that ADR still holds for the stack and the indigo-light-mode choice, but the specifics below supersede its "indigo accent" framing.

- **Sharp corners.** `--radius` is `0.25rem` (4px); all the `--radius-*` steps derive from it. Don't bump radii per-component.
- **Flat surfaces.** No elevation shadows — a utility-layer rule in `globals.css` neutralizes every `shadow-*` to `box-shadow: none`. Hairline `border`s define edges; use `border` for separation, never a shadow.
- **Indigo, sparingly.** The UI is mostly **monochrome neutral grays**. The `--accent`/`--accent-foreground` and `--sidebar-accent`/`--sidebar-accent-foreground` tokens are neutral (not indigo). Indigo lives only in `--primary` and `--ring`, so it surfaces only on primary buttons, focus rings, links, and the active nav item's icon (`app-sidebar.tsx` adds `text-primary` to the active icon). Don't tint hovers/cards/backgrounds indigo — the accent reads as intentional precisely because it's rare.
- **Global `cursor-pointer`.** A base-layer rule gives `button`s and ARIA-interactive roles (`menuitem`, `option`, `switch`, `tab`, `checkbox`, `radio`, …) a pointer cursor — Base UI / shadcn omit it by default. Don't add `cursor-pointer` per component.
- **Minimal auth surfaces.** `(auth)/login/page.tsx` is a single centered column — logo mark + product name + one Google button + "Lazer staff only" — no card, no gradient. Keep auth/marketing screens uncluttered.

### Brand (intentionally off-accent)

The brand mark is `public/icon.svg` — **magenta** (`#FF00C5`) — exposed via `<LogoMark>` / `<LogoWordmark>` in `src/components/brand/logo.tsx`. The product name is `APP_NAME` ("Lazer Home") in `src/lib/constants.ts`. The magenta mark against the indigo UI accent is **intentional** — the logo is the brand, the indigo is the product chrome; don't "fix" the mismatch.

### Charts (hand-rolled SVG — no charting library)

Data viz is **hand-rolled inline `<svg>`**; **do not add a charting dependency**
(no Recharts/visx/Chart.js/etc.). This is deliberate — a library's default look
(rounded, shadowed, multi-color, boxed legends) fights the flat/editorial design
language, and our charts are simple enough to draw directly. Two instances live on
`/performance`: the distribution **scatter** (`compensation-scatter.tsx`, in the
compensation section) and the level-distribution **bar chart**
(`level-distribution-bar-chart.tsx`, in the staff-levels section — both on the one
dashboard). **Follow the same approach for future charts** rather than reaching for
a library.

Dataviz styling rules (consistent with the design language above):

- **Thin marks, recessive gridlines.** Gridlines/axis baseline are hairlines
  (`stroke-border`, `strokeWidth={1}`); the mark is small (`r={3.5}`).
- **Indigo is the data-ink.** A single data series uses the **`--primary`** token
  (`fill-primary`) — the one place data earns the accent. Keep it to one series; a
  single-series chart needs **no legend** (the `<figcaption>` caption names the
  metric).
- **Detail via native `<title>`.** Per-mark hover/label is a `<title>` child of the
  element — no tooltip library, and it doubles as the accessible label (with a
  `role="img"` + `aria-label` on the `<svg>`).
- **Non-zero baseline is OK for scatters.** Dots carry no area-from-zero meaning
  (unlike bars), so padding the domain off zero to keep the spread legible is
  honest. **Bar charts must start at a zero baseline** — bar area encodes the value,
  so a clipped baseline lies (the level-distribution bar chart enforces this).
- **Interaction stays lightweight.** The scatter's per-dot detail is a native
  `<title>`; the bar chart adds a small hover-tooltip bubble (its own SVG `<rect>` +
  `<text>`, clamped inside the plot) — still no tooltip library.
- **Fixed `viewBox`, fluid width.** Draw in a fixed viewBox coordinate space that
  scales to the container (`className="h-auto w-full"`), and use
  `vectorEffect="non-scaling-stroke"` so hairlines stay crisp at any width.

## Routing & auth gating

Route groups split public from authenticated, and **route protection lives in a layout, not middleware** — consistent with the project's deliberate no-middleware stance (see ADR-0006).

- `src/app/(app)/**` — authenticated pages. `(app)/layout.tsx` is an async Server Component that calls `getCurrentUser()` and `redirect("/login")` when there's no session, **then** `getCurrentStaffAccess(user)` and redirects any non-`ok` status to the single `/profile-setup` route (status is `ok` = active staff record with employment — see [flows.md](./flows.md)). Every page under `(app)` is gated by this two-step check. The layout also computes **`visibleNavHrefs`** (see *App shell & sidebar* → permission-aware nav) and threads it into `AppShell`.
- `src/app/(onboarding)/**` — the post-login **block** screen for authenticated users who can't enter the app yet: a single `profile-setup/page.tsx` that resolves `getCurrentStaffAccess` itself and shows different title/body for the `incomplete` vs `not_setup` cases (same screen). There is **no group `layout.tsx`** — the page self-gates like `/login`: it redirects unauthenticated users to `/login` and bounces `ok` users back to `/` so a fixed-up profile is never stuck here. The screen uses `OnboardingNotice` (`src/components/auth/onboarding-notice.tsx`) — a login-styled full-screen notice (logo + title + message + `SignOutButton`).
- `src/app/(auth)/**` — public pages (currently only `/login`, which redirects to `/` if already signed in).
- The old default `src/app/page.tsx` was deleted; the home dashboard is now `(app)/page.tsx` served at `/`.

Pages today: `/login` (landing + Google button), `/` (home dashboard with placeholder stat cards), `/staff` ("Browse staff" — searchable/filterable directory of everyone) and `/staff/[id]` (a person's profile), `/profile` ("My profile" — your own staff profile, same presentation as `/staff/[id]`), `/companies` ("Companies" — the CRM company list, see below) and `/companies/[id]` (a company's detail page), `/contacts` ("Contacts" — the CRM people list, see below) and `/contacts/[id]` (a contact's detail page), `/opportunities` (the CRM pipeline kanban board, see below), `/projects` (client engagements + staffing, see below), `/feedback` (peer feedback — two-tab list + `/feedback/new` give-feedback page + `/feedback/[id]` detail, see [performance.md](./domains/performance.md)), `/performance` (compensation & headcount analytics — `staff.viewCompensation`-gated, `notFound()`s others; see [performance.md](./domains/performance.md)), `/settings` (account info synced from Google + sign out), `/profile-setup` (the single onboarding block screen, two messages), plus the localhost-only `/admin` surface (see below). `NAV_ITEMS` (`src/components/app-shell/nav.ts`, the source of truth) currently has **eleven entries**: Home (`IconHome`), My profile → `/profile` (`IconUser`), Staff (`IconUsers`), Companies → `/companies` (`IconBuildingSkyscraper`), Contacts → `/contacts` (`IconAddressBook`), Opportunities → `/opportunities` (`IconTargetArrow`), Projects → `/projects` (`IconBriefcase`), Timesheets → `/timesheets` (`IconClock`), Performance → `/performance` (`IconChartBar`, **permission-gated** on `staff.viewCompensation`, and a **submenu**: Dashboard → `/performance` + Edit levels → `/performance/levels/edit` gated on `ratings.edit` — see *App shell & sidebar*), Feedback → `/feedback` (`IconMessageHeart`), Settings (`IconSettings`). Admin is **not** in `NAV_ITEMS` but **does** appear in the sidebar footer as a **conditional, localhost-only** entry (`IconTool` → `/admin`): `app-sidebar.tsx` renders it only when its `isLocal` prop is true (see the app-shell section below), so it's visible when developing locally and absent in any non-loopback deployment.

### `/admin` — localhost-only tooling surface

The admin area lives **outside the `(app)` route group on purpose**: the `(app)` layout gates on an active staff record, but the staff-upload tool is what *creates* those records (chicken-and-egg). `src/app/admin/layout.tsx` is the whole gate — it `notFound()`s the entire segment unless `isLocalhost()` (`src/lib/admin.ts`), so there's no RBAC check here; being local *is* the boundary. Its own chrome (header + "Back to app" link) replaces the app shell. `admin/page.tsx` is a card grid linking to the four tools:

- `admin/manage-users` — edit application users' RBAC role + ban status inline, save all at once (`manage-users.tsx`; see [domains/permissions.md](./domains/permissions.md)). `promote-self-button.tsx` is a dev shortcut here.
- `admin/bulk-edit-roles` — edit staff employment facts (role, line of business, billability, management) across the team in one pass (`bulk-edit-roles.tsx`).
- `admin/upload-staff` — import staff from a Rippling CSV export (`staff-import.tsx`).
- `admin/upload-pto` — import leave records from a Rippling CSV export (`pto-import.tsx`).

The two CSV importers share the **`CsvImport`** harness (`csv-import.tsx`) — a file → preview → commit flow that owns the PapaParse config, file/reset lifecycle, badge summary, preview sections and commit orchestration; each page supplies only its columns, labels, transform, and the two next-safe-action hooks (preview + commit). Both importers render read-only previews via `data-table.tsx` (a TanStack React Table wrapper over the vendored `table` primitive). The two inline-edit tools (`bulk-edit-roles`, `manage-users`) instead share `editable-table.tsx` (its own TanStack-based editable grid) and the shared filter controls. Those generic filter controls now live app-wide in **`src/components/form/filters.tsx`** (`ALL` sentinel, `FilterLabel`, `SelectFilter`, `SegmentedFilter`, `TriStateFilter`/`TRISTATE`) so the staff directory and performance dashboard can share them; **`src/components/admin/table-filters.tsx` is now a thin re-export shim** over that module, kept so the admin tables keep their existing import path.

### Profile pages — `ProfileView` (shared by `/profile` and `/staff/[id]`)

`ProfileView` (`src/components/staff/profile-view.tsx`) is the single presentation for a staff profile. **Both** `/profile` (own record) and `/staff/[id]` (anyone) render it; they differ only in which `staffId` they resolve and pass in.

**Two-column layout** (`max-w-6xl`, `lg:grid-cols-[20rem_1fr]`):

- **Left rail** — three stacked cards: a **meta/identity card** (avatar, name, an employment summary line — role · line of business · employment type · billable/non-billable, all via the label maps below — plus Email, Joined date, and a read-only **"Reports to" link** to `/staff/[managerId]` when set); a **Links card** (LinkedIn/GitHub/Portfolio, `CardAction` Edit when `canEdit`); and a **Compensation card**, rendered only when the `canViewCompensation` prop is true.
- **Right column** — a single card wrapping a **Base UI `Tabs` (`variant="line"`)**, default tab **Overview**. Tabs: **Overview**, **Manual of Me**, **Ways of Working**, **Resume**, **History**. The **Overview** tab stacks flat sub-sections separated by hairline rules (`divide-y`) via the local `TabSection` helper (heading + optional action + content): **Skills** (Edit → `/staff/[id]/skills` page when `canEdit`), **Client intro** (`EditClientIntroDialog` when `canEdit`), **Projects** (`StaffProjectsSection`, see below), and **Time off** (only when `pto` is non-null; heading + external Rippling "Manage" link live in the tab wrapper, content is `PtoContent`). The **Manual of Me**, **Ways of Working**, **Resume**, and **History** tabs each hold one `TabSection` — each survey tab shows its read view (`ManualOfMeSection` / `WaysOfWorkingSection`), an "N of M answered" line, and an Edit/Fill-out link (only when `canEdit`) to its dedicated guided editor page. See *Ways of Working* in [domains/staff-profiles.md](./domains/staff-profiles.md).

The pages don't touch `db` — they `await` actions-layer **reads** (server-only async functions, not `'use server'` actions; see [ADR 0010](./decisions/0010-actions-layer-owns-db-access.md)):

- **`/staff/[id]`** (`(app)/staff/[id]/page.tsx`) `Promise.all`s `getStaffProfile(id)` / `getStaffProjects(id)` / `getStaffPto(id)` / `getStaffAvatar(id)` / `getCurrentUser()` / `getManualOfMe(id)` / `getWaysOfWorking(id)`, then resolves `canEditStaff` + `canViewCompensation` and awaits `getStaffHistory(id, canViewComp)` **separately** (its `includeCompensation` arg depends on the comp gate — comp entries must be filtered at the read since history renders in a client component). `notFound()`s on an unknown id, and uses `generateMetadata` to title the tab with the person's name (shares the `React.cache`d `getStaffProfile` query). Profile/projects/history/avatar reads aren't ownership-scoped — the `(app)` gate is the boundary — but `getStaffPto` returns `null` for non-owners without `pto.review` (so `ProfileView` hides the Time off section) and the Compensation card / comp history are gated by `canViewCompensation` (own always; others need `staff.viewCompensation`). See [domains/permissions.md](./domains/permissions.md) and [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md).
- **`/profile`** (`(app)/profile/page.tsx`) is the same, with the id resolved from the session (`getCurrentStaffId()`) and then passed into the very same parameterized `getStaff*` cores — there's **no separate "my" read**. (Own profile always passes `canViewCompensation = true` and `getStaffHistory(staffId, true)`.)

`getStaffProfile` returns the profile fields (incl. `resume`/`resumeUpdatedAt`) plus the **latest** `staffEmployment` row via the [ADR 0007](./decisions/0007-staff-employment-effective-dating.md) ordering. **Staff enums (role, employment type, billable type, PTO type) display via the label maps in `src/lib/staff-enums.ts`** (`ROLE_LABELS` etc.), **not** `humanizeEnum()` — the label maps preserve acronyms like `QA` that `humanizeEnum` would mangle to "Qa". Calendar dates display via the timezone-safe `formatDate()`, instant `timestamp`s (e.g. `resumeUpdatedAt`) via `formatTimestamp(value: Date)`, and avatar fallbacks via `initialsFor()` — all from `src/lib/format.ts` (`initialsFor` was moved here to be shared; see the dates gotcha in [staff-profiles.md](./domains/staff-profiles.md)). (`humanizeEnum` is still fine for enums with no acronyms and no dedicated label map.)

**Edit (Links, Client intro, Résumé).** The **Links card** puts its Edit button in the card header via **`CardAction`** (`src/components/ui/card.tsx` — a header slot that right-aligns into the grid); **Client intro** and **Résumé** live in the right-column tabs, so their Edit buttons sit in the `TabSection` heading's `action` slot instead. Either way the buttons come from `edit-links-dialog.tsx` / `edit-client-intro-dialog.tsx` / `edit-resume-dialog.tsx` — client components taking a **`staffId` prop** and rendering the shared **`FormDialog`** (`src/components/form/form-dialog.tsx`), whose `children` is a render prop handed a `close` callback. `FormDialog` (via its `useRemountOnOpen` hook) **remounts the inner form on each open** — it bumps a `formKey` when opening so the form's `defaultValues` refresh from the latest server data after a save rather than going stale, while keeping the form mounted through the close animation (so the popup doesn't collapse to its header before fading out). `useRemountOnOpen` also takes an optional `controlled` `{ open, onOpenChange }` for dialogs driven by a parent (the CRM inline create-from-form dialogs). The form uses **`useHookFormAction`** (tight binding; see `.claude/rules/forms.md`) wired to the `updateStaff*` action with the imported `.schema.ts` as the `zodResolver`; the shared **`FormDialogFooter`** renders the server-error line + Cancel/Save, driving the Save button's `loading` off `action.isPending` and reading errors off `action.result.serverError`, and the action's `onSuccess` calls `close`. (Forms whose shape diverges from the action input instead use **loose binding** — plain `useForm` + `useAction` with a manual `onSubmit` that transforms the data; both patterns are documented in `.claude/rules/forms.md`.) The actions are gated server-side by the `authorizeStaffEdit` hook via `metadata({ authorize })` — backed by `canEditStaff` (own → always; other → `staff.edit`); the dialog UI is still rendered for everyone, so a non-permitted user clicking Save on someone else's profile gets the server error rather than a hidden button. See [domains/permissions.md](./domains/permissions.md), [ADR 0014](./decisions/0014-rbac-better-auth-access-control.md).

**Skills — edited on a dedicated page, not a dialog.** The Skills sub-section (Overview tab) shows its Edit link (only when `canEdit`) navigating to `/staff/[id]/skills` rather than opening a `FormDialog` — the three-bucket editor (`edit-skills-form.tsx`) is too large for a dialog. See *Skills* in [domains/staff-profiles.md](./domains/staff-profiles.md) and [ADR 0018](./decisions/0018-skills-inline-jsonb-catalogue.md).

**Projects sub-section (Overview tab).** `StaffProjectsSection` (`src/components/staff/staff-projects-section.tsx`) — purely presentational over `getStaffProjects` (see [domains/staff-profiles.md](./domains/staff-profiles.md)) — lists each project the person is staffed on or delivery-manages as a row: name + `companyName · relationship labels` + a `ProjectStatusBadge` (reused from the projects domain). Empty → "No projects yet." **Project names are not links yet** — there's no per-project detail route (only the `/projects` list). Known gap.

The **résumé dialog additionally** has an "Upload PDF" button: it reads the file to base64 (`FileReader`, guarded to ~6 MB), runs the `parseResumePdf` action via a separate `useAction`, and on success `setValue`s the returned text into the textarea (dirty + validated) for review before the user saves via `updateStaffResume`. The résumé card shows the text with `whitespace-pre-wrap` plus an "Updated …" line from `formatTimestamp(resumeUpdatedAt)`. See [ADR 0013](./decisions/0013-resume-pdf-parse-not-store.md).

**Compensation card — read-only, gated.** Rendered only when the `canViewCompensation` prop is true (own profile always; others need `staff.viewCompensation`). `CompensationSection` (`src/components/staff/compensation-section.tsx`) lists base / hourly rate / currency from the latest employment row, with the **guaranteed & discretionary bonus rows shown only when non-zero**; amounts are formatted via `formatMoney` from the pure `src/lib/currency.ts`, and the **currency shows as its bare code** (e.g. `CAD`, not a translated label — there is no currency-label map; the former `CURRENCY_LABELS` was dead and has been removed). There is **no edit affordance** — compensation is import-only (see [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md)).

**History tab.** History is now an **inline tab** (formerly a slide-out sheet — `history-sheet.tsx` was deleted). `HistoryTimeline` (`src/components/staff/history-timeline.tsx`) is a **purely presentational** component taking the `HistoryEntry[]` from `getStaffHistory()` and rendering each entry as a category `Badge` + date + summary in a left-border timeline. Category-agnostic: only `EMPLOYMENT` entries are wired today (allocation is still a stub); comp amounts are **folded into those entries' summaries** rather than forming a separate category (`CATEGORY_LABEL` has no `COMPENSATION` key). It renders inside a **client** component (the profile's tabbed panel), so comp is filtered upstream at the read (`getStaffHistory`'s `includeCompensation`), not here — see the history feed flow in [staff-profiles.md](./domains/staff-profiles.md).

### `/staff` — Browse staff directory

`(app)/staff/page.tsx` (Server Component) `await`s `getStaffDirectory()` once, then renders the `StaffDirectory` client component (`src/components/staff/staff-directory.tsx`) with the entries + filter options. **All search and filtering is client-side over that single server fetch** (no URL params, no per-filter server query): name search, plus line-of-business / role / employment-type filters (options from `staffDirectoryFilterOptions`), a **skills filter** (see below), plus an **"active only" toggle that defaults ON** — inactive staff are fetched but hidden until it's turned off. ### `/companies` + `/contacts` — CRM companies and people (two pages)

- **Manager picker — `manager-combobox-field.tsx` (`ManagerComboboxField`).** A **thin wrapper over the shared `EntityCombobox`** (below): it wraps the base single-select in a `FormField` and passes the form's chosen `companyId` through `searchArgs` to `searchContacts`, which scopes results to *other contacts at the same company*. The caller renders it only once a company is chosen and keys it by `companyId` (remounting on a company switch so no stale colleague lingers — a contact with no company can't have a manager). Clearable, since a manager is optional; reports `{ id, name } | null`. The same-company rule is also re-enforced server-side in `createContact` — see [domains/crm.md](./domains/crm.md), [ADR 0022](./decisions/0022-contact-manager-self-reference.md). (`CompanyCombobox` / `CompanyComboboxField` remain separate controls — see below.)

It reports the chosen `{ id, name } | null` and is clearable, so it serves both the contact form's optional company and the opportunity form's required one (which passes an `error`). (Website is an optional field; the inline dialog has no partner subtitle.)
- **`company-combobox.tsx`** — a debounced, searchable company picker built on the vendored `combobox` (Base UI Combobox). ### `/opportunities` — Opportunities kanban board (CRM)

## App shell & sidebar

`src/components/app-shell/`:

- `app-shell.tsx` — Client Component; wraps `SidebarProvider` and renders the sidebar plus the page content in a padded (`p-4 md:p-6`) `<div>`. **There is no global header bar** — the old `<header>` showing a `titleForPath(pathname)` page title was removed, so `AppShell` no longer reads `usePathname()`. Each page owns its own in-page `<h2>` title (and sets the browser-tab title via `export const metadata`). It sets `defaultOpen={false}` so the sidebar starts **collapsed as a floating icon island**. Takes **`isLocal`** and **`visibleNavHrefs`** props and passes them straight through to `AppSidebar`. (`TooltipProvider` is no longer here — it now lives in the root layout; see below.)
- `app-sidebar.tsx` — the sidebar itself; uses `<Sidebar variant="floating" collapsible="icon">` (floating icon rail), `size-5` icons. The open/close **toggle lives in its footer** — a `SidebarMenuButton` (Tabler `IconLayoutSidebar`) calling `useSidebar().toggleSidebar` (label toggles Expand/Collapse). It renders only the `NAV_ITEMS` whose href is in the **`visibleNavHrefs`** prop (the permission-aware nav filter — see *Add a nav item* → permission-gated). It also takes an **`isLocal`** prop that drives the **localhost-only Admin footer entry** (`IconTool` → `/admin`, rendered only when `isLocal`). The prop originates in the server `(app)/layout.tsx`, which computes `await isLocalhost()` (`src/lib/admin.ts`) and threads it **server-layout → `AppShell` → `AppSidebar`** — the same loopback-host boundary that gates the `/admin` segment itself, so the nav affordance and the segment agree.
- `nav-user.tsx` — the user dropdown in the sidebar footer.
- `nav.ts` — the nav source of truth (no JSX, so it's importable anywhere). Exports `NAV_ITEMS`, the `NavItem`/`NavSubItem` types, and two active-path helpers (`isActivePath` prefix-match for parents, `isActiveSubPath` exact-match for sub-items — below).

Collapsed nav items get tooltips automatically via `SidebarMenuButton`'s `tooltip` prop (set to `item.title`).

**Add a nav item:** edit `NAV_ITEMS` in `src/components/app-shell/nav.ts` (`{ title, href, icon }` with a **Tabler** icon, typed as Tabler's `Icon`). It drives the sidebar entries; `isActivePath()` (also in `nav.ts`) highlights the active one. Nothing else to touch. (There's no longer a `titleForPath` helper — the page-title header was removed.)

**Submenus (`children`).** A `NavItem` may carry an optional **`children?: NavSubItem[]`** — a `{ title, href, permission? }` list with no icon of its own (children inherit the parent's icon context). Each child's `permission` gates it **independently** of the parent. The Performance entry is the first submenu: Dashboard (`/performance`) + Edit levels (`/performance/levels/edit`, gated on `ratings.edit`). Rendering (`NavMenuItem` in `app-sidebar.tsx`) depends on the count of *visible* children:

- **≤1 visible child → a plain link** (the original single-entry behavior). This is the graceful-degradation path: a user who can't see the sensitive sub-page (e.g. finance sees only the Performance dashboard, not Edit levels) gets a normal link to the section, no lone submenu.
- **≥2 visible children, sidebar collapsed (the default icon rail)** → a **flyout `DropdownMenu` anchored to the icon** (`side="right"`), because the indented sub-list is CSS-hidden in the collapsed rail. This is the state users see most, since the sidebar defaults to collapsed.
- **≥2 visible children, sidebar expanded** → an indented **`SidebarMenuSub`** beneath the parent link. The parent link still navigates to its own `href` (its section home); children render below.

Active state: the parent uses **`isActivePath`** (prefix match), the sub-items use **`isActiveSubPath`** (exact match) — so the `/performance` "Dashboard" child does not read as active when you're on its sibling `/performance/levels/edit` (whose path is a prefix-sibling, not a match).

**Permission-gated nav items.** A `NavItem` (or a `NavSubItem`) may carry an optional **`permission?: PermissionCheck`**. The server `(app)/layout.tsx` evaluates each item against the current user via `userHasPermission` and **flat-maps** parents + their permission-visible child hrefs into **`visibleNavHrefs: string[]`** (the hrefs the user may see), threading it **server layout → `AppShell` → `AppSidebar`**. The sidebar renders only items/children whose href is in that set. **Only serializable hrefs cross the server→client boundary** — the icon components stay client-side, so we can't just filter the item objects on the server. This is a **generic, reusable gate** for any future protected page (the Performance item uses `permission: { staff: ["viewCompensation"] }` and its Edit-levels child `permission: { ratings: ["edit"] }`). It's an **affordance only** — hiding the link is not the security boundary; the page itself must still gate (`/performance` `notFound()`s unauthorized users and its read re-checks `requirePermission`, and `/performance/levels/edit` re-checks `ratings.edit`). Mirrors the localhost-only Admin footer entry's server-threaded pattern.

## Icon-only buttons (tooltip convention)

Icon-only controls must use **`IconButton`** (`src/components/icon-button.tsx`): it requires a `label`, which renders a **tooltip** and sets `aria-label` — so every icon button is discoverable and accessible. Don't ship a bare icon-only button. (Use it for standalone icon-only controls.)

`TooltipProvider` lives in the **root layout** (`src/app/layout.tsx`), not `AppShell`, so tooltips work app-wide (auth screens included). `IconButton` must render inside it — the root provider covers everything.

## Error / not-found / loading (Next 16 specifics)

This is a **modified Next.js build** — these conventions differ from public Next. Verify against `node_modules/next/dist/docs/` before changing them.

- `error.tsx` — **Client Component.** The retry prop is **`unstable_retry`** in this build, NOT `reset`. Call `unstable_retry()` to re-render the segment. There's a **root `error.tsx`** and an **`(app)/error.tsx`**: the latter is a segment boundary nested *inside* the authenticated layout, so an error in an authenticated page renders its fallback **with the AppShell (sidebar/nav) still intact** rather than blowing away the whole shell.
