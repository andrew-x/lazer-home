# Frontend / UI

**Status: built.** The authenticated UI shell, auth screens, and error/404/loading conventions exist, plus real domain pages: staff (directory + profiles), CRM (`/companies`, `/contacts`, `/opportunities`), projects (`/projects`), and peer feedback (`/feedback`). The home dashboard is still placeholder stat cards. Path-scoped working rules are in `.claude/rules/ui.md` (loads when you touch `src/components/**` or `src/app/**`) — read it before writing UI code; this doc is the architectural *why*.

## Component library

shadcn on **Base UI** primitives (`base-nova` preset, **Tabler icons**, Geist font). Config: `components.json` (`style: "base-nova"`, `baseColor: "neutral"`, RSC on, aliases `@/components`, `@/lib`, `@/components/ui`, `@/hooks`).

- **Icons: `@tabler/icons-react`** (`Icon*` components, e.g. `IconHome`, `IconSettings`). Lucide was **removed** (`lucide-react` uninstalled) — all app + vendored `src/components/ui/**` icons use Tabler, and `nav.ts` types icons via Tabler's `Icon` type. Don't add other icon libraries.
- **Add components, don't hand-write them:** `bunx --bun shadcn@latest add <name>` (e.g. `table`, `dialog`, `tabs`, `badge`). Re-add with `--overwrite` to update. **Gotcha:** the shadcn registry emits Lucide imports — after re-adding any `ui/**` component, re-swap its icons to Tabler.
  - **Exception — `dialog.tsx` and `textarea.tsx` were vendored by hand**, not via the CLI. The `shadcn add dialog` run wanted to **overwrite our hand-edited `button.tsx`** (which carries the deliberate `loading` prop) and pulls Lucide, so `dialog.tsx` was instead written to mirror the existing `sheet.tsx`: a centered modal on `@base-ui/react/dialog` with a Tabler `IconX` close. Treat both as vendored primitives all the same (don't hand-edit beyond the icon swap). The `<Dialog>`/`<DialogTrigger render={…}>`/`DialogContent`/`DialogHeader`/`DialogFooter`/`DialogClose` API matches the sheet's.
- **Vendored primitives** live in `src/components/ui/**` and are treated as generated code: don't hand-edit, and Biome skips linting them via an `overrides` block in `biome.json` (turns off a11y/exhaustive-deps/document-cookie rules that the registry code trips — extended with `useSemanticElements`/`useKeyWithClickEvents` off, still needed by `input-group`). Current set also includes `select`, `switch`, `toggle`/`toggle-group`, `popover`, `calendar`, a composed `date-picker`, `tabs` (added for the CRM detail pages — the app's **first tabbed UI**, see [domains/crm.md](./domains/crm.md)), and (for the CRM company picker) `combobox` + `input-group` (icons swapped Lucide → Tabler as usual). The `combobox` is built on **Base UI's Combobox** — the right fit for the Base UI preset; it replaced the cmdk-based `command` primitive (now deleted), and `input-group` (formerly a `command` dependency) is now a `combobox` dependency. The **`date-picker`** (`src/components/ui/date-picker.tsx`) is our own composition of the vendored `popover` + `calendar` (built on `react-day-picker`, added for it) over the project's Base UI primitives, with Lucide icons swapped to Tabler; it takes/returns a `"YYYY-MM-DD"` string (timezone-agnostic, per `.claude/rules/database.md`) and formats the trigger label via `formatDate` from `@/lib/format` (**`date-fns` was removed** — no longer a dependency). Used by the bulk-edit save bar. **Exception — `Button`'s `loading` prop:** `button.tsx` carries a deliberate hand-edit. `<Button loading>` renders a leading `IconLoader2` spinner (inherits variant color/size), auto-disables (`disabled || loading`), and sets `aria-busy` — the canonical way to satisfy the forms rule's "drive button loading state from `isPending`/`isExecuting`" (used at the auth buttons, the shared `FormDialogFooter` Save button, and the shared `CsvImport` harness's confirm button). **Gotcha:** re-adding Button via `shadcn --overwrite` drops this prop; re-apply it after any overwrite.
- **Polymorphism uses a `render` prop, NOT Radix's `asChild`.** Base UI is not Radix — your training data is likely wrong here. To render a button as a link: `<Button render={<Link href="/x" />}>` (see `not-found.tsx`).

## Theming & tokens

- Tokens are oklch CSS variables in `src/app/globals.css` (`:root`), surfaced to Tailwind v4 via `@theme inline`. Imports: `tailwindcss`, `tw-animate-css`, `shadcn/tailwind.css`.
- `--font-heading` maps to Geist sans.
- **Light mode only.** A `.dark` block and the `dark` custom variant exist (carried in from the registry) but are **unused** — there is no theme toggle. Don't wire one up without a decision.
- Use **semantic classes** (`bg-primary`, `text-muted-foreground`, `bg-sidebar-accent`, `border`), never raw hex or `indigo-600`. Conditional classes via `cn()` from `@/lib/utils`.

### Design language (current source of truth)

A deliberate, editorial look that **avoids the generic AI-app aesthetic** (rounded + shadowed + accent-heavy). This refines the theme described in [decisions/0005](./decisions/0005-ui-stack.md) — that ADR still holds for the stack and the indigo-light-mode choice, but the specifics below supersede its "indigo accent" framing.

- **Sharp corners.** `--radius` is `0.25rem` (4px); all the `--radius-*` steps derive from it. Don't bump radii per-component.
- **Flat surfaces.** No elevation shadows — a utility-layer rule in `globals.css` neutralizes every `shadow-*` to `box-shadow: none`. Hairline `border`s define edges; use `border` for separation, never a shadow.
- **Indigo, sparingly.** The UI is mostly **monochrome neutral grays**. The `--accent`/`--accent-foreground` and `--sidebar-accent`/`--sidebar-accent-foreground` tokens are neutral (not indigo). Indigo lives only in `--primary` and `--ring`, so it surfaces only on primary buttons, focus rings, links, and the active nav item's icon (`app-sidebar.tsx` adds `text-primary` to the active icon). Don't tint hovers/cards/backgrounds indigo — the accent reads as intentional precisely because it's rare.
- **Global `cursor-pointer`.** A base-layer rule gives `button`s and ARIA-interactive roles (`menuitem`, `option`, `switch`, `tab`, `checkbox`, `radio`, …) a pointer cursor — Base UI / shadcn omit it by default. Don't add `cursor-pointer` per component.
- **Minimal auth surfaces.** `(auth)/login/page.tsx` is a single centered column — logo mark + product name + one Google button + "Lazer staff only" — no card, no gradient. Keep auth/marketing screens uncluttered.

### Brand (intentionally off-accent)

The brand mark is `public/icon.svg` — **magenta** (`#FF00C5`) — exposed via `<LogoMark>` / `<Logo>` in `src/components/brand/logo.tsx`. The product name is `APP_NAME` ("Lazer Home") in `src/lib/constants.ts`. The magenta mark against the indigo UI accent is **intentional** — the logo is the brand, the indigo is the product chrome; don't "fix" the mismatch.

## Routing & auth gating

Route groups split public from authenticated, and **route protection lives in a layout, not middleware** — consistent with the project's deliberate no-middleware stance (see ADR-0006).

- `src/app/(app)/**` — authenticated pages. `(app)/layout.tsx` is an async Server Component that calls `getCurrentUser()` and `redirect("/login")` when there's no session, **then** `getCurrentStaff(user)` and redirects any non-`ok` status to the single `/profile-setup` route (status is `ok` = active staff record with employment — see [flows.md](./flows.md)). Every page under `(app)` is gated by this two-step check.
- `src/app/(onboarding)/**` — the post-login **block** screen for authenticated users who can't enter the app yet: a single `profile-setup/page.tsx` that resolves `getCurrentStaff` itself and shows different title/body for the `incomplete` vs `not_setup` cases (same screen). There is **no group `layout.tsx`** — the page self-gates like `/login`: it redirects unauthenticated users to `/login` and bounces `ok` users back to `/` so a fixed-up profile is never stuck here. The screen uses `OnboardingNotice` (`src/components/auth/onboarding-notice.tsx`) — a login-styled full-screen notice (logo + title + message + `SignOutButton`).
- `src/app/(auth)/**` — public pages (currently only `/login`, which redirects to `/` if already signed in).
- The old default `src/app/page.tsx` was deleted; the home dashboard is now `(app)/page.tsx` served at `/`.

Pages today: `/login` (landing + Google button), `/` (home dashboard with placeholder stat cards), `/staff` ("Browse staff" — searchable/filterable directory of everyone) and `/staff/[id]` (a person's profile), `/profile` ("My profile" — your own staff profile, same presentation as `/staff/[id]`), `/companies` ("Companies" — the CRM company list, see below) and `/companies/[id]` (a company's read-only detail page), `/contacts` ("Contacts" — the CRM people list, see below) and `/contacts/[id]` (a contact's read-only detail page), `/opportunities` (the CRM pipeline kanban board, see below), `/projects` (client engagements + staffing, see below), `/feedback` (peer feedback — two-tab list + `/feedback/new` give-feedback page + `/feedback/[id]` detail, see [performance.md](./domains/performance.md)), `/settings` (account info synced from Google + sign out), `/profile-setup` (the single onboarding block screen, two messages), plus the localhost-only `/admin` surface (see below). `NAV_ITEMS` (`src/components/app-shell/nav.ts`) has **nine entries**: Home (`IconHome`), My profile → `/profile` (`IconUser`), Staff (`IconUsers`), Companies → `/companies` (`IconBuildingSkyscraper`), Contacts → `/contacts` (`IconAddressBook`), Opportunities → `/opportunities` (`IconTargetArrow`), Projects → `/projects` (`IconBriefcase`), Feedback → `/feedback` (`IconMessageHeart`), Settings (`IconSettings`). Admin is **not** in `NAV_ITEMS` but **does** appear in the sidebar footer as a **conditional, localhost-only** entry (`IconTool` → `/admin`): `app-sidebar.tsx` renders it only when its `isLocal` prop is true (see the app-shell section below), so it's visible when developing locally and absent in any non-loopback deployment.

### `/admin` — localhost-only tooling surface

The admin area lives **outside the `(app)` route group on purpose**: the `(app)` layout gates on an active staff record, but the staff-upload tool is what *creates* those records (chicken-and-egg). `src/app/admin/layout.tsx` is the whole gate — it `notFound()`s the entire segment unless `isLocalhost()` (`src/lib/admin.ts`), so there's no RBAC check here; being local *is* the boundary. Its own chrome (header + "Back to app" link) replaces the app shell. `admin/page.tsx` is a card grid linking to the four tools:

- `admin/manage-users` — edit application users' RBAC role + ban status inline, save all at once (`manage-users.tsx`; see [domains/permissions.md](./domains/permissions.md)). `promote-self-button.tsx` is a dev shortcut here.
- `admin/bulk-edit-roles` — edit staff employment facts (role, line of business, billability, management) across the team in one pass (`bulk-edit-roles.tsx`).
- `admin/upload-staff` — import staff from a Rippling CSV export (`staff-import.tsx`).
- `admin/upload-pto` — import leave records from a Rippling CSV export (`pto-import.tsx`).

The two CSV importers share the **`CsvImport`** harness (`csv-import.tsx`) — a file → preview → commit flow that owns the PapaParse config, file/reset lifecycle, badge summary, preview sections and commit orchestration; each page supplies only its columns, labels, transform, and the two next-safe-action hooks (preview + commit). Both importers render read-only previews via `data-table.tsx` (a TanStack React Table wrapper over the vendored `table` primitive). The two inline-edit tools (`bulk-edit-roles`, `manage-users`) instead share `editable-table.tsx` (its own TanStack-based editable grid) and `table-filters.tsx`.

### Profile pages — `ProfileView` (shared by `/profile` and `/staff/[id]`)

`ProfileView` (`src/components/staff/profile-view.tsx`) is the single presentation for a staff profile — name, avatar, employment badges, links, client intro, résumé, skills, **compensation** (card shown only when the `canViewCompensation` prop is true), history drawer, PTO. **Both** `/profile` (own record) and `/staff/[id]` (anyone) render it; they differ only in which `staffId` they resolve and pass in. The pages don't touch `db` — they `await` actions-layer **reads** (server-only async functions, not `'use server'` actions; see [ADR 0010](./decisions/0010-actions-layer-owns-db-access.md)):

- **`/staff/[id]`** (`(app)/staff/[id]/page.tsx`) `Promise.all`s `getStaffProfile(id)` / `getStaffPto(id)` / `getStaffAvatar(id)` / `getCurrentUser()`, then resolves `canEditStaff` + `canViewCompensation` and awaits `getStaffHistory(id, canViewComp)` **separately** (its `includeCompensation` arg depends on the comp gate — comp entries must be filtered at the read since `HistorySheet` is a client component). `notFound()`s on an unknown id, and uses `generateMetadata` to title the tab with the person's name (shares the `React.cache`d `getStaffProfile` query). Profile/history/avatar reads aren't ownership-scoped — the `(app)` gate is the boundary — but `getStaffPto` returns `null` for non-owners without `pto.review` (so `ProfileView` hides the PTO section) and the Compensation card / comp history are gated by `canViewCompensation` (own always; others need `staff.viewCompensation`). See [domains/permissions.md](./domains/permissions.md) and [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md).
- **`/profile`** (`(app)/profile/page.tsx`) is the same, with the id resolved from the session (`getCurrentStaffId()`); the `getMy*` reads are thin wrappers over the `getStaff*` cores.

`getStaffProfile` returns the profile fields (incl. `resume`/`resumeUpdatedAt`) plus the **latest** `staffEmployment` row via the [ADR 0007](./decisions/0007-staff-employment-effective-dating.md) ordering. Enums display via `humanizeEnum()`, calendar dates via the timezone-safe `formatDate()`, instant `timestamp`s (e.g. `resumeUpdatedAt`) via `formatTimestamp(value: Date)`, and avatar fallbacks via `initialsFor()` — all from `src/lib/format.ts` (`initialsFor` was moved here to be shared; see the dates gotcha in [staff-profiles.md](./domains/staff-profiles.md)).

**Edit (Links + Client intro + Résumé cards).** Each editable card puts its **Edit button in the card header via `CardAction`** (`src/components/ui/card.tsx` — a header slot that right-aligns into the grid; use it for header-level affordances rather than absolutely positioning a button). The buttons live in `edit-links-dialog.tsx` / `edit-client-intro-dialog.tsx` / `edit-resume-dialog.tsx` — client components taking a **`staffId` prop** and rendering the shared **`FormDialog`** (`src/components/form/form-dialog.tsx`), whose `children` is a render prop handed a `close` callback. `FormDialog` (via its `useRemountOnOpen` hook) **remounts the inner form on each open** — it bumps a `formKey` when opening so the form's `defaultValues` refresh from the latest server data after a save rather than going stale, while keeping the form mounted through the close animation (so the popup doesn't collapse to its header before fading out). `useRemountOnOpen` also takes an optional `controlled` `{ open, onOpenChange }` for dialogs driven by a parent (the CRM inline create-from-form dialogs). The form uses **`useHookFormAction`** (tight binding; see `.claude/rules/forms.md`) wired to the `updateStaff*` action with the imported `.schema.ts` as the `zodResolver`; the shared **`FormDialogFooter`** renders the server-error line + Cancel/Save, driving the Save button's `loading` off `action.isPending` and reading errors off `action.result.serverError`, and the action's `onSuccess` calls `close`. (Forms whose shape diverges from the action input instead pair `useZodForm` — `src/hooks/useZodForm.tsx`, a thin `useForm` + auto-`zodResolver` wrapper — with `useAction`; both patterns are documented in `.claude/rules/forms.md`.) The actions are gated server-side by the `authorizeStaffEdit` hook via `metadata({ authorize })` — backed by `canEditStaff` (own → always; other → `staff.edit`); the dialog UI is still rendered for everyone, so a non-permitted user clicking Save on someone else's profile gets the server error rather than a hidden button. See [domains/permissions.md](./domains/permissions.md), [ADR 0014](./decisions/0014-rbac-better-auth-access-control.md).

**Skills card — edited on a dedicated page, not a dialog.** Unlike the three cards above, the Skills card's Edit link (shown only when `canEdit`) navigates to `/staff/[id]/skills` rather than opening a `FormDialog` — the three-bucket editor (`edit-skills-form.tsx`) is too large for a dialog. That page `notFound()`s any viewer who fails `canEditStaff`. The read view (`skills-section.tsx`) groups skills by proficiency level as `Badge` chips. See *Skills* in [domains/staff-profiles.md](./domains/staff-profiles.md) and [ADR 0018](./decisions/0018-skills-inline-jsonb-catalogue.md).

The **résumé dialog additionally** has an "Upload PDF" button: it reads the file to base64 (`FileReader`, guarded to ~6 MB), runs the `parseResumePdf` action via a separate `useAction`, and on success `setValue`s the returned text into the textarea (dirty + validated) for review before the user saves via `updateStaffResume`. The PDF itself is never uploaded for storage — only its extracted text. Upload errors render inline. The résumé card shows the text with `whitespace-pre-wrap` plus an "Updated …" line from `formatTimestamp(resumeUpdatedAt)`. See [ADR 0013](./decisions/0013-resume-pdf-parse-not-store.md).

**Compensation card — read-only, gated.** Rendered only when the `canViewCompensation` prop is true (own profile always; others need `staff.viewCompensation`). `CompensationSection` (`src/components/staff/compensation-section.tsx`) lists base / hourly rate / currency from the latest employment row, with the **guaranteed & discretionary bonus rows shown only when non-zero**; amounts are formatted via `formatMoney` from the pure `src/lib/currency.ts`, and the **currency shows as its bare code** (e.g. `CAD`, not a translated label — there is no currency-label map; the former `CURRENCY_LABELS` was dead and has been removed). There is **no edit affordance** — compensation is import-only (see [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md)).

**History drawer.** The header's "History" button is `HistorySheet` (`src/components/staff/history-sheet.tsx`) — a **purely presentational** client component taking the `HistoryEntry[]` from `getStaffHistory()` and rendering each entry as a category `Badge` + date + summary inside a right-side `Sheet`. Category-agnostic: only `EMPLOYMENT` entries are wired today (allocation is still a stub); comp amounts are **folded into those entries' summaries** rather than forming a separate category (`CATEGORY_LABEL` has no `COMPENSATION` key). Because it's a **client** component, comp is filtered upstream at the read (`getStaffHistory`'s `includeCompensation`), not here — see the history feed flow in [staff-profiles.md](./domains/staff-profiles.md).

### `/staff` — Browse staff directory

`(app)/staff/page.tsx` (Server Component) `await`s `getStaffDirectory()` once, then renders the `StaffDirectory` client component (`src/components/staff/staff-directory.tsx`) with the entries + filter options. **All search and filtering is client-side over that single server fetch** (no URL params, no per-filter server query): name search, plus line-of-business / role / employment-type filters (options from `staffDirectoryFilterOptions`), a **skills filter** (see below), plus an **"active only" toggle that defaults ON** — inactive staff are fetched but hidden until it's turned off. Results render as a card grid of `staff-card.tsx`, each linking to `/staff/[id]`.

**Skills filter (`SkillsFilter` + minimum level).** A grouped, searchable multi-select Base UI `Combobox` (chips) fed from `SKILL_CATEGORIES` — picking multiple skills applies **AND** semantics (a person must hold all of them). When ≥1 skill is selected, a "Minimum level" segmented `ToggleGroup` (Any / Intermediate+ / Senior) appears; it raises the required proficiency bar. Filtering delegates to the pure `matchesSkillFilter(entry.skills, skills, minimum)` from `src/lib/skills.ts` (`"Any"` → `undefined` minimum → no level constraint). While a skill filter is active, each `StaffCard` gets a `highlightedSkills` prop and renders `Badge`s for the person's matched skills with their level label — so the card shows *why* it matched. All of this runs in-memory over the once-fetched list.

### `/companies` + `/contacts` — CRM companies and people (two pages)

Companies and contacts each get their **own page** (`(app)/companies/page.tsx`, `(app)/contacts/page.tsx`) — the old combined "Companies & Contacts" page was split so each entity has focused space. Both are Server Components with the same shape: one **server-side-paginated** table over a `get*Page` read `Promise.all`'d with `getCurrentUser()`, page in its own query param (`companiesPage` / `contactsPage`), and an "Add" dialog gated on one `canEdit = userHasPermission(user, { crm: ["edit"] })` flag.

- **Pagination is server-side**, not client-side over one fetch (unlike `/staff`) — the datasets are expected to grow. `pagination-controls.tsx` renders `Link`-based prev/next that rebuild the URL preserving other params (it takes the full `searchParams` + its own `paramKey` + `basePath`); pages are clamped server-side.
- **`companies-table.tsx` / `contacts-table.tsx`** are presentational (plain shadcn `table`). The companies table shows a **"Partner" badge only when `isPartner` is true** — there is no "Client" badge (a company is neither client nor partner by default; see [domains/crm.md](./domains/crm.md)). The contacts table shows the joined `companyName`, the resolved `managerName` (the "Manager" column, `—` when none), and `linkedinUrl` as an external "Profile" link (`IconBrandLinkedin`, opens in a new tab) in a "LinkedIn" column. **Both tables now drill into the detail pages:** the name cell links to `/companies/[id]` / `/contacts/[id]`, and the contacts table's company cell links to `/companies/[id]` (falling back to `EmptyCell` when the contact has no company). Two small top-level shared components back these cells: **`src/components/empty-cell.tsx`** renders the muted `—` placeholder for a missing value, and **`src/components/external-link.tsx`** the new-tab outbound link — reused wherever a table cell is empty-or-link.
- **"Add company" / "Add contact" dialogs** (`add-company-dialog.tsx` / `add-contact-dialog.tsx`) render **only when the user holds the `crm.edit` capability** — each page computes the `canEdit` flag and conditionally renders its trigger. The server actions are gated regardless, so this is an affordance, not the boundary. Forms use the tight `useHookFormAction` binding (see `.claude/rules/forms.md`); the contact form's company picker is the shared `CompanyComboboxField` and its manager picker is `ManagerComboboxField` (both below).
- **Contact form is create-only** — there's no edit flow, so `linkedinUrl` and `managerId` can only be set at creation. Choosing a company **resets** any prior manager selection (a manager is scoped to the chosen company).
- **Manager picker — `manager-combobox-field.tsx` (`ManagerComboboxField`).** A **thin wrapper over the shared `EntityCombobox`** (below): it wraps the base single-select in a `FormField` and passes the form's chosen `companyId` through `searchArgs` to `searchContacts`, which scopes results to *other contacts at the same company*. The caller renders it only once a company is chosen and keys it by `companyId` (remounting on a company switch so no stale colleague lingers — a contact with no company can't have a manager). Clearable, since a manager is optional; reports `{ id, name } | null`. The same-company rule is also re-enforced server-side in `createContact` — see [domains/crm.md](./domains/crm.md), [ADR 0022](./decisions/0022-contact-manager-self-reference.md). (`CompanyCombobox` / `CompanyComboboxField` remain separate controls — see below.)
- **Shared company picker — `company-combobox-field.tsx` (`CompanyComboboxField`).** The single "pick or create a company" control, used **identically by both the contact form (`add-contact-dialog.tsx`) and the opportunity form (`add-opportunity-dialog.tsx`)**. It composes the `company-combobox` search-picker with a **"New company" button** that opens the small `CreateCompanyInlineDialog`. Creation is **persist-first**: the inline dialog runs `createCompany`, and only on success reports `{ id, name }` back so the field auto-selects the freshly-persisted company — no nested transaction and no refetch round-trip. It reports the chosen `{ id, name } | null` and is clearable, so it serves both the contact form's optional company and the opportunity form's required one (which passes an `error`). (Website is an optional field; the inline dialog has no partner subtitle.)
- **`company-combobox.tsx`** — a debounced, searchable company picker built on the vendored `combobox` (Base UI Combobox). Built-in filtering is disabled (`filter={null}`) — filtering is server-side: it runs `searchCompanies` via `useAction`, keyed off the input debounced through `useDebouncedValue` (`src/hooks/`, 250ms). It **only searches once the user has typed** (a blank query returns nothing and clears results), and is clearable (a contact's company is optional). The selected company is kept in the item list even when absent from current results, so its label/selected state render. Selecting reports `{ id, name }`. See [domains/crm.md](./domains/crm.md).

### `/companies/[id]` + `/contacts/[id]` — CRM detail pages (read-only)

The app's **first drill-down detail views** and its **first tabbed pages** (first use of the vendored `tabs` primitive). Both are Server Components that `await` a `get*Detail` read (server-only, not a `'use server'` action) and `notFound()` on a miss; each has a `generateMetadata` titling the tab (company name / contact name). **Read-only — no edit affordances yet** (consistent with contacts/companies being create-only; see [domains/crm.md](./domains/crm.md)). Both cap width at `max-w-4xl`.

- **`company-detail-view.tsx`** — identity header (building icon, name, optional "Partner" badge, website `ExternalLink`) + `Tabs` (**Opportunities / Projects / Contacts**), each a plain `table` of the rows tied to that `companyId` with an empty-state note. Opportunities/projects have no detail page yet, so they render as plain rows; the **Contacts tab links each person to `/contacts/[id]`** — it serves as the company's people directory.
- **`contact-detail-view.tsx`** — identity header (`Avatar` initials, name, optional role) + a **Details `Card`** (email/phone/LinkedIn/company/manager, all optional; empty → em dash; company & manager render as links to their detail pages) + `Tabs` (**Opportunities / Projects**). The Opportunities tab shows **two labelled groups** — "Referred by this contact" and "Also involved in" — so attribution stays honest; the Projects tab lists projects *derived* from the contact's referred opportunities (contacts don't attach to projects directly). Opportunity/project rows link their company.
- Both share **`opportunity-status-badge.tsx`** — renders a leaf status as its pipeline **group** label (`groupOfStatus(...).label`), `default` variant for `closed_won`, `secondary` otherwise. And the small `TabLabel` (label + muted count) so tab counts show without switching. Tables reuse `EmptyCell` / `ExternalLink`.

### `/opportunities` — Opportunities kanban board (CRM)

`(app)/opportunities/page.tsx` (Server Component) `await`s `getOpportunitiesBoard()` (non-paginated — the pipeline is small) `Promise.all`'d with `getCurrentUser()`, computes `canEdit = userHasPermission(user, { crm: ["edit"] })`, and renders `OpportunityBoard` (was a paginated `opportunities-table.tsx`, now **removed**) plus a `canEdit`-gated "Add opportunity" dialog. **dnd-kit** (`@dnd-kit/core` + `@dnd-kit/sortable` + `@dnd-kit/utilities` — new deps) powers the drag-and-drop.

- **`opportunity-board.tsx`** (`OpportunityBoard`) — the client board. **Cards are the canonical state; columns are derived** from `OPPORTUNITY_GROUPS` (`src/lib/opportunity-pipeline.ts`) + which groups are locally expanded, so toggling collapse never re-keys stored state. Multi-substatus groups (Scoping/Allocating/Closing) render **collapsed by default** into one column (cards show a substatus `Badge`) with an expand toggle, or **expanded** into subcolumns; single-status groups are plain columns. A **name search box filters cards client-side** — but drag math uses the *full* (unfiltered) column neighbours, so reordering while searching can't corrupt order against hidden cards. Drags are **optimistic**: `onDragEnd` computes an absolute `{ status, position }` (via `resolveTargetStatus` + `computePosition` over the destination column's neighbours), applies it locally, and runs `updateOpportunityPosition`; on error it reverts to a pre-drag snapshot (the failed write left the DB unchanged). A `useEffect` re-adopts fresh server data only when the payload *signature* changes, so a post-move revalidation doesn't clobber in-flight moves. Cards sort by `(position, createdAt)` — same tie-break as the server. When `!canEdit` the board renders **without** the `DndContext` (static). Dropping onto a collapsed group picks the group's first substatus (dragged from earlier) or last (from later).
- **`opportunity-board-column.tsx`** / **`opportunity-card.tsx`** — a droppable column (dnd-kit) with an optional expand/collapse toggle, and the draggable card (company name, source, owner names, optional substatus badge; a `DragOverlay` clone renders while dragging).
- **`add-opportunity-dialog.tsx`** is the form (a large dialog, `sm:max-w-2xl`). Company is a single required `CompanyComboboxField` (the same shared picker the contact form uses, here passed an `error`); related **contacts** and referral **source contacts** use the `ContactsComboboxField` wrapper, while **owners** and referral **source staff** are bare `EntityMultiCombobox` (staff are search-only). The conditional referral rules (`staff_referral` needs ≥1 source staff, `contact_referral` needs ≥1 source contact) are validated by calling `createOpportunitySchema.safeParse(...)` in the form's `onSubmit` (there's no `zodResolver`), matching the server `superRefine`. Mapping those issues onto the form is now done by the **shared `applyServerIssues` helper** (`src/components/form/apply-server-issues.ts`), which replaced the hand-rolled "iterate issues → `setError`" loop duplicated across the create forms — it takes a `fieldMap` translating each schema field to its form path (incl. reconstructing dotted `array.index.subField` paths for `useFieldArray` rows, as the project form needs).
- **`entity-multi-combobox.tsx`** (`src/components/form/`) — a reusable multi-select on Base UI **ComboboxChips**, server-filtered via an injected search action (any action satisfying the generic **`SearchAction`** contract in `src/lib/search.ts` — e.g. `searchContacts`/`searchStaff`), with an **optional inline "create" affordance**. **Company + contacts are searchable OR inline-creatable; owners + referral staff are search-only** (staff are never created inline — they come from the imported roster).
- **`create-contact-inline-dialog.tsx` / `create-company-inline-dialog.tsx`** — small stacked dialogs (`sm:max-w-sm`, sized down for visual separation from the form that spawns them) that reuse the existing `createContact` / `createCompany` actions so a missing contact/company can be created without leaving the form (this is why `createContact` now returns `{ id }`). `CreateCompanyInlineDialog` is shared: it's the dialog behind the "New company" button in `CompanyComboboxField`, so **both** the contact and opportunity forms create companies through it (persist-first via `createCompany`, then select). `CreateContactInlineDialog` is opportunity-form-only. See [domains/crm.md](./domains/crm.md).

### `/projects` — Projects (client engagements + staffing)

`(app)/projects/page.tsx` (Server Component) mirrors the `/opportunities` pattern: a single **server-side-paginated** `projects-table.tsx` over `getProjectsPage` `Promise.all`'d with `getCurrentUser()`, page in a `projectsPage` query param (reusing `crm/pagination-controls.tsx` with `basePath="/projects"`), and an "Add project" dialog gated on `canEdit = userHasPermission(user, { projects: ["edit"] })` — **`projects.edit`, not `crm.edit`** (see [domains/projects.md](./domains/projects.md)). The table columns are Name, Company (joined), Delivery managers (comma-joined names, `—` when none), and Roles (count).

- **`add-project-dialog.tsx`** — the create form in a large `FormDialog` (`sm:max-w-2xl`, scrollable). It uses **loose binding** (`useForm` + `useAction`, per `.claude/rules/forms.md`) because the form shape ≠ the action input (a `useFieldArray` roles repeater and combobox objects vs. flat id arrays). On submit it maps form values → the action payload and runs `createProjectSchema.safeParse` client-side, mapping issues (including per-row `roles.N.field` paths) into `setError` — matching the server schema — before `execute`. Company is a required `CompanyCombobox` (fed the `projects/searchCompanies` action); delivery managers use `EntityMultiCombobox` (search-only, `projects/searchStaff`); each role row has a single-staff `EntityCombobox`, a line-of-business `EnumSelect`, two `DatePicker`s, and an hours/day number input.
- **`entity-combobox.tsx`** (`EntityCombobox`, `src/components/form/`) — the **single-select** sibling of `EntityMultiCombobox`, and the **shared single-select base** for the CRM/projects pickers. Server-filtered via an injected search action; reports `{ id, name } | null`; search-only (no inline create). Its **`searchArgs` prop** passes extra, non-query arguments to the search action (e.g. a `companyId` scope) and re-runs the search when it changes (callers must keep it referentially stable, e.g. `useMemo`). Used directly for a project role's one staff member, and wrapped by **`CompanyCombobox`** (adapts to the flat `{ value, selectedName }` props, defaults to the `crm.edit` company search) and **`ManagerComboboxField`** (passes `companyId` via `searchArgs` to `searchContacts`).
- **`enum-select.tsx`** (`src/components/form/`) — `EnumSelect`, the enum-driven `Select` helper **extracted from `add-opportunity-dialog.tsx`** for reuse; takes an options tuple + a labels record (here `LINE_OF_BUSINESS` / `LINE_OF_BUSINESS_LABELS` from `src/lib/line-of-business.ts`).

## App shell & sidebar

`src/components/app-shell/`:

- `app-shell.tsx` — Client Component; wraps `SidebarProvider` and renders the sidebar plus the page content in a padded (`p-4 md:p-6`) `<div>`. **There is no global header bar** — the old `<header>` showing a `titleForPath(pathname)` page title was removed, so `AppShell` no longer reads `usePathname()`. Each page owns its own in-page `<h2>` title (and sets the browser-tab title via `export const metadata`). It sets `defaultOpen={false}` so the sidebar starts **collapsed as a floating icon island**. Takes an **`isLocal`** prop and passes it straight through to `AppSidebar`. (`TooltipProvider` is no longer here — it now lives in the root layout; see below.)
- `app-sidebar.tsx` — the sidebar itself; uses `<Sidebar variant="floating" collapsible="icon">` (floating icon rail), `size-5` icons. The open/close **toggle lives in its footer** — a `SidebarMenuButton` (Tabler `IconLayoutSidebar`) calling `useSidebar().toggleSidebar` (label toggles Expand/Collapse). It also takes an **`isLocal`** prop that drives the **localhost-only Admin footer entry** (`IconTool` → `/admin`, rendered only when `isLocal`). The prop originates in the server `(app)/layout.tsx`, which computes `await isLocalhost()` (`src/lib/admin.ts`) and threads it **server-layout → `AppShell` → `AppSidebar`** — the same loopback-host boundary that gates the `/admin` segment itself, so the nav affordance and the segment agree.
- `nav-user.tsx` — the user dropdown in the sidebar footer.
- `nav.ts` — the nav source of truth (no JSX, so it's importable anywhere).

Collapsed nav items get tooltips automatically via `SidebarMenuButton`'s `tooltip` prop (set to `item.title`).

**Add a nav item:** edit `NAV_ITEMS` in `src/components/app-shell/nav.ts` (`{ title, href, icon }` with a **Tabler** icon, typed as Tabler's `Icon`). It drives the sidebar entries; `isActivePath()` (also in `nav.ts`) highlights the active one. Nothing else to touch. (There's no longer a `titleForPath` helper — the page-title header was removed.)

## Icon-only buttons (tooltip convention)

Icon-only controls must use **`IconButton`** (`src/components/icon-button.tsx`): it requires a `label`, which renders a **tooltip** and sets `aria-label` — so every icon button is discoverable and accessible. Don't ship a bare icon-only button. (Use it for standalone icon-only controls.)

`TooltipProvider` lives in the **root layout** (`src/app/layout.tsx`), not `AppShell`, so tooltips work app-wide (auth screens included). `IconButton` must render inside it — the root provider covers everything.

## Auth UI (Google-only)

- Sign in: `authClient.signIn.social({ provider: "google", callbackURL: "/" })` (`src/components/auth/google-sign-in-button.tsx`). On success the browser is redirected to Google, so the component only handles the error path.
- Sign out: `authClient.signOut()` then `router.replace("/login")` (`src/components/auth/sign-out-button.tsx`).
- Email/password is intentionally disabled server-side — see [decisions/0006](./decisions/0006-google-only-auth-and-layout-gating.md).

## Error / not-found / loading (Next 16 specifics)

This is a **modified Next.js build** — these conventions differ from public Next. Verify against `node_modules/next/dist/docs/` before changing them.

- `error.tsx` — **Client Component.** The retry prop is **`unstable_retry`** in this build, NOT `reset`. Call `unstable_retry()` to re-render the segment.
- `global-error.tsx` — replaces the root layout, so it renders its own `<html>`/`<body>` with inline styles (app CSS isn't available). Also takes `unstable_retry`. Keep it minimal and self-contained.
- `not-found.tsx` — Server Component; trigger with `notFound()` from `next/navigation`.
- `(app)/loading.tsx` — a Skeleton-based suspense fallback for the authenticated segment.
