# Data model ‚Äî the spine

The five domains share one model. This is the most important doc for understanding the system: changes here ripple everywhere.

**Status: mostly proposed.** Entities and relationships below are the intended design. Realized so far: the **Person** anchor (auth `user`), the **Staff profiles** tables (`staff`, `staff_employment`, `staff_pto`), the **CRM** slice (`companies`, `contacts`, `opportunities` + four opportunity junction tables), the **Projects** slice (`projects`, `project_delivery_managers`, `project_roles` ‚Äî the first cut of Allocation), and the first **Performance** slice (`feedback` ‚Äî peer feedback) ‚Äî see "What's realized in code".

## What's realized in code

- **Person** is realized by better-auth's **`user`** table (`src/lib/db/auth-schema.ts`) ‚Äî the _login/identity_ anchor. (`user`/`session`/`account`/`verification` live in our schema and migrations, not better-auth's.) A logged-in user maps to its staff record via the nullable, unique **`staff.userId`** FK (`onDelete: set null`) ‚Äî see [domains/staff-profiles.md](./domains/staff-profiles.md). The user's **`role`** (a `text()` column owned by better-auth's `auth-schema.ts`) drives RBAC; valid values and the role‚Üípermission matrix live in `src/lib/permissions.ts`, validated app-side by `roleSchema` (a `pgEnum` would be clobbered by `auth:generate`). See [domains/permissions.md](./domains/permissions.md).
- **Staff profiles domain** (`src/lib/db/staff-schema.ts`, barrelled by `src/lib/db/schema.ts`):
  - **`staff`** ‚Äî the durable person record (notNull/unique external `ripplingId`, name, non-unique email, optional `userId` link to auth `user`, profile links, `clientIntro` (+ `clientIntroUpdatedAt`), free-text `resume` (+ `resumeUpdatedAt`), an inline **`skills`** jsonb list, lifecycle dates, `isActive`). Both `*UpdatedAt` instants are stamped **explicitly by their update action** when the text changes, not via `$onUpdate` (which would fire on every row write, e.g. an import re-sync). `skills` is the schema's **first jsonb/array column** ‚Äî `StaffSkill[]` (`{ name, level }[]`, NOT NULL default `[]`) picked from the hardcoded catalogue in `src/lib/skills.ts`, deliberately **not** a normalized `skill` table; see [ADR 0018](./decisions/0018-skills-inline-jsonb-catalogue.md).
  - **`staff_employment`** ‚Äî time-varying employment facts (line of business, role, employment type, billability, `utilizationTarget`, `billableType` (NOT NULL, default `HUB`), a boolean `isManagement`, and **compensation** ‚Äî `base`/`hourlyRate`/`guaranteedBonus`/`discretionaryBonus` (`numeric(12,2)`, all **NOT NULL**) + `currency` (**NOT NULL**)), as **one effective-dated row per change**; current state = latest `effectiveFromDate`. FK ‚Üí `staff.id` cascade. `isManagement` and `billableType` are **orthogonal to `role`** (someone can hold a working role *and* be management for it) and set in-app only. **Compensation lives here** rather than a separate table (comp changes coincide with employment changes) ‚Äî it's **import-only** (no edit UI), **required** (the importer skips a row it can't supply comp for; no carry-forward on the import path), and **view-gated** by `staff.viewCompensation` (own comp always visible). See [ADR 0007](./decisions/0007-staff-employment-effective-dating.md), [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md), and [domains/staff-profiles.md](./domains/staff-profiles.md).
  - **`staff_pto`** ‚Äî leave spans (`startDate`/`endDate`, `type`, `isPending`, **notNull/unique `ripplingId`** = Rippling "Leave request ID"). FK ‚Üí `staff.id` cascade. Discrete rows, **not** effective-dated; reduces a person's available capacity for allocations. Populated by a second local-only importer (`/admin/upload-pto`) ‚Äî see [domains/staff-profiles.md](./domains/staff-profiles.md).
- **`lineOfBusinessEnum`** (`line_of_business`: `CORPORATE`, `CORE`, `FINTECH`, `COMMERCE`, `DESIGN`) is a **shared/global enum**, reused beyond staff (now by `project_roles`), not a staff-specific type. Its value tuple + labels live in a pure, client-importable module **`src/lib/line-of-business.ts`** (`LINE_OF_BUSINESS`, `LINE_OF_BUSINESS_LABELS`), the single source both the `pgEnum` in `staff-schema.ts` and the projects zod schema/form import ‚Äî the same single-source pattern opportunities uses for its `source`/`status` enums (see [ADR 0016](./decisions/0016-junction-table-and-shared-enum-conventions.md)).
- **Local-only CSV import tools** are the only things that write the staff domain so far: `/admin/upload-staff` (Rippling export ‚Üí `staff` + effective-dated `staff_employment`) and `/admin/upload-pto` (Rippling leave export ‚Üí `staff_pto`). See [domains/staff-profiles.md](./domains/staff-profiles.md).
- **CRM slice** (`src/lib/db/crm-schema.ts`, barrelled by `schema.ts`; migrations `drizzle/0012_spotty_mysterio.sql`, `drizzle/0013_clean_firelord.sql` (adds `contacts.phone`), `drizzle/0014_nostalgic_crusher_hogan.sql`, `drizzle/0020_unique_blue_shield.sql` (restructures `opportunity_status` to 14 leaf values, adds `opportunities.position`) + `drizzle/0021_glamorous_goliath.sql` (adds `contacts.linkedinUrl` + self-referential `contacts.managerId`)):
  - **`companies`** ‚Äî organisations we deal with. `isPartner` is a **standalone boolean** flag (default false) marking partners ‚Äî not a client/partner split; a company need be neither. `name` notNull, optional `websiteUrl`. We chose *Company* over the narrower *Client* the spine originally named ‚Äî see [ADR 0015](./decisions/0015-crm-company-over-client.md).
  - **`contacts`** ‚Äî people, optionally employed by a company (`companyId` FK ‚Üí `companies`, `onDelete: set null`; null when unknown or once the company is removed). `firstName`/`lastName`/`email` notNull, **`email` unique**, optional `phone`, optional free-text `role`, optional `linkedinUrl`, and an optional **`managerId`** ‚Äî the repo's **first self-referential FK** (‚Üí `contacts.id`, `onDelete: set null`), "managed by" another contact, constrained app-side to a contact at the same company. See [ADR 0022](./decisions/0022-contact-manager-self-reference.md).
  - **`opportunities`** ‚Äî pipeline deals, each **required** to belong to a company (`companyId` FK ‚Üí `companies`, **`onDelete: restrict`** ‚Äî contrast the contacts' set-null; a company with live opportunities can't be deleted). `source` + `status` pgEnums (`status` is a **flat 14-value leaf enum in pipeline order**; the two-level kanban grouping is derived in code in `src/lib/opportunity-pipeline.ts`, not in the DB), optional `nextSteps`, and **`position`** (`doublePrecision`, a single global fractional index for manual kanban ordering ‚Äî a drag writes the midpoint between neighbours, one-row update; index `opportunities_status_position_idx` on `(status, position)`). See [ADR 0021](./decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md). Their related people (contacts, owners, referral sources) live in **four junction tables** ‚Äî the repo's first many-to-many pattern (see [below](#junction-tables--the-first-many-to-many-pattern)). `owners` and `source_staff` link to **`staff`**, the first CRM ‚Üî staff FK.
  - Open reads + `crm.edit`-gated create mutations (one capability gates all CRM writes ‚Äî companies, contacts *and* opportunities) + `/companies`, `/contacts` and `/opportunities` pages exist (companies and contacts are on **separate pages** now). The **won-Opportunity ‚Üí Project link now exists at the schema level** (`projects.opportunityId`, [ADR 0019](./decisions/0019-project-opportunity-link.md)) but the **handoff flow is still proposed** ‚Äî nothing populates it yet, so projects are still created standalone. See [domains/crm.md](./domains/crm.md).
- **Projects slice** (`src/lib/db/projects-schema.ts`, barrelled by `schema.ts`; migrations `drizzle/0015_premium_vertigo.sql`, `drizzle/0017_amused_corsair.sql` (adds `projects.opportunityId`)):
  - **`projects`** ‚Äî billable work that always belongs to a company (`companyId` FK ‚Üí `companies`, **`onDelete: restrict`** ‚Äî like opportunities, a company with live projects can't be deleted). `name` notNull. id prefix `proj`. Also carries an **optional `opportunityId`** (nullable FK ‚Üí `opportunities`, **`onDelete: restrict`**, index `projects_opportunity_idx`): the CRM ‚Üí delivery link, **1:N** (one opportunity ‚Üí many projects; a project relates to at most one). Optional so projects can be created standalone; **restrict** mirrors `companyId`. Schema-only today ‚Äî no flow populates it, and the same-company invariant (opportunity's company == project's company) is **not** enforced. See [ADR 0019](./decisions/0019-project-opportunity-link.md).
  - **`project_delivery_managers`** ‚Äî junction (many staff per project) following the CRM junction convention exactly ([ADR 0016](./decisions/0016-junction-table-and-shared-enum-conventions.md)): surrogate `text` PK, `unique(projectId, staffId)`, index on `staffId`, both FKs cascade.
  - **`project_roles`** ‚Äî a **staffing line** (NOT a pure junction ‚Äî carries columns): `projectId` ‚Üí projects (cascade), `staffId` ‚Üí staff (**restrict**), `lineOfBusiness` (shared enum), `startDate`/`endDate` (date, string mode), `hoursPerDay` (numeric(4,2), default 8). Indexed on both FKs; **no `unique` on the FK pair** (a person can hold multiple lines on one project). This is the **first concrete cut of the proposed Allocation entity** ‚Äî stored as simple mutable rows, *not* effective-dated history. See [domains/projects.md](./domains/projects.md), [domains/allocations.md](./domains/allocations.md), and [ADR 0017](./decisions/0017-project-roles-as-first-allocation-cut.md).
  - Open reads + `projects.edit`-gated create + `/projects` page exist; create + read only (no edit/delete). See [domains/projects.md](./domains/projects.md).
- **Performance slice** (`src/lib/db/performance-schema.ts`, barrelled by `schema.ts`; migration `drizzle/0018_wild_beast.sql`):
  - **`feedback`** ‚Äî peer feedback: `fromStaffId` + `toStaffId` (both FK ‚Üí `staff.id`, both `onDelete: cascade`, each indexed), a 5-point `feedback_rating` pgEnum, required `context`, optional `keepDoing`/`stopDoing`/`startDoing` (at least one required, enforced in the zod schema)/`other`, and an optional `messageToRecipient`. **Point-in-time** (an immutable event log ‚Äî NOT effective-dated; a person may leave feedback about the same person repeatedly, so **no unique `(from, to)`**). The rating enum's values/labels live in the pure, client-importable **`src/lib/feedback-rating.ts`** (same single-source pattern as `line-of-business.ts`). Privacy is enforced **by the read projections, not the table** ‚Äî three tiers (give = open to active staff via the `authorizeFeedbackCreate` hook; recipient sees only giver name + `messageToRecipient`; `feedback.review` = manager/admin see all in full). See [domains/performance.md](./domains/performance.md) and [ADR 0023](./decisions/0023-feedback-privacy-tiers.md).
- Nothing else (the won-Opportunity ‚Üí Project *flow* ‚Äî the `opportunityId` FK exists but is never populated ‚Äî a full Allocation domain, TimeEntry, review cycles/goals) exists yet. (The legacy `staff_profile` example table was deleted ‚Äî `drizzle/0003_tranquil_miek.sql` drops it.)

## Core entities

- **Person** ‚Äî an employee/contractor. Anchors most of the system. Realized as **`staff`** (durable identity) + **`staff_employment`** (effective-dated role/billability facts); login identity is the auth `user`, linked via the optional `staff.userId` (1:1, null until first sign-in).
  - **Skill** ‚Äî capability + proficiency. _(Realized, but **not** as the many:many table this originally proposed:_ an inline `jsonb` array on `staff` (`skills`), catalogue-driven from `src/lib/skills.ts`. Cross-staff skill matching for allocations is still proposed and would need an indexed rework ‚Äî see [ADR 0018](./decisions/0018-skills-inline-jsonb-catalogue.md).)
  - **Compensation** (base/hourly/bonuses + currency) is **realized** as effective-dated columns on `staff_employment` itself, not a separate table ‚Äî import-only, required, view-gated. See [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md). External charge rates (what we bill a client) are still proposed and would follow the same history-as-rows pattern.
- **Company** ‚Äî an organisation we deal with (CRM); an `isPartner` flag marks partners (standalone, not a client/partner split). _(Realized ‚Äî `companies`.)_ Chosen over the narrower *Client* the spine first named; see [ADR 0015](./decisions/0015-crm-company-over-client.md).
  - **Contact** (optional company per contact) ‚Äî a person, optionally at a Company, optionally "managed by" another contact at the same company (self-referential `managerId`). _(Realized ‚Äî `contacts`; `companyId` nullable, `onDelete: set null`; `managerId` nullable self-ref, `onDelete: set null`, same-company invariant app-enforced ‚Äî see [ADR 0022](./decisions/0022-contact-manager-self-reference.md).)_
  - **Opportunity** ‚Äî a pipeline deal that always belongs to a Company; when _won_, will become/link to a Project. _(Realized ‚Äî `opportunities` + junction tables; `companyId` required, `onDelete: restrict`. The Project link exists at the schema level (`projects.opportunityId`, 1:N) but nothing populates it ‚Äî see [ADR 0019](./decisions/0019-project-opportunity-link.md).)_
- **Project** (a.k.a. engagement) ‚Äî billable work for a Company. The hub linking CRM ‚Üî delivery. _(Realized (first slice) ‚Äî `projects`; required `companyId`, `onDelete: restrict`; optional `opportunityId` (FK ‚Üí `opportunities`, `restrict`, 1:N); delivery managers via `project_delivery_managers`. Create + read only; the `opportunityId` column exists but the won-Opportunity ‚Üí Project handoff flow is still proposed.)_ See [domains/projects.md](./domains/projects.md).
- **Allocation** ‚Äî a _time-ranged_ assignment of a Person to a Project (start/end, % or hours, project role). The heart of capacity planning. _(Partially realized ‚Äî `project_roles` is the first cut: a staff member on a line of business for a date range at N hours/day. Simple rows, not effective-dated history yet ‚Äî see [ADR 0017](./decisions/0017-project-roles-as-first-allocation-cut.md).)_
- **TimeEntry** ‚Äî hours a Person logged against a Project (and optionally a task) on a date; billable or not. Aggregated into **Timesheets** for approval.
- **Feedback** ‚Äî point-in-time peer feedback from one Person about another (rating + context + keep/stop/start + optional recipient message). _(Realized ‚Äî `feedback`; staff‚Üîstaff, tiered visibility ‚Äî see [domains/performance.md](./domains/performance.md) and [ADR 0023](./decisions/0023-feedback-privacy-tiers.md).)_
- **ReviewCycle / PerformanceReview / Goal** ‚Äî periodic assessment of a Person, often informed by their project work and utilization. _(Proposed.)_

## Key derived concepts

- **Utilization** = billable hours √∑ available hours for a Person over a period. Drives capacity (for allocations) and feeds performance.
- **Forecast vs. actuals** ‚Äî Allocations are the _plan_; TimeEntries are the _actuals_. Comparing them is a central reporting need.
- **Margin** = (charge rate ‚àí cost rate) √ó billable hours, per Project/Person.

## Nuances to respect

- Everything time-bound (allocations, rates, reviews, **employment**) needs effective-dated handling ‚Äî a person's role or rate changes over time. The realized pattern is **history-as-rows** (`staff_employment`): a new row per change, current state = latest `effectiveFromDate`. See [ADR 0007](./decisions/0007-staff-employment-effective-dating.md). **Two deliberate exceptions:** (1) **Skills** are stored inline and edited in place, with no per-skill history ‚Äî see [ADR 0018](./decisions/0018-skills-inline-jsonb-catalogue.md); (2) the first Allocation cut (`project_roles`) also stores simple mutable rows rather than effective-dated history, so it can't reconstruct a past plan ‚Äî a deliberate scope call for the first slice, see [ADR 0017](./decisions/0017-project-roles-as-first-allocation-cut.md).
- Rates and salaries are highly sensitive. **Compensation** is now realized (on `staff_employment`) and access-gated by `staff.viewCompensation` (own always visible) ‚Äî see [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md), authorization in [architecture.md](./architecture.md), and the RBAC model in [domains/permissions.md](./domains/permissions.md).
- **Dates and times are timezone-agnostic.** Calendar dates (effective-from, PTO spans, join/termination) are `date()` strings (`"YYYY-MM-DD"`, no zone); instants are plain `timestamp` _without_ time zone ‚Äî treat all stored datetimes as wall-clock and convert zones only at the UI edge. See [`.claude/rules/database.md`](../.claude/rules/database.md).

### Junction tables ‚Äî the first many-to-many pattern

The four `opportunity_*` join tables (`src/lib/db/crm-schema.ts`) are the repo's **first** many-to-many relations and set the convention future join tables should copy (`project_delivery_managers` follows it exactly; `project_roles` follows the FK/index parts but is a **data-carrying** line, not a pure junction ‚Äî see [domains/projects.md](./domains/projects.md)):

- **Surrogate `text` PK** via `generateId(prefix)` (e.g. `opp-contact`), not a composite PK ‚Äî consistent with the app-minted-CUID2 ID convention everywhere else ([`.claude/rules/database.md`](../.claude/rules/database.md)).
- **`unique(...)` on the FK pair** ‚Äî gives set-semantics (a row can't be linked twice). Writers still dedupe input id arrays before insert so a repeat can't trip this index (see `createOpportunity`).
- **`index(...)` on the *non-owning* FK** (e.g. the `contactId`/`staffId` side) for efficient reverse lookups ("which opportunities is this contact on?").
- **Both FKs `onDelete: "cascade"`** ‚Äî a link row is meaningless without both endpoints.

Reading the "other side" cheaply: resolve the linked names with a **single grouped follow-up query** over the relevant ids, not a per-row query (see `getOpportunitiesBoard`'s owner-name resolution) ‚Äî avoid N+1.
