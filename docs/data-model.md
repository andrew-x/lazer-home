# Data model ‚Äî the spine

The five domains share one model. This is the most important doc for understanding the system: changes here ripple everywhere.

**Status: mostly proposed.** Entities and relationships below are the intended design. Realized so far: the **Person** anchor (auth `user`), the **Staff profiles** tables (`staff`, `staff_employment`, `staff_pto`), the **CRM** slice (`companies`, `contacts`, `opportunities` + four opportunity junction tables + the `contact_entries`/`opportunity_entries` note & next-step logs), the **Projects** slice (`projects`, `project_delivery_managers`, `project_roles` ‚Äî the first cut of Allocation), the **Timesheets** slice (`timesheets`, `time_entries` ‚Äî weekly actuals), the first **Performance** slice (`feedback` ‚Äî peer feedback), and a generic **`responses`** survey store (two consumers: the staff-profile **Manual of Me** and **Ways of Working** surveys) ‚Äî see "What's realized in code".

## What's realized in code

> **Migrations were squashed, then resumed incrementally.** The whole drizzle history collapsed into a single baseline **`drizzle/0000_light_shape.sql`** (commit `54ccefc`), so the old per-slice `drizzle/00XX_*.sql` filenames no longer exist ‚Äî the substance of every one still lives in `0000` and the schema modules under `src/lib/db/`. Two incremental migrations now sit on top: **`drizzle/0001_wonderful_thing.sql`** (the CRM timestamped-entries tables ‚Äî `contact_entries`/`opportunity_entries` + the shared `crm_entry_kind` enum, and the drop of the scalar `opportunities.nextSteps` column ‚Äî [ADR 0030](./decisions/0030-crm-timestamped-entries-notes-next-steps.md)) and **`drizzle/0002_loud_sister_grimm.sql`** (the Opportunity ‚Üî Project link inversion + role `status`/provenance ‚Äî [ADR 0019](./decisions/0019-project-opportunity-link.md), [ADR 0031](./decisions/0031-opportunity-project-planner-and-role-status.md)). Below we cite the **schema file** for each slice rather than a migration.

- **Person** is realized by better-auth's **`user`** table (`src/lib/db/auth-schema.ts`) ‚Äî the _login/identity_ anchor. (`user`/`session`/`account`/`verification` live in our schema and migrations, not better-auth's.) A logged-in user maps to its staff record via the nullable, unique **`staff.userId`** FK (`onDelete: set null`) ‚Äî see [domains/staff-profiles.md](./domains/staff-profiles.md). The user's **`role`** (a `text()` column owned by better-auth's `auth-schema.ts`) drives RBAC; valid values and the role‚Üípermission matrix live in `src/lib/permissions.ts`, validated app-side by `roleSchema` (a `pgEnum` would be clobbered by `auth:generate`). See [domains/permissions.md](./domains/permissions.md).
- **Staff profiles domain** (`src/lib/db/staff-schema.ts`, barrelled by `src/lib/db/schema.ts`):
  - **`staff`** ‚Äî the durable person record (notNull/unique external `ripplingId`, name, non-unique email, optional `userId` link to auth `user`, an optional self-referential **`managerId`** ‚Üí `staff.id` (`onDelete: set null`) ‚Äî "reports to", populated **exclusively by the staff CSV import** (matched on the `Manager - Work email` column) with **no in-app editor**, so a re-import never fights a manual edit; it mirrors `contacts.managerId` (which remains the repo's *first* self-referential FK) ‚Äî see [ADR 0026](./decisions/0026-staff-manager-self-reference.md), profile links, `clientIntro` (+ `clientIntroUpdatedAt`), free-text `resume` (+ `resumeUpdatedAt`), an inline **`skills`** jsonb list, lifecycle dates, `isActive`). Both `*UpdatedAt` instants are stamped **explicitly by their update action** when the text changes, not via `$onUpdate` (which would fire on every row write, e.g. an import re-sync). `skills` is the schema's **first jsonb/array column** ‚Äî `StaffSkill[]` (`{ name, level }[]`, NOT NULL default `[]`) picked from the hardcoded catalogue in `src/lib/skills.ts`, deliberately **not** a normalized `skill` table; see [ADR 0018](./decisions/0018-skills-inline-jsonb-catalogue.md).
  - **`staff_employment`** ‚Äî time-varying employment facts (line of business, role, employment type, billability, `utilizationTarget`, `billableType` (NOT NULL, default `HUB`), a boolean `isManagement`, and **compensation** ‚Äî `base`/`hourlyRate`/`guaranteedBonus`/`discretionaryBonus` (`numeric(12,2)`, all **NOT NULL**) + `currency` (**NOT NULL**)), as **one effective-dated row per change**; current state = latest `effectiveFromDate`. FK ‚Üí `staff.id` cascade. `isManagement` and `billableType` are **orthogonal to `role`** (someone can hold a working role *and* be management for it) and set in-app only. **Compensation lives here** rather than a separate table (comp changes coincide with employment changes) ‚Äî it's **import-only** (no edit UI), **required** (the importer skips a row it can't supply comp for; **no carry-forward on the *import* path** ‚Äî every imported row must carry full comp), and **view-gated** by `staff.viewCompensation` (own comp always visible). The no-carry-forward rule is specific to the import path; the **bulk-edit path** (a *non-comp* change spawning a new effective-dated row) *does* **carry comp forward** from the previous row, so advancing someone's role/LoB never clears their salary. See [ADR 0007](./decisions/0007-staff-employment-effective-dating.md), [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md), and [domains/staff-profiles.md](./domains/staff-profiles.md).
  - **`staff_pto`** ‚Äî leave spans (`startDate`/`endDate`, `type`, `isPending`, **notNull/unique `ripplingId`** = Rippling "Leave request ID"). FK ‚Üí `staff.id` cascade. Discrete rows, **not** effective-dated; reduces a person's available capacity for allocations. Populated by a second local-only importer (`/admin/upload-pto`) ‚Äî see [domains/staff-profiles.md](./domains/staff-profiles.md).
- **`responses`** ‚Äî a **generic survey-answer store** (`src/lib/db/responses-schema.ts`, barrelled by `schema.ts`), keyed by **`(staffId, questionId)`** with a `unique` on the pair (‚Üí upsert, one **current** answer per person per question ‚Äî *not* history like `staff_employment`, and the opposite of the append-only `feedback` log). FK `staffId` ‚Üí `staff.id` cascade. `questionId` is plain **`text`** validated in app code (**not a `pgEnum`**), so a new survey needs **no migration**. Three nullable payload shapes (`listResponse` jsonb `string[]`, `textResponse`, `jsonResponse`) ‚Äî a question uses whichever fits, and `upsertResponse` writes the used column and **nulls the other** so a row never carries a stale value from the wrong shape. **Two consumers today**, both staff-profile surveys and both a code-only add (no migration ‚Äî exactly what ADR 0028 anticipated): **Manual of Me** (`src/lib/manual-of-me.ts`, 7 free-text ids ‚Üí `textResponse` only) and **Ways of Working** (`src/lib/ways-of-working.ts`, 28 `WOW_`-prefixed ids ‚Üí `textResponse` for free-text/single-select, `listResponse` for multi-selects and the AI usage/savings matrix buckets). The write schema's `questionId` is now `z.union([manualOfMeQuestionId, waysOfWorkingQuestionId])`; each new survey adds its enum to that union. See [ADR 0028](./decisions/0028-generic-responses-table-app-validated-question-ids.md) and [domains/staff-profiles.md](./domains/staff-profiles.md).
- **`lineOfBusinessEnum`** (`line_of_business`: `CORPORATE`, `CORE`, `FINTECH`, `COMMERCE`, `DESIGN`) is a **shared/global enum**, not a staff-specific type. It's carried by **three entities: `staff_employment`, `opportunities`, and `projects`** (all NOT NULL) ‚Äî the practice that a person works in / a deal belongs to / an engagement bills under. It is **no longer on `project_roles`** (moved up to the project ‚Äî [ADR 0025](./decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)). Its value tuple + labels live in a pure, client-importable module **`src/lib/line-of-business.ts`** (`LINE_OF_BUSINESS`, `LINE_OF_BUSINESS_LABELS`), the single source the `pgEnum` in `staff-schema.ts` and the CRM/projects zod schemas + forms all import ‚Äî the same single-source pattern opportunities uses for its `source`/`status` enums (see [ADR 0016](./decisions/0016-junction-table-and-shared-enum-conventions.md)).
- **Local-only CSV import tools** are the only things that write the staff domain so far: `/admin/upload-staff` (Rippling export ‚Üí `staff` + effective-dated `staff_employment`) and `/admin/upload-pto` (Rippling leave export ‚Üí `staff_pto`). See [domains/staff-profiles.md](./domains/staff-profiles.md).
- **CRM slice** (`companies` + `contacts` + `contact_entries` + the shared `crm_entry_kind` enum in `src/lib/db/crm-schema.ts`; `opportunities` + its four junction tables + `opportunity_entries` + `source`/`status` pgEnums now in **`src/lib/db/opportunities-schema.ts`** ‚Äî split out of `crm-schema.ts`, [ADR 0025](./decisions/0025-line-of-business-on-opportunity-and-project-not-role.md); both barrelled by `schema.ts`):
  - **`companies`** ‚Äî organisations we deal with. `isPartner` is a **standalone boolean** flag (default false) marking partners ‚Äî not a client/partner split; a company need be neither. `name` notNull, optional `websiteUrl`, and an optional **`ownerId`** ‚Äî the accountable staff owner (FK ‚Üí `staff`, `onDelete: set null`; a second CRM ‚Üî staff link, deliberately set-null not cascade). We chose *Company* over the narrower *Client* the spine originally named ‚Äî see [ADR 0015](./decisions/0015-crm-company-over-client.md).
  - **`contacts`** ‚Äî people, optionally employed by a company (`companyId` FK ‚Üí `companies`, `onDelete: set null`; null when unknown or once the company is removed). `firstName`/`lastName`/`email` notNull, **`email` unique**, optional `phone`, optional free-text `role`, optional `linkedinUrl`, an optional **`managerId`** ‚Äî the repo's **first self-referential FK** (‚Üí `contacts.id`, `onDelete: set null`), "managed by" another contact, constrained app-side to a contact at the same company (see [ADR 0022](./decisions/0022-contact-manager-self-reference.md)) ‚Äî and an optional **`ownerId`** ‚Üí `staff` (`onDelete: set null`), the accountable staff owner (like `companies.ownerId`).
  - **`opportunities`** ‚Äî pipeline deals, each **required** to belong to a company (`companyId` FK ‚Üí `companies`, **`onDelete: restrict`** ‚Äî contrast the contacts' set-null; a company with live opportunities can't be deleted). `source` + `status` pgEnums (`status` is a **flat 14-value leaf enum in pipeline order**; the two-level kanban grouping is derived in code in `src/lib/opportunity-pipeline.ts`, not in the DB), a **required `lineOfBusiness`** (the shared enum ‚Äî which practice the deal belongs to; a project created from it defaults to the same, [ADR 0025](./decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)), a nullable **`projectId`** (the CRM ‚Üí delivery link ‚Äî **many opps ‚Üí one project**, FK ‚Üí `projects`, `onDelete: restrict`; see the Projects slice below) and **`position`** (`doublePrecision`, a single global fractional index for manual kanban ordering ‚Äî a drag writes the midpoint between neighbours, one-row update; index `opportunities_status_position_idx` on `(status, position)`). The old scalar `nextSteps` column is **gone** ‚Äî replaced by the `opportunity_entries` log ([ADR 0030](./decisions/0030-crm-timestamped-entries-notes-next-steps.md)). See [ADR 0021](./decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md). Their related people (contacts, owners, referral sources) live in **four junction tables** ‚Äî the repo's first many-to-many pattern (see [below](#junction-tables--the-first-many-to-many-pattern)). `owners` and `source_staff` link to **`staff`**, the first CRM ‚Üî staff FK.
  - **`contact_entries` / `opportunity_entries`** ‚Äî two running logs of authored, timestamped free-text entries per parent: **notes** (longer prose) and **next steps** (short reminders), distinguished by a **shared `crm_entry_kind` pgEnum** (`note`/`next_step`). One **concrete table per parent** (a real FK, `onDelete: cascade` ‚Äî no polymorphic FK); same shape (`kind`, `body`, `authorStaffId` FK ‚Üí `staff` set-null = attribution not ownership, timestamps, index on `(<parent>Id, kind, createdAt)`). **Append logs** (like `feedback`, unlike the upserted `responses`), any `crm.edit` holder may edit any entry. The latest next step is surfaced in the contacts table / kanban card via a `DISTINCT ON` subquery. See [ADR 0030](./decisions/0030-crm-timestamped-entries-notes-next-steps.md).
  - Open reads + `crm.edit`-gated create **and edit** mutations (one capability gates all CRM writes ‚Äî companies, contacts *and* opportunities) + `/companies`, `/contacts` and `/opportunities` pages exist (companies and contacts are on **separate pages** now). Opportunities now have a three-tab **detail drawer** (an **Info** tab whose fields edit **one at a time in place** via `updateOpportunityField` ‚Äî company/status/line of business/owners/source/contacts/referral sources; a **Notes** tab holding the notes & next-steps logs; and a **Project plan** tab that is a **weekly Gantt-like planner** over the linked project's roles). The **Opportunity ‚Üí Project handoff** links `opportunities.projectId` (**many opps ‚Üí one project**) when a project is created from or associated to an opportunity, and a delivery-stage rule requires one before advancing ([ADR 0024](./decisions/0024-opportunity-project-handoff-and-placeholder-roles.md), [ADR 0031](./decisions/0031-opportunity-project-planner-and-role-status.md)). See [domains/crm.md](./domains/crm.md).
- **Projects slice** (`src/lib/db/projects-schema.ts`, barrelled by `schema.ts`; imports `opportunities` from `./opportunities-schema`):
  - **`projects`** ‚Äî billable work that always belongs to a company (`companyId` FK ‚Üí `companies`, **`onDelete: restrict`** ‚Äî like opportunities, a company with live projects can't be deleted). `name` notNull, a **required project-level `lineOfBusiness`** (the shared enum ‚Äî which practice bills the engagement; defaulted from the originating opportunity, NOT per-role), a **required lifecycle `status`** (`projectStatusEnum`: `tentative`/`confirmed`/`paused`/`cancelled`, NOT NULL, DB default `tentative`; tuple + labels in the pure module `src/lib/project-status.ts`, single-source like `line-of-business.ts` ‚Äî set at create, **no edit flow yet**). id prefix `proj`. The CRM ‚Üí delivery link **lives on the opportunity now**, not here: `opportunities.projectId` (nullable FK ‚Üí `projects`, **`onDelete: restrict`**), **many opportunities ‚Üí one project** ‚Äî a project can be built up from several deals (an original + extensions / change requests). The old `projects.opportunityId` column + its partial unique index were **removed** in the inversion, and **same-company is now server-enforced** at the link entry points (`associateOpportunityProject` + the company-scoped `searchProjects`). See [ADR 0019](./decisions/0019-project-opportunity-link.md), [ADR 0031](./decisions/0031-opportunity-project-planner-and-role-status.md).
  - **`project_delivery_managers`** ‚Äî junction (many staff per project) following the CRM junction convention exactly ([ADR 0016](./decisions/0016-junction-table-and-shared-enum-conventions.md)): surrogate `text` PK, `unique(projectId, staffId)`, index on `staffId`, both FKs cascade.
  - **`project_roles`** ‚Äî a **staffing line** (NOT a pure junction ‚Äî carries columns): `projectId` ‚Üí projects (cascade), **`staffId` ‚Üí staff (restrict, NULLABLE)**, `roleType` (NOT NULL, new `projectRoleTypeEnum`), optional `name`, `startDate`/`endDate` (date, string mode), `hoursPerDay` (numeric(4,2), default 8), plus two provenance/planning columns: **`status`** (`projectRoleStatusEnum`: `tentative`/`confirmed`, NOT NULL, DB default `tentative` ‚Äî soft plan vs. committed, tuple + labels in the pure module `src/lib/project-role-status.ts`) and **`opportunityId`** (nullable FK ‚Üí `opportunities`, `onDelete: set null`, indexed ‚Äî which deal created the role). **No `lineOfBusiness`** ‚Äî that moved up to the project ([ADR 0025](./decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)). Indexed on all three FKs; **no `unique` on the FK pair** (a person can hold multiple lines on one project). A **null `staffId` is a placeholder / open position** defined before it's staffed (identified by `roleType` + optional `name`); dates and hours are required on every role. `status`/`opportunityId` are **server-controlled provenance, never user input**; a role auto-confirms when its opportunity is won ([ADR 0031](./decisions/0031-opportunity-project-planner-and-role-status.md)). **`roleType` (discipline ‚Äî `ENGINEER`/`DESIGNER`/`ARCHITECT`/`QA`/`SPECIALIST`) is orthogonal to the project's `lineOfBusiness`** (what kind of work vs. which practice bills it); its tuple + labels live in the pure module `src/lib/project-role-type.ts` (mirrors `line-of-business.ts`). This is the **first concrete cut of the proposed Allocation entity** ‚Äî stored as simple mutable rows, *not* effective-dated history. See [domains/projects.md](./domains/projects.md), [domains/allocations.md](./domains/allocations.md), [ADR 0017](./decisions/0017-project-roles-as-first-allocation-cut.md), and [ADR 0031](./decisions/0031-opportunity-project-planner-and-role-status.md).
  - **The Opportunity ‚Üí Project link is populated on the opportunity side** (`opportunities.projectId`, **many opps ‚Üí one project**). `createProject` sets the originating opportunity's `projectId` (and tags created roles with that opportunity + `tentative`); `associateOpportunityProject` links an opportunity to an **existing** project (same-company enforced). A `requiresProject(status)` rule still blocks advancing an opportunity into delivery stages (Allocating+) without one, and at most one project per opportunity (single-valued FK). See [ADR 0019](./decisions/0019-project-opportunity-link.md), [ADR 0024](./decisions/0024-opportunity-project-handoff-and-placeholder-roles.md), [ADR 0031](./decisions/0031-opportunity-project-planner-and-role-status.md).
  - Open reads + `projects.edit`-gated writes + `/projects` page exist. Projects are **create + top-level edit** (`updateProject` edits name, line of business, status, and delivery managers from the planner ‚Äî **no company edit, no delete**), and **role CRUD exists via the opportunity planner** (`createProjectRole`/`updateProjectRole`/`deleteProjectRole`/`extendProjectRole` + `associateOpportunityProject`), scoped to this opportunity's tentative roles. See [domains/projects.md](./domains/projects.md).
- **Performance slice** (`src/lib/db/performance-schema.ts`, barrelled by `schema.ts`):
  - **`feedback`** ‚Äî peer feedback: `fromStaffId` + `toStaffId` (both FK ‚Üí `staff.id`, both `onDelete: cascade`, each indexed), a 5-point `feedback_rating` pgEnum, required `context`, optional `keepDoing`/`stopDoing`/`startDoing` (at least one required, enforced in the zod schema)/`other`, and an optional `messageToRecipient`. **Point-in-time** (an immutable event log ‚Äî NOT effective-dated; a person may leave feedback about the same person repeatedly, so **no unique `(from, to)`**). The rating enum's values/labels live in the pure, client-importable **`src/lib/feedback-rating.ts`** (same single-source pattern as `line-of-business.ts`). Privacy is enforced **by the read projections, not the table** ‚Äî three tiers (give = open to active staff via the `authorizeFeedbackCreate` hook; recipient sees only giver name + `messageToRecipient`; `feedback.review` = manager/admin see all in full). See [domains/performance.md](./domains/performance.md) and [ADR 0023](./decisions/0023-feedback-privacy-tiers.md).
  - The Performance domain's second slice ‚Äî the **compensation analytics dashboard** (`/performance`) ‚Äî **adds no table**: it aggregates the latest `staff_employment` comp per active staff member, anonymized, gated by `staff.viewCompensation`. Cross-currency comparison normalizes via live FX rates ([ADR 0029](./decisions/0029-external-fx-rates-and-currency-normalization.md)). See [domains/performance.md](./domains/performance.md).
- **Timesheets slice** (`src/lib/db/timesheets-schema.ts`, barrelled by `schema.ts`):
  - **`timesheets`** ‚Äî one person's week: `staffId` ‚Üí `staff` (cascade), **`weekStartDate`** (`date`, ISO Monday), `status` (`timesheet_status`: `draft`|`submitted`), `submittedAt` (nullable). **`unique(staffId, weekStartDate)`** ‚Äî one sheet per person per week; the row is created **lazily** on first save/submit. id prefix `ts`.
  - **`time_entries`** ‚Äî per-day rows: `timesheetId` ‚Üí `timesheets` (cascade), `date`, `hours` (`numeric(4,2)`), targeting **exactly one** of `projectId` ‚Üí `projects` (**`restrict`**, nullable) or a non-billable `category` (`time_entry_category`: `PTO`/`UNALLOCATED_BENCH`/`INTERNAL_ADMIN`). The XOR is enforced by a DB `CHECK` (`time_entries_target_check`) *and* zod. id prefix `te`. Category values live in the pure module `src/lib/timesheet-category.ts` (single-source pattern, like `line-of-business.ts`); the PTO bucket is **independent of `staff_pto`** (no sync).
  - These are the **actuals** complementing the `project_roles` allocation plan. Save is a whole-week transactional replace; submit locks the week (`timesheets.edit` capability edits any/locked week, bypassing the ¬±1-week owner edit window). No approval or billing yet. See [domains/timesheets.md](./domains/timesheets.md) and [ADR 0027](./decisions/0027-timesheet-weekly-model-and-edit-window.md).
- Nothing else (a full Allocation domain, review cycles/goals) exists yet. (The legacy `staff_profile` example table is long gone ‚Äî it predates the squashed `0000_light_shape.sql` baseline, which never creates it.)

## Core entities

- **Person** ‚Äî an employee/contractor. Anchors most of the system. Realized as **`staff`** (durable identity) + **`staff_employment`** (effective-dated role/billability facts); login identity is the auth `user`, linked via the optional `staff.userId` (1:1, null until first sign-in).
  - **Skill** ‚Äî capability + proficiency. _(Realized, but **not** as the many:many table this originally proposed:_ an inline `jsonb` array on `staff` (`skills`), catalogue-driven from `src/lib/skills.ts`. The staff directory now **filters people by skill (+ minimum level) client-side** over the once-fetched list ‚Äî no indexed query. Cross-staff skill matching for *allocations* is still proposed and would need an indexed rework ‚Äî see [ADR 0018](./decisions/0018-skills-inline-jsonb-catalogue.md).)
  - **Compensation** (base/hourly/bonuses + currency) is **realized** as effective-dated columns on `staff_employment` itself, not a separate table ‚Äî import-only, required, view-gated. See [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md). External charge rates (what we bill a client) are still proposed and would follow the same history-as-rows pattern.
- **Company** ‚Äî an organisation we deal with (CRM); an `isPartner` flag marks partners (standalone, not a client/partner split), plus an optional staff **owner** (`ownerId`, set-null). _(Realized ‚Äî `companies`; create/read/edit.)_ Chosen over the narrower *Client* the spine first named; see [ADR 0015](./decisions/0015-crm-company-over-client.md).
  - **Contact** (optional company per contact) ‚Äî a person, optionally at a Company, optionally "managed by" another contact at the same company (self-referential `managerId`), with an optional staff **owner** (`ownerId`, set-null). _(Realized ‚Äî `contacts`; create/read/edit; `companyId` nullable, `onDelete: set null`; `managerId` nullable self-ref, `onDelete: set null`, same-company invariant app-enforced ‚Äî see [ADR 0022](./decisions/0022-contact-manager-self-reference.md).)_
  - **Opportunity** ‚Äî a pipeline deal that always belongs to a Company; when it reaches delivery stages, links to a Project. _(Realized ‚Äî `opportunities` + junction tables; `companyId` required, `onDelete: restrict`. The Project link is now **`opportunities.projectId`** (nullable FK, `restrict`, at most one project per opportunity, but **many opps ‚Üí one project**), populated by the handoff/planner flow ‚Äî see [ADR 0019](./decisions/0019-project-opportunity-link.md), [ADR 0024](./decisions/0024-opportunity-project-handoff-and-placeholder-roles.md), [ADR 0031](./decisions/0031-opportunity-project-planner-and-role-status.md).)_
- **Project** (a.k.a. engagement) ‚Äî billable work for a Company. The hub linking CRM ‚Üî delivery, and now a **shared delivery vehicle** several opportunities can feed roles into. _(Realized (first slice) ‚Äî `projects`; required `companyId`, `onDelete: restrict`; the opportunity link lives on `opportunities.projectId` (**many opps ‚Üí one project**), populated when a project is created from or associated to an opportunity; delivery managers via `project_delivery_managers`; a lifecycle `status` (`tentative`/`confirmed`/`paused`/`cancelled`, default `tentative`) set at create and **editable post-create** via `updateProject` (the planner's Edit-project dialog, which also edits name/line of business/delivery managers); roles are editable via the opportunity planner. Company can't be changed after create, and there's no project-delete.)_ See [domains/projects.md](./domains/projects.md).
- **Allocation** ‚Äî a _time-ranged_ assignment of a Person to a Project (start/end, % or hours, project role). The heart of capacity planning. _(Partially realized ‚Äî `project_roles` is the first cut: a staff member (or a **placeholder/open position** when `staffId` is null) of a given role type for a date range at N hours/day (line of business lives on the parent project, not the role), carrying a **`status`** (`tentative` ‚Üí `confirmed`, auto-confirmed on the opportunity's win = soft vs. hard) and an **`opportunityId`** provenance FK. Simple rows, not effective-dated history yet ‚Äî see [ADR 0017](./decisions/0017-project-roles-as-first-allocation-cut.md), [ADR 0031](./decisions/0031-opportunity-project-planner-and-role-status.md), [ADR 0025](./decisions/0025-line-of-business-on-opportunity-and-project-not-role.md).)_
- **TimeEntry** ‚Äî hours a Person logged against a Project (or a non-billable bucket) on a date. Grouped into a **Timesheet** = one Person's week (`draft`‚Üí`submitted`). _(Realized ‚Äî `time_entries` + `timesheets`; per-day rows, whole-week save, submit-locks. Approval + billing deferred ‚Äî see [domains/timesheets.md](./domains/timesheets.md) and [ADR 0027](./decisions/0027-timesheet-weekly-model-and-edit-window.md).)_
- **Feedback** ‚Äî point-in-time peer feedback from one Person about another (rating + context + keep/stop/start + optional recipient message). _(Realized ‚Äî `feedback`; staff‚Üîstaff, tiered visibility ‚Äî see [domains/performance.md](./domains/performance.md) and [ADR 0023](./decisions/0023-feedback-privacy-tiers.md).)_
- **ReviewCycle / PerformanceReview / Goal** ‚Äî periodic assessment of a Person, often informed by their project work and utilization. _(Proposed.)_

## Key derived concepts

- **Utilization** = billable hours √∑ available hours for a Person over a period. Drives capacity (for allocations) and feeds performance.
- **Forecast vs. actuals** ‚Äî Allocations (`project_roles`) are the _plan_; TimeEntries (now realized) are the _actuals_. Both exist; **reconciling them is still unbuilt** and a central reporting need.
- **Margin** = (charge rate ‚àí cost rate) √ó billable hours, per Project/Person.

## Nuances to respect

- Everything time-bound (allocations, rates, reviews, **employment**) needs effective-dated handling ‚Äî a person's role or rate changes over time. The realized pattern is **history-as-rows** (`staff_employment`): a new row per change, current state = latest `effectiveFromDate`. See [ADR 0007](./decisions/0007-staff-employment-effective-dating.md). **Two deliberate exceptions:** (1) **Skills** are stored inline and edited in place, with no per-skill history ‚Äî see [ADR 0018](./decisions/0018-skills-inline-jsonb-catalogue.md); (2) the first Allocation cut (`project_roles`) also stores simple mutable rows rather than effective-dated history, so it can't reconstruct a past plan ‚Äî a deliberate scope call for the first slice, see [ADR 0017](./decisions/0017-project-roles-as-first-allocation-cut.md).
- Rates and salaries are highly sensitive. **Compensation** is now realized (on `staff_employment`) and access-gated by `staff.viewCompensation` (own always visible) ‚Äî see [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md), authorization in [architecture.md](./architecture.md), and the RBAC model in [domains/permissions.md](./domains/permissions.md).
- **Dates and times are timezone-agnostic.** Calendar dates (effective-from, PTO spans, join/termination) are `date()` strings (`"YYYY-MM-DD"`, no zone); instants are plain `timestamp` _without_ time zone ‚Äî treat all stored datetimes as wall-clock and convert zones only at the UI edge. See [`.claude/rules/database.md`](../.claude/rules/database.md).

### Junction tables ‚Äî the first many-to-many pattern

The four `opportunity_*` join tables (`src/lib/db/opportunities-schema.ts`) are the repo's **first** many-to-many relations and set the convention future join tables should copy (`project_delivery_managers` follows it exactly; `project_roles` follows the FK/index parts but is a **data-carrying** line, not a pure junction ‚Äî see [domains/projects.md](./domains/projects.md)):

- **Surrogate `text` PK** via `generateId(prefix)` (e.g. `opp-contact`), not a composite PK ‚Äî consistent with the app-minted-CUID2 ID convention everywhere else ([`.claude/rules/database.md`](../.claude/rules/database.md)).
- **`unique(...)` on the FK pair** ‚Äî gives set-semantics (a row can't be linked twice). Writers still dedupe input id arrays before insert so a repeat can't trip this index (see `createOpportunity`).
- **`index(...)` on the *non-owning* FK** (e.g. the `contactId`/`staffId` side) for efficient reverse lookups ("which opportunities is this contact on?").
- **Both FKs `onDelete: "cascade"`** ‚Äî a link row is meaningless without both endpoints.

Reading the "other side" cheaply: resolve the linked names with a **single grouped follow-up query** over the relevant ids, not a per-row query (see `getOpportunitiesBoard`'s owner-name resolution) ‚Äî avoid N+1.
