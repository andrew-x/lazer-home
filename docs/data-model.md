# Data model — the spine

The five domains share one model. This is the most important doc for understanding the system: changes here ripple everywhere.

**Status: mostly proposed.** Entities and relationships below are the intended design. Realized so far: the **Person** anchor (auth `user`), the **Staff profiles** tables (`staff`, `staff_employment`, `staff_pto`), the **CRM** slice (`companies`, `contacts`, `opportunities` + four opportunity junction tables + the `contact_entries`/`company_entries`/`opportunity_entries` note & next-step logs), the **Projects** slice (`projects`, `project_delivery_managers`, `project_roles` — the first cut of Allocation), the **Timesheets** slice (`timesheets`, `time_entries` — weekly actuals), the first **Performance** slices (`feedback` — peer feedback; `staff_rating` — overall levels L0–L4), and a generic **`responses`** survey store (two consumers: the staff-profile **Manual of Me** and **Ways of Working** surveys) — see "What's realized in code".

## What's realized in code

> **Migrations have been squashed more than once — treat the filenames as disposable.** The whole drizzle history most recently collapsed into a single fresh baseline **`drizzle/0000_lethal_rictor.sql`** (commit `ba60c36`, superseding the earlier `0000_light_shape.sql` baseline and the incrementals that had accumulated on it — the CRM entries tables, the Opportunity↔Project inversion, and `staff_rating`, all now folded into the baseline). The substance of every past migration lives in `0000` and the schema modules under `src/lib/db/`. **Three incremental migrations** currently sit on top (in apply order): **`drizzle/0001_past_agent_zero.sql`** (adds `contacts.relationship_strength` — see [domains/crm.md](./domains/crm.md#relationship-strength)), then **`drizzle/0002_gray_corsair.sql`** (the LoB-to-role + derived-project-status change — adds `paused`/`cancelled` to `project_role_status`, adds `project_roles.line_of_business` (backfilled from the parent project), then drops `projects.status`/`projects.line_of_business` and the `project_status` type — [ADR 0033](./decisions/0033-line-of-business-on-role-derived-project-status.md)), then **`drizzle/0003_gifted_kylun.sql`** (a single `RENAME COLUMN` on `project_roles`: the role's optional free-text label `name` → `description`, still nullable text, max 200 in the schema), then **`drizzle/0004_blue_terrax.sql`** (adds `company_entries` — the company-level notes log, mirroring `contact_entries`; see [domains/crm.md](./domains/crm.md#timestamped-entries-notes--next-steps)), then **`drizzle/0005_secret_zarek.sql`** (adds a nullable **`location`** text column to `companies`, `contacts`, and `staff` — the free-text `"City, CC"` home-base label, see [Location](#location-free-text)). Below we cite the **schema file** for each slice rather than a migration.

- **Person** is realized by better-auth's **`user`** table (`src/lib/db/auth-schema.ts`) — the _login/identity_ anchor. (`user`/`session`/`account`/`verification` live in our schema and migrations, not better-auth's.) A logged-in user maps to its staff record via the nullable, unique **`staff.userId`** FK (`onDelete: set null`) — see [domains/staff-profiles.md](./domains/staff-profiles.md). The user's **`role`** (a `text()` column owned by better-auth's `auth-schema.ts`) drives RBAC; valid values and the role→permission matrix live in `src/lib/auth/permissions.ts`, validated app-side by `roleSchema` (a `pgEnum` would be clobbered by `auth:generate`). See [domains/permissions.md](./domains/permissions.md).
- **Staff profiles domain** (`src/lib/db/staff-schema.ts`, barrelled by `src/lib/db/schema.ts`):
  - **`staff`** — the durable person record (notNull/unique external `ripplingId`, name, non-unique email, optional `userId` link to auth `user`, an optional self-referential **`managerId`** → `staff.id` (`onDelete: set null`) — "reports to", populated **exclusively by the staff CSV import** (matched on the `Manager - Work email` column) with **no in-app editor**, so a re-import never fights a manual edit; it mirrors `contacts.managerId` (which remains the repo's *first* self-referential FK) — see [ADR 0026](./decisions/0026-staff-manager-self-reference.md), profile links, `clientIntro` (+ `clientIntroUpdatedAt`), free-text `resume` (+ `resumeUpdatedAt`), an inline **`skills`** jsonb list, lifecycle dates, `isActive`). Both `*UpdatedAt` instants are stamped **explicitly by their update action** when the text changes, not via `$onUpdate` (which would fire on every row write, e.g. an import re-sync). `skills` is the schema's **first jsonb/array column** — `StaffSkill[]` (`{ name, level }[]`, NOT NULL default `[]`) picked from the hardcoded catalogue in `src/lib/staff/skills.ts`, deliberately **not** a normalized `skill` table; see [ADR 0018](./decisions/0018-skills-inline-jsonb-catalogue.md).
  - **`staff_employment`** — time-varying employment facts (line of business, role, employment type, billability, `utilizationTarget`, `billableType` (NOT NULL, default `HUB`), a boolean `isManagement`, and **compensation** — `base`/`hourlyRate`/`guaranteedBonus`/`discretionaryBonus` (`numeric(12,2)`, all **NOT NULL**) + `currency` (**NOT NULL**)), as **one effective-dated row per change**; current state = latest `effectiveFromDate`. FK → `staff.id` cascade. `isManagement` and `billableType` are **orthogonal to `role`** (someone can hold a working role *and* be management for it) and set in-app only. **Compensation lives here** rather than a separate table (comp changes coincide with employment changes) — it's **import-only** (no edit UI), **required** (the importer skips a row it can't supply comp for; **no carry-forward on the *import* path** — every imported row must carry full comp), and **view-gated** by `staff.viewCompensation` (own comp always visible). The no-carry-forward rule is specific to the import path; the **bulk-edit path** (a *non-comp* change spawning a new effective-dated row) *does* **carry comp forward** from the previous row, so advancing someone's role/LoB never clears their salary. See [ADR 0007](./decisions/0007-staff-employment-effective-dating.md), [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md), and [domains/staff-profiles.md](./domains/staff-profiles.md).
  - **`staff_pto`** — leave spans (`startDate`/`endDate`, `type`, `isPending`, **notNull/unique `ripplingId`** = Rippling "Leave request ID"). FK → `staff.id` cascade. Discrete rows, **not** effective-dated; reduces a person's available capacity for allocations. Populated by a second local-only importer (`/admin/upload-pto`) — see [domains/staff-profiles.md](./domains/staff-profiles.md).
- **`responses`** — a **generic survey-answer store** (`src/lib/db/responses-schema.ts`, barrelled by `schema.ts`), keyed by **`(staffId, questionId)`** with a `unique` on the pair (→ upsert, one **current** answer per person per question — *not* history like `staff_employment`, and the opposite of the append-only `feedback` log). FK `staffId` → `staff.id` cascade. `questionId` is plain **`text`** validated in app code (**not a `pgEnum`**), so a new survey needs **no migration**. Three nullable payload shapes (`listResponse` jsonb `string[]`, `textResponse`, `jsonResponse`) — a question uses whichever fits, and `upsertResponse` writes the used column and **nulls the other** so a row never carries a stale value from the wrong shape. **Two consumers today**, both staff-profile surveys and both a code-only add (no migration — exactly what ADR 0028 anticipated): **Manual of Me** (`src/lib/staff/manual-of-me.ts`, 7 free-text ids → `textResponse` only) and **Ways of Working** (`src/lib/staff/ways-of-working.ts`, 28 `WOW_`-prefixed ids → `textResponse` for free-text/single-select, `listResponse` for multi-selects and the AI usage/savings matrix buckets). The write schema's `questionId` is now `z.union([manualOfMeQuestionId, waysOfWorkingQuestionId])`; each new survey adds its enum to that union. See [ADR 0028](./decisions/0028-generic-responses-table-app-validated-question-ids.md) and [domains/staff-profiles.md](./domains/staff-profiles.md).
- **`lineOfBusinessEnum`** (`line_of_business`: `CORPORATE`, `CORE`, `FINTECH`, `COMMERCE`, `DESIGN`) is a **shared/global enum**, not a staff-specific type. It's carried by **three entities: `staff_employment`, `opportunities`, and `project_roles`** (all NOT NULL) — the practice that a person works in / a deal belongs to / a staffing line bills under. It **moved back onto `project_roles`** and is **no longer on `projects`** (a project's LoBs are *derived* from its roles, so one project can span practices — [ADR 0033](./decisions/0033-line-of-business-on-role-derived-project-status.md), reversing the project-level placement of [ADR 0025](./decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)). Its value tuple + labels live in a pure, client-importable module **`src/lib/crm/line-of-business.ts`** (`LINE_OF_BUSINESS`, `LINE_OF_BUSINESS_LABELS`), the single source the `pgEnum` in `staff-schema.ts` and the CRM/projects zod schemas + forms all import — the same single-source pattern opportunities uses for its `source`/`status` enums (see [ADR 0016](./decisions/0016-junction-table-and-shared-enum-conventions.md)).
- **Local-only CSV import tools** are the only things that write the staff domain so far: `/admin/upload-staff` (Rippling export → `staff` + effective-dated `staff_employment`) and `/admin/upload-pto` (Rippling leave export → `staff_pto`). See [domains/staff-profiles.md](./domains/staff-profiles.md).
- **CRM slice** (`companies` + `contacts` + `contact_entries` + `company_entries` + the shared `crm_entry_kind` enum in `src/lib/db/crm-schema.ts`; `opportunities` + its four junction tables + `opportunity_entries` + `source`/`status` pgEnums now in **`src/lib/db/opportunities-schema.ts`** — split out of `crm-schema.ts`, [ADR 0025](./decisions/0025-line-of-business-on-opportunity-and-project-not-role.md); both barrelled by `schema.ts`):
  - **`companies`** — organisations we deal with. `isPartner` is a **standalone boolean** flag (default false) marking partners — not a client/partner split; a company need be neither. `name` notNull, optional `websiteUrl`, and an optional **`ownerId`** — the accountable staff owner (FK → `staff`, `onDelete: set null`; a second CRM ↔ staff link, deliberately set-null not cascade). We chose *Company* over the narrower *Client* the spine originally named — see [ADR 0015](./decisions/0015-crm-company-over-client.md). A company's **status (Partner / Client / Prospect) is *derived*, not stored** — `isPartner` is the only stored relationship signal; Client (a confirmed project) and Prospect (an open opportunity) are computed from the pipeline ([ADR 0034](./decisions/0034-company-status-derived-tags.md), [domains/crm.md](./domains/crm.md#company-status-tags)).
  - **`contacts`** — people, optionally employed by a company (`companyId` FK → `companies`, `onDelete: set null`; null when unknown or once the company is removed). `firstName`/`lastName`/`email` notNull, **`email` unique**, optional `phone`, optional free-text `role`, optional `linkedinUrl`, an optional **`managerId`** — the repo's **first self-referential FK** (→ `contacts.id`, `onDelete: set null`), "managed by" another contact, constrained app-side to a contact at the same company (see [ADR 0022](./decisions/0022-contact-manager-self-reference.md)), an optional **`ownerId`** → `staff` (`onDelete: set null`), the accountable staff owner (like `companies.ownerId`), and an optional **`relationshipStrength`** — a plain **`integer`** on a **1–5** scale (1 New/Unestablished … 5 Champion/Trusted Partner), **null = not yet rated**. Level meanings live in the pure, client-importable module `src/lib/crm/relationship-strength.ts` (mirrors `feedback-rating.ts`); edited inline as a star rating on the contact page.
  - **`opportunities`** — pipeline deals, each **required** to belong to a company (`companyId` FK → `companies`, **`onDelete: restrict`** — contrast the contacts' set-null; a company with live opportunities can't be deleted). `source` + `status` pgEnums (`status` is a **flat 14-value leaf enum in pipeline order**; the two-level kanban grouping is derived in code in `src/lib/crm/opportunity-pipeline.ts`, not in the DB), a **required `lineOfBusiness`** (the shared enum — which practice the deal belongs to; a project created from it defaults to the same, [ADR 0025](./decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)), a nullable **`projectId`** (the CRM → delivery link — **many opps → one project**, FK → `projects`, `onDelete: restrict`; see the Projects slice below) and **`position`** (`doublePrecision`, a single global fractional index for manual kanban ordering — a drag writes the midpoint between neighbours, one-row update; index `opportunities_status_position_idx` on `(status, position)`). The old scalar `nextSteps` column is **gone** — replaced by the `opportunity_entries` log ([ADR 0030](./decisions/0030-crm-timestamped-entries-notes-next-steps.md)). See [ADR 0021](./decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md). Their related people (contacts, owners, referral sources) live in **four junction tables** — the repo's first many-to-many pattern (see [below](#junction-tables--the-first-many-to-many-pattern)). `owners` and `source_staff` link to **`staff`**, the first CRM ↔ staff FK.
  - **`contact_entries` / `company_entries` / `opportunity_entries`** — running logs of authored, timestamped free-text entries per parent: **notes** (longer prose) and **next steps** (short reminders), distinguished by a **shared `crm_entry_kind` pgEnum** (`note`/`next_step`). One **concrete table per parent** (a real FK, `onDelete: cascade` — no polymorphic FK); same shape (`kind`, `body`, `authorStaffId` FK → `staff` set-null = attribution not ownership, timestamps, index on `(<parent>Id, kind, createdAt)`). **Append logs** (like `feedback`, unlike the upserted `responses`), any `crm.edit` holder may edit any entry. Contacts & opportunities use both kinds; **companies use only `note`** in the UI (the table still carries both for shape parity). The latest next step is surfaced in the contacts table / kanban card via a `DISTINCT ON` subquery. See [ADR 0030](./decisions/0030-crm-timestamped-entries-notes-next-steps.md).
  - Open reads + `crm.edit`-gated create **and edit** mutations (one capability gates all CRM writes — companies, contacts *and* opportunities) + `/companies`, `/contacts` and `/opportunities` pages exist (companies and contacts are on **separate pages** now). Opportunities now have a three-tab **detail drawer** (an **Info** tab whose fields edit **one at a time in place** via `updateOpportunityField` — company/status/line of business/owners/source/contacts/referral sources; a **Notes** tab holding the notes & next-steps logs; and a **Project plan** tab that is a **weekly Gantt-like planner** over the linked project's roles). The **Opportunity → Project handoff** links `opportunities.projectId` (**many opps → one project**) when a project is created from or associated to an opportunity, and a delivery-stage rule requires one before advancing ([ADR 0024](./decisions/0024-opportunity-project-handoff-and-placeholder-roles.md), [ADR 0031](./decisions/0031-opportunity-project-planner-and-role-status.md)). See [domains/crm.md](./domains/crm.md).
- **Projects slice** (`src/lib/db/projects-schema.ts`, barrelled by `schema.ts`; imports `opportunities` from `./opportunities-schema`):
  - **`projects`** — billable work that always belongs to a company (`companyId` FK → `companies`, **`onDelete: restrict`** — like opportunities, a company with live projects can't be deleted). **Deliberately thin**: just `name` (notNull), `companyId`, timestamps. id prefix `proj`. It has **no stored `status` and no stored `lineOfBusiness`** — both were **dropped** and are now **derived from its roles** (`src/lib/projects/project-derived.ts`: `deriveProjectStatus` "least-committed wins" over role statuses; `deriveProjectLinesOfBusiness` = distinct role LoBs), so a project can span practices ([ADR 0033](./decisions/0033-line-of-business-on-role-derived-project-status.md); the `project_status` pgEnum and `project-status.ts` are gone). The CRM → delivery link lives on the opportunity: `opportunities.projectId` (nullable FK → `projects`, **`onDelete: restrict`**), **many opportunities → one project**. **Same-company is server-enforced** at the link entry points (`associateOpportunityProject` + the company-scoped `searchProjects`). See [ADR 0019](./decisions/0019-project-opportunity-link.md), [ADR 0031](./decisions/0031-opportunity-project-planner-and-role-status.md), [ADR 0033](./decisions/0033-line-of-business-on-role-derived-project-status.md).
  - **`project_delivery_managers`** — junction (many staff per project) following the CRM junction convention exactly ([ADR 0016](./decisions/0016-junction-table-and-shared-enum-conventions.md)): surrogate `text` PK, `unique(projectId, staffId)`, index on `staffId`, both FKs cascade.
  - **`project_roles`** — a **staffing line** (NOT a pure junction — carries columns): `projectId` → projects (cascade), **`staffId` → staff (restrict, NULLABLE)**, `roleType` (NOT NULL, `projectRoleTypeEnum`), **`lineOfBusiness`** (NOT NULL, the shared enum — **moved back onto the role**, a project's LoBs derive from these; defaults from the opportunity in the planner — [ADR 0033](./decisions/0033-line-of-business-on-role-derived-project-status.md)), optional `description` (free-text label, nullable, max 200), `startDate`/`endDate` (date, string mode), `hoursPerDay` (numeric(4,2), default 8), plus two provenance/planning columns: **`status`** (`projectRoleStatusEnum`, NOT NULL, DB default `tentative` — now **four states** `tentative`/`confirmed`/`paused`/`cancelled`; the last two are enum-only, no UI yet; tuple + labels + badge variants in the pure module `src/lib/projects/project-role-status.ts`) and **`opportunityId`** (nullable FK → `opportunities`, `onDelete: set null`, indexed — which deal created the role). Indexed on all three FKs; **no `unique` on the FK pair** (a person can hold multiple lines on one project). A **null `staffId` is a placeholder / open position** (identified by `roleType` + optional `description`); dates and hours are required on every role. `status`/`opportunityId` are **server-controlled provenance, never user input**; a role auto-confirms when its opportunity is won ([ADR 0031](./decisions/0031-opportunity-project-planner-and-role-status.md)). **`roleType` (discipline — `ENGINEER`/`DESIGNER`/`ARCHITECT`/`QA`/`SPECIALIST`) is orthogonal to `lineOfBusiness`** (what kind of work vs. which practice bills it); its tuple + labels live in the pure module `src/lib/projects/project-role-type.ts`. This is the **first concrete cut of the proposed Allocation entity** — stored as simple mutable rows, *not* effective-dated history. See [domains/projects.md](./domains/projects.md), [domains/allocations.md](./domains/allocations.md), [ADR 0017](./decisions/0017-project-roles-as-first-allocation-cut.md), [ADR 0031](./decisions/0031-opportunity-project-planner-and-role-status.md), and [ADR 0033](./decisions/0033-line-of-business-on-role-derived-project-status.md).
  - **The Opportunity → Project link is populated on the opportunity side** (`opportunities.projectId`, **many opps → one project**). `createProject` sets the originating opportunity's `projectId` (and tags created roles with that opportunity + `tentative`); `associateOpportunityProject` links an opportunity to an **existing** project (same-company enforced). A `requiresProject(status)` rule still blocks advancing an opportunity into delivery stages (Allocating+) without one, and at most one project per opportunity (single-valued FK). See [ADR 0019](./decisions/0019-project-opportunity-link.md), [ADR 0024](./decisions/0024-opportunity-project-handoff-and-placeholder-roles.md), [ADR 0031](./decisions/0031-opportunity-project-planner-and-role-status.md).
  - Open reads + `projects.edit`-gated writes + `/projects` page exist. Projects are **create** (standalone `createProject`, or **one-click `createProjectFromOpportunity`** — no form) **+ edit** (`updateProject` edits only name + delivery managers — no company edit, and no status/LoB since those derive), **role CRUD via the opportunity planner** (`createProjectRole`/`updateProjectRole`/`deleteProjectRole`/`extendProjectRole` + `associateOpportunityProject`, scoped to this opportunity's tentative roles), and a **detach/delete** flow (`removeProjectFromOpportunity` and `deleteOpportunity` share `detachProjectFromOpportunity` — a sole-owned project is deleted, else its roles are removed and it's unlinked). See [domains/projects.md](./domains/projects.md) and [ADR 0033](./decisions/0033-line-of-business-on-role-derived-project-status.md).
- **Performance slice** (`src/lib/db/performance-schema.ts`, barrelled by `schema.ts`):
  - **`feedback`** — peer feedback: `fromStaffId` + `toStaffId` (both FK → `staff.id`, both `onDelete: cascade`, each indexed), a 5-point `feedback_rating` pgEnum, required `context`, optional `keepDoing`/`stopDoing`/`startDoing` (at least one required, enforced in the zod schema)/`other`, and an optional `messageToRecipient`. **Point-in-time** (an immutable event log — NOT effective-dated; a person may leave feedback about the same person repeatedly, so **no unique `(from, to)`**). The rating enum's values/labels live in the pure, client-importable **`src/lib/performance/feedback-rating.ts`** (same single-source pattern as `line-of-business.ts`). Privacy is enforced **by the read projections, not the table** — three tiers (give = open to active staff via the `authorizeFeedbackCreate` hook; recipient sees only giver name + `messageToRecipient`; `feedback.review` = manager/admin see all in full). See [domains/performance.md](./domains/performance.md) and [ADR 0023](./decisions/0023-feedback-privacy-tiers.md).
  - The Performance domain's second slice — the **compensation analytics dashboard** (`/performance` Compensation tab) — **adds no table**: it aggregates the latest `staff_employment` comp per active staff member, anonymized, gated by `staff.viewCompensation`. Cross-currency comparison normalizes via live FX rates ([ADR 0029](./decisions/0029-external-fx-rates-and-currency-normalization.md)). See [domains/performance.md](./domains/performance.md).
  - **`staff_rating`** — the Performance domain's third slice, an **overall performance level (L0–L4)** per person: `staffId` (FK → `staff.id` cascade, indexed), `effectiveDate` (`date`), **`level` (`integer`, NULLABLE** — null/absent = unrated, as a real historied event), `evaluatedByUserId` (FK → `user.id`, `set null`), timestamps; a DB `CHECK` (`staff_rating_level_range`) enforces `level is null or 0..4`. **Effective-dated exactly like `staff_employment`** (new dated row per change, current = latest `effectiveDate`, ties broken by `createdAt`; [ADR 0007](./decisions/0007-staff-employment-effective-dating.md)). Level values live in the pure module **`src/lib/staff/staff-rating.ts`** (mirrors `feedback-rating.ts`; [ADR 0016](./decisions/0016-junction-table-and-shared-enum-conventions.md)); the current-row ordering fragment is `latestRatingFirst`. Surfaced as a **section on the `/performance` dashboard** (sharing its filters/currency, no tabs), gated by a new **`ratings.view`/`ratings.edit`** capability — **manager/admin only, NO self-view** (stricter than compensation; not finance). See [domains/performance.md](./domains/performance.md) and [ADR 0032](./decisions/0032-staff-rating-levels-effective-dated-manager-only.md).
- **Timesheets slice** (`src/lib/db/timesheets-schema.ts`, barrelled by `schema.ts`):
  - **`timesheets`** — one person's week: `staffId` → `staff` (cascade), **`weekStartDate`** (`date`, ISO Monday), `status` (`timesheet_status`: `draft`|`submitted`), `submittedAt` (nullable). **`unique(staffId, weekStartDate)`** — one sheet per person per week; the row is created **lazily** on first save/submit. id prefix `ts`.
  - **`time_entries`** — per-day rows: `timesheetId` → `timesheets` (cascade), `date`, `hours` (`numeric(4,2)`), targeting **exactly one** of `projectId` → `projects` (**`restrict`**, nullable) or a non-billable `category` (`time_entry_category`: `PTO`/`UNALLOCATED_BENCH`/`INTERNAL_ADMIN`). The XOR is enforced by a DB `CHECK` (`time_entries_target_check`) *and* zod. id prefix `te`. Category values live in the pure module `src/lib/timesheets/timesheet-category.ts` (single-source pattern, like `line-of-business.ts`); the PTO bucket is **independent of `staff_pto`** (no sync).
  - These are the **actuals** complementing the `project_roles` allocation plan. Save is a whole-week transactional replace; submit locks the week (`timesheets.edit` capability edits any/locked week, bypassing the ±1-week owner edit window). No approval or billing yet. See [domains/timesheets.md](./domains/timesheets.md) and [ADR 0027](./decisions/0027-timesheet-weekly-model-and-edit-window.md).
- Nothing else (a full Allocation domain, review cycles/goals) exists yet. (The legacy `staff_profile` example table is long gone — it predates the squashed `0000_lethal_rictor.sql` baseline, which never creates it.)

## Core entities

- **Person** — an employee/contractor. Anchors most of the system. Realized as **`staff`** (durable identity) + **`staff_employment`** (effective-dated role/billability facts); login identity is the auth `user`, linked via the optional `staff.userId` (1:1, null until first sign-in).
  - **Skill** — capability + proficiency. _(Realized, but **not** as the many:many table this originally proposed:_ an inline `jsonb` array on `staff` (`skills`), catalogue-driven from `src/lib/staff/skills.ts`. The staff directory now **filters people by skill (+ minimum level) client-side** over the once-fetched list — no indexed query. Cross-staff skill matching for *allocations* is still proposed and would need an indexed rework — see [ADR 0018](./decisions/0018-skills-inline-jsonb-catalogue.md).)
  - **Compensation** (base/hourly/bonuses + currency) is **realized** as effective-dated columns on `staff_employment` itself, not a separate table — import-only, required, view-gated. See [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md). External charge rates (what we bill a client) are still proposed and would follow the same history-as-rows pattern.
- **Company** — an organisation we deal with (CRM); an `isPartner` flag marks partners (standalone, not a client/partner split), plus an optional staff **owner** (`ownerId`, set-null). _(Realized — `companies`; create/read/edit.)_ Chosen over the narrower *Client* the spine first named; see [ADR 0015](./decisions/0015-crm-company-over-client.md).
  - **Contact** (optional company per contact) — a person, optionally at a Company, optionally "managed by" another contact at the same company (self-referential `managerId`), with an optional staff **owner** (`ownerId`, set-null). _(Realized — `contacts`; create/read/edit; `companyId` nullable, `onDelete: set null`; `managerId` nullable self-ref, `onDelete: set null`, same-company invariant app-enforced — see [ADR 0022](./decisions/0022-contact-manager-self-reference.md).)_
  - **Opportunity** — a pipeline deal that always belongs to a Company; when it reaches delivery stages, links to a Project. _(Realized — `opportunities` + junction tables; `companyId` required, `onDelete: restrict`. The Project link is now **`opportunities.projectId`** (nullable FK, `restrict`, at most one project per opportunity, but **many opps → one project**), populated by the handoff/planner flow — see [ADR 0019](./decisions/0019-project-opportunity-link.md), [ADR 0024](./decisions/0024-opportunity-project-handoff-and-placeholder-roles.md), [ADR 0031](./decisions/0031-opportunity-project-planner-and-role-status.md).)_
- **Project** (a.k.a. engagement) — billable work for a Company. The hub linking CRM ↔ delivery, and a **shared delivery vehicle** several opportunities can feed roles into. _(Realized (first slice) — `projects`; a thin row (name + required `companyId`, `onDelete: restrict` + timestamps); the opportunity link lives on `opportunities.projectId` (**many opps → one project**), populated when a project is created from or associated to an opportunity; delivery managers via `project_delivery_managers`. **Status and line of business are derived from its roles, not stored** (`project-derived.ts`). `updateProject` edits only name + delivery managers; roles are edited via the opportunity planner. Company can't be changed after create; a sole-owned project is deleted via the detach/delete flow.)_ See [domains/projects.md](./domains/projects.md) and [ADR 0033](./decisions/0033-line-of-business-on-role-derived-project-status.md).
- **Allocation** — a _time-ranged_ assignment of a Person to a Project (start/end, % or hours, project role). The heart of capacity planning. _(Partially realized — `project_roles` is the first cut: a staff member (or a **placeholder/open position** when `staffId` is null) of a given role type + **line of business** for a date range at N hours/day, carrying a **`status`** (`tentative` → `confirmed`, auto-confirmed on the opportunity's win = soft vs. hard; plus enum-only `paused`/`cancelled`) and an **`opportunityId`** provenance FK. Simple rows, not effective-dated history yet — see [ADR 0017](./decisions/0017-project-roles-as-first-allocation-cut.md), [ADR 0031](./decisions/0031-opportunity-project-planner-and-role-status.md), [ADR 0033](./decisions/0033-line-of-business-on-role-derived-project-status.md). A **read-only, company-wide allocations planner** (`/allocations`) now surfaces this data as a weekly staff × weeks grid — a **view over `project_roles` + `staff`/`staff_employment` + `staff_pto`, no new table** — see [domains/allocations.md](./domains/allocations.md) and [ADR 0038](./decisions/0038-allocations-planner-pto-disclosure.md).)_
- **TimeEntry** — hours a Person logged against a Project (or a non-billable bucket) on a date. Grouped into a **Timesheet** = one Person's week (`draft`→`submitted`). _(Realized — `time_entries` + `timesheets`; per-day rows, whole-week save, submit-locks. Approval + billing deferred — see [domains/timesheets.md](./domains/timesheets.md) and [ADR 0027](./decisions/0027-timesheet-weekly-model-and-edit-window.md).)_
- **Feedback** — point-in-time peer feedback from one Person about another (rating + context + keep/stop/start + optional recipient message). _(Realized — `feedback`; staff↔staff, tiered visibility — see [domains/performance.md](./domains/performance.md) and [ADR 0023](./decisions/0023-feedback-privacy-tiers.md).)_
- **Rating / Level** — a Person's overall performance level (L0–L4) at a point in time. _(Realized — `staff_rating`; effective-dated, nullable level, manager/admin-only with no self-view — see [domains/performance.md](./domains/performance.md) and [ADR 0032](./decisions/0032-staff-rating-levels-effective-dated-manager-only.md).)_
- **ReviewCycle / PerformanceReview / Goal** — periodic assessment of a Person, often informed by their project work and utilization. _(Proposed.)_

## Key derived concepts

- **Utilization** = billable hours ÷ available hours for a Person over a period. Drives capacity (for allocations) and feeds performance.
- **Forecast vs. actuals** — Allocations (`project_roles`) are the _plan_; TimeEntries (now realized) are the _actuals_. Both exist; **reconciling them is still unbuilt** and a central reporting need.
- **Margin** = (charge rate − cost rate) × billable hours, per Project/Person.

## Nuances to respect

- Everything time-bound (allocations, rates, reviews, **employment**) needs effective-dated handling — a person's role or rate changes over time. The realized pattern is **history-as-rows** (`staff_employment`): a new row per change, current state = latest `effectiveFromDate`. See [ADR 0007](./decisions/0007-staff-employment-effective-dating.md). **Two deliberate exceptions:** (1) **Skills** are stored inline and edited in place, with no per-skill history — see [ADR 0018](./decisions/0018-skills-inline-jsonb-catalogue.md); (2) the first Allocation cut (`project_roles`) also stores simple mutable rows rather than effective-dated history, so it can't reconstruct a past plan — a deliberate scope call for the first slice, see [ADR 0017](./decisions/0017-project-roles-as-first-allocation-cut.md).
- Rates and salaries are highly sensitive. **Compensation** is now realized (on `staff_employment`) and access-gated by `staff.viewCompensation` (own always visible) — see [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md), authorization in [architecture.md](./architecture.md), and the RBAC model in [domains/permissions.md](./domains/permissions.md).
- **Dates and times are timezone-agnostic.** Calendar dates (effective-from, PTO spans, join/termination) are `date()` strings (`"YYYY-MM-DD"`, no zone); instants are plain `timestamp` _without_ time zone — treat all stored datetimes as wall-clock and convert zones only at the UI edge. See [`.claude/rules/database.md`](../.claude/rules/database.md).

### Location (free text)

**`companies`, `contacts`, and `staff` each carry a nullable `location`** — where the entity is based, stored as a **free-text `"City, CC"` label** (e.g. `"Toronto, CA"`), **not a FK or `pgEnum`** (migration `0005_secret_zarek.sql`). Valid values come from a **static world-cities dataset**, `src/lib/cities/world-cities.json` (a column-oriented parallel-array JSON carrying `lat`/`lng` per city; currently a small US/CA sample, swappable for the full set at the same path — the loader is size-agnostic), parsed once at module load by the pure module **`src/lib/cities/cities.ts`**. That module exports `searchCities(query)` (case-insensitive substring match, capped at `SEARCH_LIMIT`, `id === name === label`), `cityLabelsForCountries(codes)` (used by the seed), and **`citiesNear(label, radiusKm = NEARBY_RADIUS_KM)`** — the labels within a great-circle (haversine) radius of a given city, inclusive; it degrades to `[label]` for an unknown label (stale param / label absent from the dataset), so a "nearby" filter never matches nothing. The exported const **`NEARBY_RADIUS_KM = 100`** is the default radius — the rough "fly into one metro, meet people in the adjacent ones" ballpark (Ottawa/Gatineau, NYC/Newark). The index now carries `lat`/`lng`, plus a `byLabel` Map for O(1) origin lookup. It has **no `import "server-only"`** so the Bun seed can import it, but must **never** be imported by a client component (it would bundle the whole dataset). The label string itself is what's stored — there's no city id. **Editing UI exists on contacts and companies only** (inline picker, [domains/crm.md](./domains/crm.md#location)); **staff displays `location` read-only on the profile sidebar but has no editor** (column + seed only). **Display surfaces:** the staff profile sidebar ("Location" meta row), the contacts list table, and the companies list table each show `location`. **Location is now a filter on all three list pages** ([Location filters](#location-filters) below); it is still not part of the company name-search or status filter. Seeded values are drawn from `cityLabelsForCountries(["US","CA"])` so they match what the picker offers.

#### Location filters

All three list surfaces — **companies**, **contacts**, and the **staff directory** — carry a **location filter**: a city picker (reusing `searchCities`) plus a **"Search nearby"** switch that widens the match to every city within `NEARBY_RADIUS_KM` of the picked one (via `citiesNear`). The shared control is `src/components/form/location-filter-control.tsx` (presentational, state-agnostic — the caller owns `city`/`nearby`). **Two filtering models coexist, matching each surface's existing pattern:**

- **Companies & contacts filter server-side** (URL-param → server `WHERE`). The pages parse `city`/`nearby` search params; the reads (`getCompaniesPage`, `getContactsPage`) push `inArray(location, nearby ? citiesNear(city) : [city])` into their `where` (AND-combined with companies' existing name/status filters), calling `citiesNear` directly on the server. This gave **contacts its first-ever filter** (a new `ContactListFilters` + `contactsWhere`). See [domains/crm.md](./domains/crm.md#location).
- **The staff directory filters client-side in-memory** (its established pattern — the whole roster is fetched once, all filters run in JS, no URL params). Because the world-cities dataset must never bundle to the browser, "search nearby" is resolved through the **`nearbyCityLabels`** action (`src/actions/cities/nearbyCityLabels.ts`, auth-only, no capability gate — static public data, same justification as `searchCities`), which wraps `citiesNear` server-side; while that resolves, the directory falls back to exact-city match. This action exists **only** as that bridge — companies/contacts don't need it (their reads are already server-side). See the *Directory* flow in [domains/staff-profiles.md](./domains/staff-profiles.md).

### Junction tables — the first many-to-many pattern

The four `opportunity_*` join tables (`src/lib/db/opportunities-schema.ts`) are the repo's **first** many-to-many relations and set the convention future join tables should copy (`project_delivery_managers` follows it exactly; `project_roles` follows the FK/index parts but is a **data-carrying** line, not a pure junction — see [domains/projects.md](./domains/projects.md)):

- **Surrogate `text` PK** via `generateId(prefix)` (e.g. `opp-contact`), not a composite PK — consistent with the app-minted-CUID2 ID convention everywhere else ([`.claude/rules/database.md`](../.claude/rules/database.md)).
- **`unique(...)` on the FK pair** — gives set-semantics (a row can't be linked twice). Writers still dedupe input id arrays before insert so a repeat can't trip this index (see `createOpportunity`).
- **`index(...)` on the *non-owning* FK** (e.g. the `contactId`/`staffId` side) for efficient reverse lookups ("which opportunities is this contact on?").
- **Both FKs `onDelete: "cascade"`** — a link row is meaningless without both endpoints.

Reading the "other side" cheaply: resolve the linked names with a **single grouped follow-up query** over the relevant ids, not a per-row query (see `getOpportunitiesBoard`'s owner-name resolution) — avoid N+1.
