# Data model — the spine

The five domains share one model. This is the most important doc for understanding the system: changes here ripple everywhere.

**Status: mostly proposed.** Entities and relationships below are the intended design. Realized so far: the **Person** anchor (auth `user`), the **Staff profiles** tables (`staff`, `staff_employment`, `staff_pto`), the **CRM** slice (`companies`, `contacts`, `opportunities` + four opportunity junction tables), the **Projects** slice (`projects`, `project_delivery_managers`, `project_roles` — the first cut of Allocation), the **Timesheets** slice (`timesheets`, `time_entries` — weekly actuals), the first **Performance** slice (`feedback` — peer feedback), and a generic **`responses`** survey store (first consumer: the staff-profile **Manual of Me**) — see "What's realized in code".

## What's realized in code

- **Person** is realized by better-auth's **`user`** table (`src/lib/db/auth-schema.ts`) — the _login/identity_ anchor. (`user`/`session`/`account`/`verification` live in our schema and migrations, not better-auth's.) A logged-in user maps to its staff record via the nullable, unique **`staff.userId`** FK (`onDelete: set null`) — see [domains/staff-profiles.md](./domains/staff-profiles.md). The user's **`role`** (a `text()` column owned by better-auth's `auth-schema.ts`) drives RBAC; valid values and the role→permission matrix live in `src/lib/permissions.ts`, validated app-side by `roleSchema` (a `pgEnum` would be clobbered by `auth:generate`). See [domains/permissions.md](./domains/permissions.md).
- **Staff profiles domain** (`src/lib/db/staff-schema.ts`, barrelled by `src/lib/db/schema.ts`):
  - **`staff`** — the durable person record (notNull/unique external `ripplingId`, name, non-unique email, optional `userId` link to auth `user`, profile links, `clientIntro` (+ `clientIntroUpdatedAt`), free-text `resume` (+ `resumeUpdatedAt`), an inline **`skills`** jsonb list, lifecycle dates, `isActive`). Both `*UpdatedAt` instants are stamped **explicitly by their update action** when the text changes, not via `$onUpdate` (which would fire on every row write, e.g. an import re-sync). `skills` is the schema's **first jsonb/array column** — `StaffSkill[]` (`{ name, level }[]`, NOT NULL default `[]`) picked from the hardcoded catalogue in `src/lib/skills.ts`, deliberately **not** a normalized `skill` table; see [ADR 0018](./decisions/0018-skills-inline-jsonb-catalogue.md).
  - **`staff_employment`** — time-varying employment facts (line of business, role, employment type, billability, `utilizationTarget`, `billableType` (NOT NULL, default `HUB`), a boolean `isManagement`, and **compensation** — `base`/`hourlyRate`/`guaranteedBonus`/`discretionaryBonus` (`numeric(12,2)`, all **NOT NULL**) + `currency` (**NOT NULL**)), as **one effective-dated row per change**; current state = latest `effectiveFromDate`. FK → `staff.id` cascade. `isManagement` and `billableType` are **orthogonal to `role`** (someone can hold a working role *and* be management for it) and set in-app only. **Compensation lives here** rather than a separate table (comp changes coincide with employment changes) — it's **import-only** (no edit UI), **required** (the importer skips a row it can't supply comp for; **no carry-forward on the *import* path** — every imported row must carry full comp), and **view-gated** by `staff.viewCompensation` (own comp always visible). The no-carry-forward rule is specific to the import path; the **bulk-edit path** (a *non-comp* change spawning a new effective-dated row) *does* **carry comp forward** from the previous row, so advancing someone's role/LoB never clears their salary. See [ADR 0007](./decisions/0007-staff-employment-effective-dating.md), [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md), and [domains/staff-profiles.md](./domains/staff-profiles.md).
  - **`staff_pto`** — leave spans (`startDate`/`endDate`, `type`, `isPending`, **notNull/unique `ripplingId`** = Rippling "Leave request ID"). FK → `staff.id` cascade. Discrete rows, **not** effective-dated; reduces a person's available capacity for allocations. Populated by a second local-only importer (`/admin/upload-pto`) — see [domains/staff-profiles.md](./domains/staff-profiles.md).
- **`responses`** — a **generic survey-answer store** (`src/lib/db/responses-schema.ts`, barrelled by `schema.ts`; migration `drizzle/0030_confused_doctor_strange.sql`), keyed by **`(staffId, questionId)`** with a `unique` on the pair (→ upsert, one **current** answer per person per question — *not* history like `staff_employment`, and the opposite of the append-only `feedback` log). FK `staffId` → `staff.id` cascade. `questionId` is plain **`text`** validated in app code (**not a `pgEnum`**), so a new survey needs **no migration**. Three nullable payload shapes (`listResponse` jsonb `string[]`, `textResponse`, `jsonResponse`) — a question uses whichever fits. First and only consumer today: the staff-profile **Manual of Me** (`src/lib/manual-of-me.ts` = source of truth for its 7 free-text question ids), which uses `textResponse` only. See [ADR 0028](./decisions/0028-generic-responses-table-app-validated-question-ids.md) and [domains/staff-profiles.md](./domains/staff-profiles.md).
- **`lineOfBusinessEnum`** (`line_of_business`: `CORPORATE`, `CORE`, `FINTECH`, `COMMERCE`, `DESIGN`) is a **shared/global enum**, not a staff-specific type. It's carried by **three entities: `staff_employment`, `opportunities`, and `projects`** (all NOT NULL) — the practice that a person works in / a deal belongs to / an engagement bills under. It is **no longer on `project_roles`** (moved up to the project — [ADR 0025](./decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)). Its value tuple + labels live in a pure, client-importable module **`src/lib/line-of-business.ts`** (`LINE_OF_BUSINESS`, `LINE_OF_BUSINESS_LABELS`), the single source the `pgEnum` in `staff-schema.ts` and the CRM/projects zod schemas + forms all import — the same single-source pattern opportunities uses for its `source`/`status` enums (see [ADR 0016](./decisions/0016-junction-table-and-shared-enum-conventions.md)).
- **Local-only CSV import tools** are the only things that write the staff domain so far: `/admin/upload-staff` (Rippling export → `staff` + effective-dated `staff_employment`) and `/admin/upload-pto` (Rippling leave export → `staff_pto`). See [domains/staff-profiles.md](./domains/staff-profiles.md).
- **CRM slice** (`companies` + `contacts` in `src/lib/db/crm-schema.ts`; `opportunities` + its four junction tables + `source`/`status` pgEnums now in **`src/lib/db/opportunities-schema.ts`** — split out of `crm-schema.ts`, [ADR 0025](./decisions/0025-line-of-business-on-opportunity-and-project-not-role.md); both barrelled by `schema.ts`; migrations `drizzle/0012_spotty_mysterio.sql`, `drizzle/0013_clean_firelord.sql` (adds `contacts.phone`), `drizzle/0014_nostalgic_crusher_hogan.sql`, `drizzle/0020_unique_blue_shield.sql` (restructures `opportunity_status` to 14 leaf values, adds `opportunities.position`), `drizzle/0021_glamorous_goliath.sql` (adds `contacts.linkedinUrl` + self-referential `contacts.managerId`) + `drizzle/0024_harsh_diamondback.sql` (adds `opportunities.line_of_business`, NOT NULL, backfilled `CORE`)):
  - **`companies`** — organisations we deal with. `isPartner` is a **standalone boolean** flag (default false) marking partners — not a client/partner split; a company need be neither. `name` notNull, optional `websiteUrl`, and an optional **`ownerId`** — the accountable staff owner (FK → `staff`, `onDelete: set null`; a second CRM ↔ staff link, deliberately set-null not cascade). We chose *Company* over the narrower *Client* the spine originally named — see [ADR 0015](./decisions/0015-crm-company-over-client.md).
  - **`contacts`** — people, optionally employed by a company (`companyId` FK → `companies`, `onDelete: set null`; null when unknown or once the company is removed). `firstName`/`lastName`/`email` notNull, **`email` unique**, optional `phone`, optional free-text `role`, optional `linkedinUrl`, an optional **`managerId`** — the repo's **first self-referential FK** (→ `contacts.id`, `onDelete: set null`), "managed by" another contact, constrained app-side to a contact at the same company (see [ADR 0022](./decisions/0022-contact-manager-self-reference.md)) — and an optional **`ownerId`** → `staff` (`onDelete: set null`), the accountable staff owner (like `companies.ownerId`).
  - **`opportunities`** — pipeline deals, each **required** to belong to a company (`companyId` FK → `companies`, **`onDelete: restrict`** — contrast the contacts' set-null; a company with live opportunities can't be deleted). `source` + `status` pgEnums (`status` is a **flat 14-value leaf enum in pipeline order**; the two-level kanban grouping is derived in code in `src/lib/opportunity-pipeline.ts`, not in the DB), a **required `lineOfBusiness`** (the shared enum — which practice the deal belongs to; a project created from it defaults to the same, [ADR 0025](./decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)), optional `nextSteps`, and **`position`** (`doublePrecision`, a single global fractional index for manual kanban ordering — a drag writes the midpoint between neighbours, one-row update; index `opportunities_status_position_idx` on `(status, position)`). See [ADR 0021](./decisions/0021-opportunity-pipeline-groups-and-fractional-ordering.md). Their related people (contacts, owners, referral sources) live in **four junction tables** — the repo's first many-to-many pattern (see [below](#junction-tables--the-first-many-to-many-pattern)). `owners` and `source_staff` link to **`staff`**, the first CRM ↔ staff FK.
  - Open reads + `crm.edit`-gated create **and edit** mutations (one capability gates all CRM writes — companies, contacts *and* opportunities) + `/companies`, `/contacts` and `/opportunities` pages exist (companies and contacts are on **separate pages** now). Opportunities now have a two-tab **detail drawer** (an **Info** tab whose fields edit **one at a time in place** via `updateOpportunity` — status/line of business/next steps/owners/source/contacts/referral sources, company not editable — and a **Project plan** tab for the single linked project). The **won-Opportunity → Project handoff is now built** — `projects.opportunityId` is populated when a project is created from an opportunity, and a delivery-stage rule requires one before advancing ([ADR 0024](./decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)). See [domains/crm.md](./domains/crm.md).
- **Projects slice** (`src/lib/db/projects-schema.ts`, barrelled by `schema.ts`; imports `opportunities` from `./opportunities-schema`; migrations `drizzle/0015_premium_vertigo.sql`, `drizzle/0017_amused_corsair.sql` (adds `projects.opportunityId`), `drizzle/0023_eager_demogoblin.sql` (adds `project_role_type` enum; makes `project_roles.staff_id` nullable; adds `name` + `role_type`, backfilling existing rows to `ENGINEER` via a temporary default), `drizzle/0024_harsh_diamondback.sql` (adds `projects.line_of_business`, NOT NULL, backfilled `CORE`; **drops `project_roles.line_of_business`** — [ADR 0025](./decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)), `drizzle/0028_glossy_dorian_gray.sql` (adds the `project_status` enum + `projects.status`, NOT NULL default `tentative`), `drizzle/0030_white_raider.sql` (makes `projects_opportunity_idx` a **partial unique index** — one project per opportunity)):
  - **`projects`** — billable work that always belongs to a company (`companyId` FK → `companies`, **`onDelete: restrict`** — like opportunities, a company with live projects can't be deleted). `name` notNull, a **required project-level `lineOfBusiness`** (the shared enum — which practice bills the engagement; defaulted from the originating opportunity, NOT per-role), a **required lifecycle `status`** (`projectStatusEnum`: `tentative`/`confirmed`/`paused`/`cancelled`, NOT NULL, DB default `tentative`; tuple + labels in the pure module `src/lib/project-status.ts`, single-source like `line-of-business.ts` — set at create, **no edit flow yet**). id prefix `proj`. Also carries an **optional `opportunityId`** (nullable FK → `opportunities`, **`onDelete: restrict`**): the CRM → delivery link, **at most one project per opportunity** — enforced by a **partial unique index** `projects_opportunity_idx` (`WHERE opportunity_id is not null`, migration `drizzle/0030_white_raider.sql`; partial so standalone null-FK projects stay unconstrained, and it doubles as the FK lookup index). Optional so projects can be created standalone; **restrict** mirrors `companyId`. **Now populated by the handoff flow** ([ADR 0024](./decisions/0024-opportunity-project-handoff-and-placeholder-roles.md)); the same-company invariant (opportunity's company == project's company) is still **not** server-enforced. See [ADR 0019](./decisions/0019-project-opportunity-link.md).
  - **`project_delivery_managers`** — junction (many staff per project) following the CRM junction convention exactly ([ADR 0016](./decisions/0016-junction-table-and-shared-enum-conventions.md)): surrogate `text` PK, `unique(projectId, staffId)`, index on `staffId`, both FKs cascade.
  - **`project_roles`** — a **staffing line** (NOT a pure junction — carries columns): `projectId` → projects (cascade), **`staffId` → staff (restrict, NULLABLE)**, `roleType` (NOT NULL, new `projectRoleTypeEnum`), optional `name`, `startDate`/`endDate` (date, string mode), `hoursPerDay` (numeric(4,2), default 8). **No `lineOfBusiness`** — that moved up to the project ([ADR 0025](./decisions/0025-line-of-business-on-opportunity-and-project-not-role.md)). Indexed on both FKs; **no `unique` on the FK pair** (a person can hold multiple lines on one project). A **null `staffId` is a placeholder / open position** defined before it's staffed (identified by `roleType` + optional `name`); dates and hours are required on every role. **`roleType` (discipline — `ENGINEER`/`DESIGNER`/`ARCHITECT`/`QA`/`SPECIALIST`) is orthogonal to the project's `lineOfBusiness`** (what kind of work vs. which practice bills it); its tuple + labels live in the pure module `src/lib/project-role-type.ts` (mirrors `line-of-business.ts`). This is the **first concrete cut of the proposed Allocation entity** — stored as simple mutable rows, *not* effective-dated history. See [domains/projects.md](./domains/projects.md), [domains/allocations.md](./domains/allocations.md), [ADR 0017](./decisions/0017-project-roles-as-first-allocation-cut.md), and [ADR 0024](./decisions/0024-opportunity-project-handoff-and-placeholder-roles.md).
  - **The won-Opportunity → Project link is now populated.** `createProject` accepts an optional `opportunityId` and sets `projects.opportunityId` when a project is created from an opportunity (the detail drawer's Project plan tab or the board's delivery-stage prompt), rejecting a **second** project for the same opportunity (`opportunityHasProject` → user-safe error, backing the partial unique index); a `requiresProject(status)` rule blocks advancing an opportunity into delivery stages (Allocating+) without one. See [ADR 0024](./decisions/0024-opportunity-project-handoff-and-placeholder-roles.md).
  - Open reads + `projects.edit`-gated create + `/projects` page exist; create + read only (no edit/delete). See [domains/projects.md](./domains/projects.md).
- **Performance slice** (`src/lib/db/performance-schema.ts`, barrelled by `schema.ts`; migration `drizzle/0022_elite_khan.sql`):
  - **`feedback`** — peer feedback: `fromStaffId` + `toStaffId` (both FK → `staff.id`, both `onDelete: cascade`, each indexed), a 5-point `feedback_rating` pgEnum, required `context`, optional `keepDoing`/`stopDoing`/`startDoing` (at least one required, enforced in the zod schema)/`other`, and an optional `messageToRecipient`. **Point-in-time** (an immutable event log — NOT effective-dated; a person may leave feedback about the same person repeatedly, so **no unique `(from, to)`**). The rating enum's values/labels live in the pure, client-importable **`src/lib/feedback-rating.ts`** (same single-source pattern as `line-of-business.ts`). Privacy is enforced **by the read projections, not the table** — three tiers (give = open to active staff via the `authorizeFeedbackCreate` hook; recipient sees only giver name + `messageToRecipient`; `feedback.review` = manager/admin see all in full). See [domains/performance.md](./domains/performance.md) and [ADR 0023](./decisions/0023-feedback-privacy-tiers.md).
  - The Performance domain's second slice — the **compensation analytics dashboard** (`/performance`) — **adds no table**: it aggregates the latest `staff_employment` comp per active staff member, anonymized, gated by `staff.viewCompensation`. Cross-currency comparison normalizes via live FX rates ([ADR 0029](./decisions/0029-external-fx-rates-and-currency-normalization.md)). See [domains/performance.md](./domains/performance.md).
- **Timesheets slice** (`src/lib/db/timesheets-schema.ts`, barrelled by `schema.ts`; migration `drizzle/0026_rich_the_captain.sql`):
  - **`timesheets`** — one person's week: `staffId` → `staff` (cascade), **`weekStartDate`** (`date`, ISO Monday), `status` (`timesheet_status`: `draft`|`submitted`), `submittedAt` (nullable). **`unique(staffId, weekStartDate)`** — one sheet per person per week; the row is created **lazily** on first save/submit. id prefix `ts`.
  - **`time_entries`** — per-day rows: `timesheetId` → `timesheets` (cascade), `date`, `hours` (`numeric(4,2)`), targeting **exactly one** of `projectId` → `projects` (**`restrict`**, nullable) or a non-billable `category` (`time_entry_category`: `PTO`/`UNALLOCATED_BENCH`/`INTERNAL_ADMIN`). The XOR is enforced by a DB `CHECK` (`time_entries_target_check`) *and* zod. id prefix `te`. Category values live in the pure module `src/lib/timesheet-category.ts` (single-source pattern, like `line-of-business.ts`); the PTO bucket is **independent of `staff_pto`** (no sync).
  - These are the **actuals** complementing the `project_roles` allocation plan. Save is a whole-week transactional replace; submit locks the week (`timesheets.edit` capability edits any/locked week, bypassing the ±1-week owner edit window). No approval or billing yet. See [domains/timesheets.md](./domains/timesheets.md) and [ADR 0027](./decisions/0027-timesheet-weekly-model-and-edit-window.md).
- Nothing else (a full Allocation domain, review cycles/goals) exists yet. (The legacy `staff_profile` example table was deleted — `drizzle/0003_tranquil_miek.sql` drops it.)

## Core entities

- **Person** — an employee/contractor. Anchors most of the system. Realized as **`staff`** (durable identity) + **`staff_employment`** (effective-dated role/billability facts); login identity is the auth `user`, linked via the optional `staff.userId` (1:1, null until first sign-in).
  - **Skill** — capability + proficiency. _(Realized, but **not** as the many:many table this originally proposed:_ an inline `jsonb` array on `staff` (`skills`), catalogue-driven from `src/lib/skills.ts`. The staff directory now **filters people by skill (+ minimum level) client-side** over the once-fetched list — no indexed query. Cross-staff skill matching for *allocations* is still proposed and would need an indexed rework — see [ADR 0018](./decisions/0018-skills-inline-jsonb-catalogue.md).)
  - **Compensation** (base/hourly/bonuses + currency) is **realized** as effective-dated columns on `staff_employment` itself, not a separate table — import-only, required, view-gated. See [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md). External charge rates (what we bill a client) are still proposed and would follow the same history-as-rows pattern.
- **Company** — an organisation we deal with (CRM); an `isPartner` flag marks partners (standalone, not a client/partner split), plus an optional staff **owner** (`ownerId`, set-null). _(Realized — `companies`; create/read/edit.)_ Chosen over the narrower *Client* the spine first named; see [ADR 0015](./decisions/0015-crm-company-over-client.md).
  - **Contact** (optional company per contact) — a person, optionally at a Company, optionally "managed by" another contact at the same company (self-referential `managerId`), with an optional staff **owner** (`ownerId`, set-null). _(Realized — `contacts`; create/read/edit; `companyId` nullable, `onDelete: set null`; `managerId` nullable self-ref, `onDelete: set null`, same-company invariant app-enforced — see [ADR 0022](./decisions/0022-contact-manager-self-reference.md).)_
  - **Opportunity** — a pipeline deal that always belongs to a Company; when it reaches delivery stages, links to a Project. _(Realized — `opportunities` + junction tables; `companyId` required, `onDelete: restrict`. The Project link (`projects.opportunityId`, **at most one project per opportunity**) is now populated by the handoff flow — see [ADR 0019](./decisions/0019-project-opportunity-link.md), [ADR 0024](./decisions/0024-opportunity-project-handoff-and-placeholder-roles.md).)_
- **Project** (a.k.a. engagement) — billable work for a Company. The hub linking CRM ↔ delivery. _(Realized (first slice) — `projects`; required `companyId`, `onDelete: restrict`; optional `opportunityId` (FK → `opportunities`, `restrict`, at most one project per opportunity), now populated when a project is created from an opportunity; delivery managers via `project_delivery_managers`; a lifecycle `status` (`tentative`/`confirmed`/`paused`/`cancelled`, default `tentative`) set at create. Create + read only — status can't be changed post-create yet.)_ See [domains/projects.md](./domains/projects.md).
- **Allocation** — a _time-ranged_ assignment of a Person to a Project (start/end, % or hours, project role). The heart of capacity planning. _(Partially realized — `project_roles` is the first cut: a staff member (or a **placeholder/open position** when `staffId` is null) of a given role type for a date range at N hours/day (line of business lives on the parent project, not the role). Simple rows, not effective-dated history yet — see [ADR 0017](./decisions/0017-project-roles-as-first-allocation-cut.md), [ADR 0024](./decisions/0024-opportunity-project-handoff-and-placeholder-roles.md), [ADR 0025](./decisions/0025-line-of-business-on-opportunity-and-project-not-role.md).)_
- **TimeEntry** — hours a Person logged against a Project (or a non-billable bucket) on a date. Grouped into a **Timesheet** = one Person's week (`draft`→`submitted`). _(Realized — `time_entries` + `timesheets`; per-day rows, whole-week save, submit-locks. Approval + billing deferred — see [domains/timesheets.md](./domains/timesheets.md) and [ADR 0027](./decisions/0027-timesheet-weekly-model-and-edit-window.md).)_
- **Feedback** — point-in-time peer feedback from one Person about another (rating + context + keep/stop/start + optional recipient message). _(Realized — `feedback`; staff↔staff, tiered visibility — see [domains/performance.md](./domains/performance.md) and [ADR 0023](./decisions/0023-feedback-privacy-tiers.md).)_
- **ReviewCycle / PerformanceReview / Goal** — periodic assessment of a Person, often informed by their project work and utilization. _(Proposed.)_

## How the domains connect

```
Company ──< Contact                        (optional company; FK set-null)
Contact ──< Contact                        (optional self-ref managerId "managed by"; FK set-null; same-company, app-enforced)
Company ──o Person (staff)                 (optional ownerId; FK set-null) [realized]
Contact ──o Person (staff)                 (optional ownerId; FK set-null) [realized]
Company ──< Opportunity                     (required company; FK restrict)
Opportunity ──o Project                     (optional opportunityId; ≤1 project per opp, partial-unique; FK restrict) [populated by the handoff flow]
Opportunity >──< Contact                   (related + referral-source, via junction tables)
Opportunity >──< Person (staff)            (owners + referral-source, via junction tables)
Company ──< Project                        (required company; FK restrict) [realized]
Project >──< Person (staff)                via project_delivery_managers (junction) [realized]
Project >──< Person (staff)                via Allocation = project_roles (first cut; staffId FK restrict, NULLABLE = placeholder) [realized]
Person (staff) ──< Timesheet              (one per person per week; draft→submitted) [realized]
Timesheet ──< TimeEntry >── Project        (the actuals; entry targets a project OR a non-billable bucket) [realized]
user (auth)    ──0:1── Person (staff)      via staff.userId (nullable, unique)
Person (staff) ──< StaffEmployment         (effective-dated; latest = current)
Person (staff) ──< StaffPto                 (leave spans; reduce capacity)
Person (staff) ──inline── Skill[]           (jsonb column on staff, not a table)
Person (staff) ──< Response                 (generic survey answers; upsert per (staff, question); Manual of Me) [realized]
Person (staff) ──< Feedback >── Person (staff)  (peer feedback; from/to, both FK cascade) [realized]
Person  ──< PerformanceReview              (within a ReviewCycle) [proposed]
```

## Key derived concepts

- **Utilization** = billable hours ÷ available hours for a Person over a period. Drives capacity (for allocations) and feeds performance.
- **Forecast vs. actuals** — Allocations (`project_roles`) are the _plan_; TimeEntries (now realized) are the _actuals_. Both exist; **reconciling them is still unbuilt** and a central reporting need.
- **Margin** = (charge rate − cost rate) × billable hours, per Project/Person.

## Nuances to respect

- Everything time-bound (allocations, rates, reviews, **employment**) needs effective-dated handling — a person's role or rate changes over time. The realized pattern is **history-as-rows** (`staff_employment`): a new row per change, current state = latest `effectiveFromDate`. See [ADR 0007](./decisions/0007-staff-employment-effective-dating.md). **Two deliberate exceptions:** (1) **Skills** are stored inline and edited in place, with no per-skill history — see [ADR 0018](./decisions/0018-skills-inline-jsonb-catalogue.md); (2) the first Allocation cut (`project_roles`) also stores simple mutable rows rather than effective-dated history, so it can't reconstruct a past plan — a deliberate scope call for the first slice, see [ADR 0017](./decisions/0017-project-roles-as-first-allocation-cut.md).
- Rates and salaries are highly sensitive. **Compensation** is now realized (on `staff_employment`) and access-gated by `staff.viewCompensation` (own always visible) — see [ADR 0020](./decisions/0020-compensation-effective-dated-import-only.md), authorization in [architecture.md](./architecture.md), and the RBAC model in [domains/permissions.md](./domains/permissions.md).
- **Dates and times are timezone-agnostic.** Calendar dates (effective-from, PTO spans, join/termination) are `date()` strings (`"YYYY-MM-DD"`, no zone); instants are plain `timestamp` _without_ time zone — treat all stored datetimes as wall-clock and convert zones only at the UI edge. See [`.claude/rules/database.md`](../.claude/rules/database.md).

### Junction tables — the first many-to-many pattern

The four `opportunity_*` join tables (`src/lib/db/opportunities-schema.ts`) are the repo's **first** many-to-many relations and set the convention future join tables should copy (`project_delivery_managers` follows it exactly; `project_roles` follows the FK/index parts but is a **data-carrying** line, not a pure junction — see [domains/projects.md](./domains/projects.md)):

- **Surrogate `text` PK** via `generateId(prefix)` (e.g. `opp-contact`), not a composite PK — consistent with the app-minted-CUID2 ID convention everywhere else ([`.claude/rules/database.md`](../.claude/rules/database.md)).
- **`unique(...)` on the FK pair** — gives set-semantics (a row can't be linked twice). Writers still dedupe input id arrays before insert so a repeat can't trip this index (see `createOpportunity`).
- **`index(...)` on the *non-owning* FK** (e.g. the `contactId`/`staffId` side) for efficient reverse lookups ("which opportunities is this contact on?").
- **Both FKs `onDelete: "cascade"`** — a link row is meaningless without both endpoints.

Reading the "other side" cheaply: resolve the linked names with a **single grouped follow-up query** over the relevant ids, not a per-row query (see `getOpportunitiesBoard`'s owner-name resolution) — avoid N+1.
