# RBAC & Permissions — Design Spec

**Date:** 2026-06-18
**Status:** Approved, ready for implementation
**Foundation:** Better Auth native access control (`createAccessControl` + the already-enabled `admin` plugin)

## Goal

Introduce role-based access control to the PSA platform. Define a small set of
capabilities (permissions), a fixed set of roles that grant them, and enforce
them at both the action route level and the row level. Make permissioning a
first-class, auditable, hard-to-break part of the system.

## Decisions (locked)

1. **Foundation:** Better Auth native access control — not a hand-rolled resolver.
2. **ADR 0012:** Close the open-staff-edit gap now. Owners always edit their own
   profile; editing *other* staff requires the `staff.edit` permission.
3. **Role cardinality:** Single role per user.
4. **Role assignment:** Deferred. Roles set via Better Auth `setRole` API / DB.
   No assignment UI in this iteration.

## Permission model

Permissions are `(resource, action)` pairs. Two resources in this iteration:

- `staff: ["edit"]` — edit profiles of *other* staff.
- `pto: ["review"]` — view aggregated PTO summaries of *other* staff.

(Better Auth's `defaultStatements` — `user` / `session` admin-plugin perms — are
merged in so the admin role keeps its built-in capabilities.)

### Roles → permissions (canonical matrix)

| Role               | `staff.edit` | `pto.review` | Notes                          |
| ------------------ | :----------: | :----------: | ------------------------------ |
| `user`             |      –       |      –       | default role for new users     |
| `delivery-manager` |      –       |      –       | no permissions (yet)           |
| `finance`          |      –       |      –       | no permissions (yet)           |
| `sales`            |      –       |      –       | no permissions (yet)           |
| `manager`          |      ✓       |      ✓       | all defined business perms     |
| `admin`            |      ✓       |      ✓       | + Better Auth admin-plugin perms |

This table is the contract. It is asserted by both a test and the `/audit-rbac`
command. Changing it requires changing all three (code, test, this doc).

## Components

### 1. `src/lib/permissions.ts` (new — single source of truth)

```ts
import { createAccessControl } from "better-auth/plugins/access";
import { defaultStatements, adminAc } from "better-auth/plugins/admin/access";

export const statement = {
  ...defaultStatements,
  staff: ["edit"],
  pto: ["review"],
} as const;

export const ac = createAccessControl(statement);

export const roles = {
  "user": ac.newRole({}),
  "delivery-manager": ac.newRole({}),
  "finance": ac.newRole({}),
  "sales": ac.newRole({}),
  "manager": ac.newRole({ staff: ["edit"], pto: ["review"] }),
  "admin": ac.newRole({ staff: ["edit"], pto: ["review"], ...adminAc.statements }),
} as const;

export type AppRole = keyof typeof roles;
export const ROLE_SLUGS = Object.keys(roles) as [AppRole, ...AppRole[]];
export const roleSchema = z.enum(ROLE_SLUGS); // app-layer validation of role values

// Permission shape: a subset of `statement`.
export type PermissionCheck = { staff?: ("edit")[]; pto?: ("review")[] };

// Imperative helpers for row-level checks (action bodies AND server reads).
export async function userHasPermission(
  user: { id: string; role?: string | null },
  permissions: PermissionCheck,
): Promise<boolean>;
export async function requirePermission(
  user: { id: string; role?: string | null },
  permissions: PermissionCheck,
): Promise<void>; // throws UserSafeActionError when denied
```

`userHasPermission` delegates to `auth.api.userHasPermission({ body: { userId, permissions } })`
(or `{ role, permissions }`) so the access-control logic has exactly one
implementation. `requirePermission` wraps it and throws `UserSafeActionError`
("You don't have permission to do that.").

### 2. Wiring Better Auth

- `src/lib/auth.ts`: `admin({ ac, roles, adminRoles: ["admin"], defaultRole: "user" })`.
- `src/lib/auth-client.ts`: `adminClient({ ac, roles })` so client `hasPermission` /
  `checkRolePermission` stay in sync.
- `user.role` **stays `text()`**. Better Auth owns `auth-schema.ts` (regenerated by
  `auth:generate`); a `pgEnum` there would be clobbered. Validity is enforced at
  the app layer via `roleSchema`. (Optional later hardening: a DB `CHECK`
  constraint in a hand-written migration.)

### 3. Action-layer enforcement (`src/lib/action.ts`)

Extend the metadata schema with an optional capability gate, keeping the existing
`role` gate for back-compat:

```ts
z.object({
  action: z.string(),
  role: z.enum(["user", "admin"]).optional(),
  permission: z.custom<PermissionCheck>().optional(),
})
```

`secureActionClient`: after `checkAuth`, if `metadata.permission` is set, call
`requirePermission(ctx.user, metadata.permission)` before the body runs.

This gives **declarative route-level** capability gating. Row-level checks (own
vs. other) still live in action bodies via `requirePermission`.

### 4. Close ADR 0012 — `staff.edit`

`updateStaffLinks` and `updateStaffClientIntro`:

- Resolve the caller's own staff row (`staff.userId === ctx.user.id`).
- If `parsedInput.staffId` is the caller's own row → allow.
- Otherwise → `await requirePermission(ctx.user, { staff: ["edit"] })`.

Remove the `// TODO: lock down` comments. A small reusable helper
(`assertCanEditStaff(user, targetStaffId)`) keeps both actions DRY and is the
single place ownership-or-permission is decided for staff edits.

### 5. `pto.review`

Guard the read path that exposes *other* people's PTO (aggregated summaries):
own PTO always visible; others require `pto.review` via `requirePermission`. The
permission + helper are in place for the aggregated-summary UI when it lands; we
apply the guard at the existing PTO read boundary that surfaces non-owner data.

## The "never break permissions" guardrail

1. **`.claude/rules/permissions.md`** — path-scoped to `src/lib/auth*.ts`,
   `src/lib/permissions.ts`, `src/lib/action.ts`, `src/actions/**`. States the
   invariant:
   - Every sensitive/mutating action declares a permission/role gate **or** a
     documented justification for being public.
   - Never bypass the action client for DB writes.
   - Row-level ownership/permission checks are mandatory where an action accepts a
     target id it could mutate across users.
   - **If you discover a bypass, escalation path, or vulnerability, STOP and flag
     it immediately. Do not silently work around it.**
2. **AGENTS.md** — a short pointer to the rule and to `/audit-rbac`.
3. **`src/lib/permissions.test.ts`** — encodes the canonical matrix above and
   fails (`bun run check` / CI) if any role's permission set drifts. Makes matrix
   changes deliberate.

## `/audit-rbac` command

`.claude/commands/audit-rbac.md` — a **read-only** audit. Reports findings,
flags vulnerabilities prominently, never auto-fixes. Checks:

1. Code matrix in `permissions.ts` matches this doc's canonical table.
2. Every file under `src/actions/**` declares `metadata` with a `permission`/`role`
   gate **or** an explicit, justified public marker — lists any ungated action.
3. No `db.insert/update/delete` outside the actions layer (the two documented
   exceptions in the server-actions rule aside).
4. Actions accepting a target id have a row-level ownership/permission check.
5. `auth.ts` / `auth-client.ts` role sets stay in sync with `permissions.ts`.

## Testing

- `permissions.test.ts`: matrix assertion (each role → exact permission set) using
  `userHasPermission`.
- Unit tests for `assertCanEditStaff`: owner allowed, non-owner without `staff.edit`
  denied, non-owner with `staff.edit`/admin allowed, missing target → safe error.
- Follow TDD: tests first, then implementation.

## Out of scope (deferred)

- Role-assignment UI.
- Multiple roles per user.
- DB-level role enum / CHECK constraint (optional later hardening).
- New aggregated-PTO-summary UI (only the permission + guard hook are built).
